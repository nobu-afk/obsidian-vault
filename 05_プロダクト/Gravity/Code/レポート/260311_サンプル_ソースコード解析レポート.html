1

サマリー

石井伸幸とは、「カオスを検知し、ゼロから構造を立ち上げ、自走させた瞬間に次のカオスへ移動する」構造構築システムである。

動詞で記述すると：

検知する → 没入する → 収集する → 構造化する → 立ち上げる → 巻き込む → 軌道に乗せる → 手放す

このシステムは「改善する」「維持する」機能を持たない。対象が構造化された時点でエネルギー供給が停止し、次の構造不在を探索し始める。これは意志や性格ではなく、動作原理である。

ドメインは「遊び→スポーツ→コンサル→人事→事業開発→コーチング」と変遷したが、実行されている動詞は30年以上一度もブレていない。「コンサルタント」「人事責任者」「コーチ」という肩書きは、このシステムがどの領域で稼働したかのログに過ぎない。

2

駆動源

エンジンの正体

「まだ誰も構造化していない混沌を見つけると、秩序を立ち上げずにはいられない」——これがこのシステムの駆動衝動である。

「改善したい」「貢献したい」という動機ではない。既に構造化された対象には一切反応しない。混沌そのものが燃料であり、「無から有を立ち上げる」動作の実行中にのみ最大出力を返す。幼少期、自然の中で「まだ見つけられていない対象」を探索することに没頭していた時点で、このエンジンは既に稼働していた。対象が「昆虫」から「事業構造」に変わっただけである。

発火条件

  1. 構造不在のカオスであること。 仕組みがない、前例がない、正解がない——この3条件でエンジンが点火する。マニュアルが整備された環境では出力がゼロに近づく。
  2. 「駅伝的構造」があること。 共通ゴールに全員が本気で走る場で最大出力が発生する。単独でも動くが、チームが持ち場で全力を出す構造がトリガーになる。
  3. 自分の現在地を破壊する「格上」に触れること。 キャリアの転換点はすべて、圧倒的に上の基準を持つ人間との出会いから始まっている。この衝撃が自己像を破壊し、次のステージへの没入を発火させる。

停止条件

  1. 運用フェーズに入った時。 「回す」フェーズになった途端、エネルギー供給が急速に低下する。怠慢ではなくシステム仕様。複数の環境で同一パターンが反復確認されている。
  2. ゴールが消失した時。 「全ての景色が灰色に見える」——ゴール不在はこのシステムにとって酸素の遮断に等しい。「何を目指すか」が空白になった瞬間、全プロセスが停止する。
  3. 外部から動作を強制された時。 興味のない対象を「やれ」と指示された瞬間、即シャットダウンする。幼少期から一貫しており、意志でコントロールできない。強制下での辻褄合わせは自己欺瞞を蓄積させ、システム全体を劣化させる。
3

フィルター

適合度が高い環境

  • 0→1フェーズの組織・事業。 仕組みが存在しない場で最大出力が出る。
  • 複数領域を同時並行で走らせられる構造。 異なるドメインの並走が相互作用を生み、出力を押し上げる。
  • 率直なフィードバックと高い基準がある場。 忖度のない指摘がエンジンのターボモードを点火させる。
  • 意思決定のスピードが速い組織。 「考えながら走る」が基本動作。

適合度が低い環境

  • 既存の仕組みを守り・改善する役割。 構造が完成した時点で「燃料切れ」になる。
  • 合意形成に時間がかかる大組織。 エンジンの回転数とギアが噛み合わず摩耗する。
  • ゴールが曖昧なまま走る環境。 「とりあえず現状維持」が長期化するとシステムクラッシュを引き起こす。
  • 裁量のない環境。 外部から動作を規定された瞬間に起動すらしない。

have toの検出

検出 1

「人事の専門家」という名詞的自己定義。 キャリアの蓄積から「人事のプロ」という自己像が形成されていたが、本質は「構造不在にゼロから仕組みを立ち上げる」動詞であり、人事である必然性はない。名詞が、ドメインを超えた転用を無意識に制限していた。
剥がした先：「人事の専門家」→「構造不在を検知し秩序を立ち上げるシステム」。

検出 2

「優等生であるべき」というバックグラウンドプロセス。 幼少期にインストールされた「模範行動を自動実行する」プログラム。作動するとwant toではなく「求められている行動」が優先され、辻褄合わせの動作が発生する。ソースコードとは無関係な動作のため、出力精度が著しく低下する。
剥がした先：「優等生」→「エンジンが反応する対象にだけ全力を注ぐ」。

検出 3

親由来の経済的ブロック。 幼少期に内面化した「お金に関する否定的な言葉」が、価格設定や収益構造の設計で無意識の制限として作動。本人が自覚的に検知し、解除プロセスを実行済み。
剥がした先：「お金を稼ぐ＝悪」→ 経済的価値の生成はシステムの出力指標の一つに過ぎない。

4

実行パターン

コアプロセス

検知する → 没入する → 収集する → 構造化する → 立ち上げる → 巻き込む → 軌道に乗せる → 手放す → 次のカオスを検知する
検知する

「まだ構造化されていない領域」を自動的に嗅ぎ取る。本人が意識的に「見つけた」のではなく、システムが勝手に検知している。

没入する

検知した瞬間にエンジンが自動全開になる。「頑張っている」のではなく、時間感覚が消失する没入状態。

収集する

書籍、人、現場——ソースを選ばず大量に高速インプットし、短期間で全体像を把握する。

構造化する

大量の情報を独自のフレームワークに変換する。既存フレームの流用ではなく、自分のOSで再解釈して新しい構造を生成する。

立ち上げる

ゼロから仕組みを実装する。設計だけでなく、自ら手を動かして形にするまでが1セット。

巻き込む

自分より優秀なメンバーを配置し、構造の耐久性を上げる。全部やるのではなく、任せる。

軌道に乗せる

仕組みが自走する状態——自分が抜けても回る状態——まで持っていく。ここが「完成」の定義。

手放す

軌道に乗った瞬間にエネルギーが停止し、次のゼロへ移動する。「飽きっぽい」のではなく動作仕様。

強みが最大化する場面

  • 「何もない」状態から短期間で仕組みの骨格を立ち上げる場面。ミナジンでのコンサル事業立ち上げ、GrowthFixの創業がこのパターン
  • 複数の異なる領域を同時並行で走らせ、それぞれの学びを相互に転用する場面。人事×事業開発×コーチングの並走がシナジーを生んでいる
  • 大量情報の高速処理と構造化が求められる場面。DMMでの50事業横断のHRBP統括はこの能力が最も発揮された事例
  • 「正解がない問い」に対して仮説を立て、高速で検証→修正を回す場面

精度が落ちる場面

  • 構造が完成した後の運用・保守フェーズ。エネルギーが急速に低下し、本来の出力の20%以下になる
  • 自分が構造の設計権を持てない状況。他者が設計した構造の中で「パーツ」として動くことはこのOSでは不可能
  • 自己欺瞞（have toによる上書き）が蓄積した状態。「本当はやりたくないのにやっている」状態が続くと、全プロセスが汚染されて停止する
  • 単一領域に長期間閉じ込められた時。並走する領域がないと、相互作用が発生せず出力が単調になる
5

取扱説明書

やるべきこと

  1. 「次のゼロ」を常に仕込んでおく。 エンジンが停止する前に次の燃料を投下する。完了してから探すのでは遅い。
  2. 最低3領域を常に並走させる。 単一領域では演算資源が余る。異なるドメインの並走が相互転用を生み、出力を最大化する。
  3. 「格上の基準」に定期的に触れる。 自己完結状態が続くとエンジンの回転数が低下する。現在地を破壊する接点を意図的に設計する。
  4. 「手放し」を初期設計に組み込む。 プロジェクト開始時点で「自分が抜けても回る状態」を完了条件に設定する。
  5. have toの蓄積を定期的にリセットする。 「優等生プログラム」が走り出すとソースコードが汚染される。定期的な点検と解除が必要。

やってはいけないこと

  1. 「運用の達人」を目指してはいけない。 このシステムは「立ち上げ」特化型。運用は別のOSを持つ人間に委ねる対象である。
  2. ゴールなしで走り続けてはいけない。 現状維持はこのシステムにとって酸素の遮断。景色が灰色になったらゴール消失の証拠。
  3. 辻褄合わせの行動をしてはいけない。 「期待に応える」「体裁を保つ」動作はhave to作動の証拠。この動作の成果は必ず脆い。
  4. 「自分がやった方が早い」で抱え込んではいけない。 「巻き込む」をスキップすると自走状態に到達しない。
  5. 「名詞」で自分を定義してはいけない。 「人事の専門家」「コーチ」はドメインラベルであり、ソースコードではない。

推奨パス

ソースコードの解析結果に基づき、次のステップを3つの選択肢から提案します。

Gravity Coaching

have toが意思決定を歪めている場合に推奨。本ケースでは3件検出されたが、いずれも自覚的に解除済みのため緊急性は低い。ただし検出2「優等生OS」は環境変化時に再起動するリスクがあり、定期モニタリングは推奨。

Gravity Scan

事業にまだ伸びしろがあると感じている場合に推奨。60分の対話で、事業に隠された伸びしろを丸裸にする。本ケースでは創業者として自ら組織を設計する立場にあり、組織が10名を超えたフェーズで再検討の価値あり。

推奨

CODE 単独完結

ソースコードが明快で動詞連鎖が30年間ブレていない。have toは自覚的に解除済み。本レポートとプロンプト集を使い、自走で精度を上げていくことが最も合理的。「次のゼロ」を検知するたびに本レポートと照合することを推奨。

Analyst's Note： 解析難易度は低い。動詞連鎖が30年間ブレておらず、ドメインが変遷しても実行される関数は同一である。

最大のリスクは2つ。第一に、ソースコードを「名詞」で上書きすること——ドメインラベルはシステムの転用可能性を殺す。第二に、「手放し」の失敗。構造完成後も運用に留まると、エネルギー低下→自己欺瞞蓄積→システム劣化のループに入る。「手放す」はこのソースコードにおいて最も重要かつ最も難易度の高い関数である。
