## Part 1: 思想層

### Why（なぜハーネスエンジニアリングか・260429 統合）

**ルール：AI ネイティブ経営の本質はハーネスエンジニアリング。会社の美学・方針を「実行時のガード」まで設計して AI に渡し、人間は「ハーネスを作る／結果から更新する」役割に最小化する。**

**Why：** AI ネイティブ経営の定義（引力源 → 翻訳器 → 求心場）は方向性を示すが、**実装の核**を一語で言えば「ハーネスエンジニアリング」。森謙吾／AINative の動画（260429）は、開発世界のハーネス（馬車の手綱・AI に正しい出力をさせる制御層）を経営に持ち込む統合フレームとして提唱。GrowthFix の主軸先行ルール・自社ドッグフード・lint_consistency.sh は全て**ハーネスエンジニアリングの個別実装**として再解釈できる。

### 核心定義

**ハーネスエンジニアリング：**
> 会社のあるべきアウトプット定義（やっていいこと／いけないこと・良いアウトプット／悪いアウトプット）を**実行時の制御**まで含めて設計し、AI にそれを渡してアウトプットを生成・自動補正させる経営手法。

**ハーネスがしっかり回っている状態 ＝ AI ネイティブ組織 OS がインストールされている状態。**

### コンテクスト ≠ ハーネス

| 概念 | 内容 | GrowthFix 現状の例 |
|---|---|---|
| **コンテクスト** | 文脈情報として与える。実行制御ではない | 思想層 6 本（Why／引力／参謀／会社／culture／判断基準）|
| **ハーネス** | 実行時のガード（コマンド許可・出力テスト・フックス・テスト基準）| `lint_consistency.sh` ／ LP `verify_deployment.sh` ／ SSOT 機械チェック |

**含意：** コンテクスト整備（思想層）だけでは AI の出力品質は再現しない。**実行時に「これは出していい／これは NG」を判定するハーネス層**が必要。

### 3 層構造（コア／ハーネス／実行エージェント）

```
[1] コア（経営の核）
    ├─ 経営の美学・Why・あるべき姿の源泉
    └─ ハーネスを設計できる人（社長＋右腕）= ここが最重要

[2] ハーネス（実行ルール・テスト・フックス）
    ├─ 良いアウトプット／悪いアウトプットの定義
    ├─ コマンド許可リスト・実行時チェック
    └─ 結果からハーネス自体を自動更新するループ

[3] 実行エージェント（人間 or AI）
    └─ ハーネスに従って作業する。誰でも・何でも代替可能
```

**究極形：** ハーネスが結果から自動学習・自己修正するレベルに達すると、ハーネス調整役の人間も最小化できる。**この 3 層さえ回せば、人間は [1] だけで足りる**。

#### ★ 実行エージェント層の物理ファイル化（260503 確立）

[3] 実行エージェントは抽象概念ではなく **物理ファイルとして実装できる**。Claude Code の場合、`~/.claude/agents/*.md` 1 枚に「モデル指定 + system prompt + tools 限定 + ハーネス装置内蔵」をパッケージ化する。

**パッケージ構成（1 ファイル）：**

```yaml
---
name: <agent-name>
description: <用途・呼び出し条件>
model: opus | sonnet | haiku  # コア用 / ハーネス用 / 軽作業用の三択
tools: Read, Edit, Write, Bash, Grep, Glob, ...  # 与える権限を最小化
---

<system prompt：役割スコープ + 必須プロトコル + 報告フォーマット + やってはいけないこと>
```

**含意：**
- ハーネスがメモリ依存（`feedback_*` メモリを Read してから判断）ではなく **エージェント定義に内蔵される**
- メモリが消えてもエージェント定義が生きていれば再現可能
- モデル経済（Opus = コア / Sonnet = ハーネス実行）と物理ファイルが 1:1 で対応
- メインスレッド（Opus）は議論に専念、サブエージェントが実行する分業構造

**GrowthFix での実装（260503 開始）：**

| エージェントファイル | 担当 | model | 内蔵ハーネス |
|---|---|:-:|---|
| `~/.claude/agents/lp-implementer.md` | LP HTML / CSS / 診断 UI 編集 | sonnet | mobile audit 自動実行 / SSOT lint / 思想層判断は委譲しない |
| `~/.claude/agents/script-writer.md` | `06_開発/scripts/` Python・bash 改修 | sonnet | 構文チェック / ドライラン規定 / 秘匿情報取扱ガード |

**判断軸（新規エージェント追加時）：**
1. 「同じ作業を週 1 回以上やるか？」YES → エージェント化候補
2. 「思想層の判断が伴うか？」YES → コア（メイン Opus）に残す・エージェント化しない
3. 「Vault 固有のハーネス（lint / audit / SSOT 整合）が必要か？」YES → system prompt に内蔵
4. 「Sonnet で品質が十分か？」YES → サブエージェント化でコスト圧縮効く

**警戒：**
- ❌ 思想層判断（コピー・タグライン・サービス設計）をエージェントに委譲する → メインスレッドに差し戻す規定を system prompt に必ず書く
- ❌ tools を全許可で渡す → 最小権限原則
- ❌ 同じハーネスを複数エージェントで重複実装 → 共通スクリプト（`lint_consistency.sh` 等）を呼ぶ参照型に統一

### TDD 経営（Test-Driven Development を経営に持ち込む）

開発世界の TDD（テスト駆動開発）の経営版：

1. **あるべき状態（Pass 条件）を先に定義する**
   - 良い Note 記事の合格基準
   - 良い LP の CV 率・モバイル監査全項目クリア
   - 良い商談トークの構造（営業.md フェーズ運用）
2. **AI にハーネスを渡してアウトプットさせる**
3. **テスト失敗時はハーネスに不足があると判定し、ハーネスを更新する**
4. **結果が再現するまでループ**

**GrowthFix での既存 TDD 萌芽：**
- `feedback_*` メモリ（90 本超）= 失敗テストの蓄積（事後）
- `_共通/_LP変更時_モバイル確認チェックリスト.md` = LP 合格基準（事前）
- `lint_consistency.sh` = SSOT 準拠の自動テスト

⚠️ **未整備：** 各業務の「良いアウトプット事前定義」が体系化されていない。各 MD の「ハーネス層」（Part 3）セクションで継続整備中。

### クローズドループ（ハーネスの自己更新）

YC AI Native（Y Combinator）の指摘：「クローズドループが回る仕組みを作れ」。

```
アウトプット
   ↓ テスト（ハーネス基準で判定）
失敗
   ↓ なぜ失敗したかを AI に分析させる
ハーネス更新（プロンプト・ルール・参考メモリ追記）
   ↓
次のアウトプット → 再テスト
```

**GrowthFix での実装例：**
- `daily` skill 実行 → 出力品質に違和感 → feedback_* メモリに追記 → 次回 daily で参照される（既に部分実装）
- `iran-update` 投稿 → エンゲージメント低下 → 失敗パターンを iran-update スキルのプロンプトに反映（既に実装：feedback_iran_analysis_precision の 5 禁則）

⚠️ **未整備：** 静的なハーネス更新（人間が memory 追記）はあるが、**結果データから AI が自動でハーネスを提案する仕組み**は未実装。

### AI ネイティブ組織 ≠ 「組織」

動画の挑発的な結論：「1 人＋ AI エージェント 30 体は組織なのか？」「AI ネイティブな組織はもはや組織ではない。それはハーネスである」。

**GrowthFix への翻訳：**
- 直訳すると「組織の引力」思想と矛盾するように見える
- 正しい翻訳は「**組織を最小化して引力を最大化する**」
- GrowthFix の構成（石井さん 1 人＋アシスタント＋業務委託＋ AI）は、まさに「組織ではなくハーネスで回る」最先端事例
- だからこそ Coaching / Shift クライアントに対して「組織 OS 化」を売れる立場

**含意：** GrowthFix の最終形（広告ゼロ・年 5-10 社・少数精鋭）は、ハーネスエンジニアリングの完成形と等価。

### トークン経済（経営指標としての AI 費）

動画で借用された外部視点：

- **AX 福島孝徳**：「人件費と AI 活用費は一緒に見る。5000 万かけるなら 1000 万人＋ 4000 万 AI でもいい」
- **YC AI Native**：「不快になるほどトークン使わせろ」「1 人当たり利益ではなくトークンあたり利益」

**GrowthFix への翻訳：**

| 旧指標 | 新指標 |
|---|---|
| 1 人当たり売上 | トークンあたり売上 |
| 1 人当たり利益 | トークンあたり利益 |
| 月額人件費 | 月額（人件費＋ AI 費）合算 |

**戦略含意：**
- AI 費を「コスト」ではなく「経営戦略指標」として扱う
- 「retail で支払えば月 ¥380 万 → サブスク約 ¥3 万 → **127 倍のレバレッジ**」（260430 月確定値）が GrowthFix の最強の権威構築数値。Sushitech 等の対外発信に組み込む
- 「トークンあたり利益」の分母選定：retail 換算（レバレッジ評価）／実支払（真の ROI）の使い分け

**集計スクリプト：** `06_開発/scripts/token_usage.py`（260429 実装）。詳細は `reference_token_usage_script.md`。

### チェンジマネジメント 3 層（クライアント展開時）

動画の結論：ハーネスエンジニアリングを組織に導入するチェンジマネジメントは 3 段階。

| 段階 | 内容 | GrowthFix Shift / Coaching での実装位置 |
|---|---|---|
| **1. トップの熱狂** | 経営者が AI ×ハーネスの重要性に気づく | Coaching セッションで火を点ける |
| **2. 右腕の覚醒** | カウンターパートが組織内に広める | Shift Week 1-2 診断で「右腕の覚醒度」評価 |
| **3. 仕組み化** | ハーネス更新インセンティブ・トークン使用判定・データ整備 | Shift Week 3-12 で実装伴走 |

**Shift 商品の差別化材料として強力。** 詳細は商品.md ／営業.md ／採用.md。

## Part 2: 実装層（GrowthFix の既存ハーネス実装）

### 既存ハーネス資産

| カテゴリ | 実装 | 用途 |
|---|---|---|

| 失敗テスト蓄積 | `memory/feedback_*` 90+ 本 | 事後の失敗→次回プロンプトへ反映 |
| LP 合格基準 | `_共通/_LP変更時_モバイル確認チェックリスト.md` | 事前定義の運用 |
| **サブエージェント定義**（260503 開始） | `~/.claude/agents/lp-implementer.md` ／ `script-writer.md` | **モデル + system prompt + tools + ハーネス装置を 1 ファイルにパッケージ化。Sonnet 専用エージェントで議論=Opus / コード=Sonnet 分業を物理化** |
| 自動 commit | `06_開発/scripts/hooks/auto_commit.sh`（Stop hook）| セッション終了時に変更があれば自動 commit + push（secrets ガード組込） |

### 警戒事項

- ❌ ハーネスを書くこと自体が目的化する：あくまで実行ガードのための手段
- ❌ 規定の塊化：思想層（Part 1）の魅力を殺してまでハーネス層を増やさない
- ❌ クライアントに「ハーネス書かせてフィー取る」モデル：ツールでなく結果を売る原則違反

### Read 効率化（260501 確立）

ハーネスエンジニアリング全体像を読む時は本 MD を Read。**他 MD（AI.md / 商品.md / 営業.md 等）から「ハーネス」言及がある場合は、本 MD へのポインタとして扱い、他 MD 全文の Read は不要。**
