### ★ 北極星

> **「この判断は、GrowthFix の引力（Why × 才能 × 偏愛）に沿っているか？」**

YES なら推進。NO なら却下。

戦略フレームワークも、コンバージョン 3 条件も、ウニ丼比率も、構造逆算も──全部この北極星の下で動く。

北極星から外れた判断は、どんなに論理的に正しくても採用しない。

「**正しいけど違和感がある**」は、必ず引力との不整合である。違和感を信じる。

### Why（なぜ判断基準を MD 化するか）

判断軸が暗黙知のまま石井さん個人に閉じていると：
- 業務委託パートナーが入った時に再現できない
- AI に相談する時に毎回バラバラに伝える必要がある
- クライアントへの「引力経営の参謀」サービス（Shift）で再現可能性が低い

→ 判断軸を MD で明示すれば、属人性を脱却して**スケール可能な OS** になる。

### 定義

**判断基準＝意思決定の言語化された再現可能パターン。**

GrowthFix では以下の系統で構成：

```
[1] 戦略レイヤー（堀田5原則・コンバージョン3条件）
     ↓ どの方向に進むか
[2] 設計レイヤー（ウニ丼モデル・主軸先行）
     ↓ どう作るか
[3] 運用レイヤー（壁打ち推奨明示・構造逆算・整合性 lint）
     ↓ どう動かすか
[4] 撤退レイヤー（適用外・引き受けない案件）
     ↓ 何を切るか
```

### 中核原則

**判断基準を運用する時の北極星：**

> **「この判断は、GrowthFix の引力（Why × 才能 × 偏愛）に沿っているか？」**

YES なら推進、NO なら却下。迷ったら本ファイルに戻る。

## Part 2: 実装層

### 第一の判断軸：「俺の時間を奪うか」

あらゆるフレームワークの前に、石井伸幸の身体感覚として確立されている判断軸がある。

> **「俺の時間を奪うな、この野郎」**

意思決定の遅い人・行動の遅い人・「念のため」を多用する人と関わると、こちらの自己主権（時間の主権）が奪われる感覚が立つ。

この感覚は怒りではなく、**自己主権侵害への防衛反応**である。Why（自己主権）と直結している。

フレームワークで悩む前に、まずこの感覚で 1 次フィルターをかける。「俺の時間を奪う案件」は、5 原則を満たしていても**引き受けない。**

### 戦略レイヤー：堀田 5 原則

GrowthFix の事業全戦略の根幹。新サービス・新企画は必ず 5 原則で点検する。

| 原則 | 内容 |
|---|---|
| **1. 意外性** | 既存の常識と異なる切り口を持つ（敵を作る戦略含む） |
| **2. 一貫性** | 創業者の Why・サービス・発信に一本の糸が通っている |
| **3. 偏愛** | 創業者が異常に好きで、止められない |
| **4. 翻訳力** | 専門知を顧客の言葉に置き換えられる |
| **5. ストーリー** | 個人体験から商品まで物語化されている |

**運用：** 新企画は 5 原則のうち 3 つ以上を満たしているか確認。満たさないなら採用しない。

### 戦略レイヤー：コンバージョン 3 条件

| 条件 | 内容 |
|---|---|
| **痛みの質** | 顧客が「今すぐ」と感じる痛みか |
| **タイミング** | 顧客が買える状態か（予算・決裁権・緊急度） |
| **代替不可性** | この提供者でなければ駄目という理由があるか |

**運用：** 商談・LP コピーは 3 条件をすべて満たすように設計。1 つでも欠けるとコンバートしない。

### 設計レイヤー：ウニ丼モデル（堀田フレーム）

**原理：自分の高価値時間を最小化し、低コスト労働でパッケージして顧客体験を最大化する。**

GrowthFix 5 サービス × ウニ丼比率（260424 確定）：

| サービス | ウニ（石井） | 丼（AI/業務委託） | 主戦場？ |
|---|:-:|:-:|:-:|
| CODE | 0% | **100%（AI）** | サブ |
| Scan | 0% | **100%（AI）** | サブ |
| Coaching | **80%（石井）** | 20% | 主軸 |
| Shift | **40%** | 60%（業務委託＋AI） | **主戦場** |
| Orbit | 段階移行 60→40 | 40→60 | 中軸 |

**最重要未決事項：** 業務委託チーム組成（Shift / Orbit の「丼」担い手）

### 設計レイヤー：コンセプトは対立軸から生まれる原則（260428 縦断分析で確立）

**ルール：新コンセプト・サービス名・ポジショニング言語は、肯定の積み上げではなく既存概念の対立軸として立てる。**

**Why：** GrowthFix の主要コンセプトはすべて**否定から生まれている**：

| コンセプト | 否定された対立概念 | 立ち上げ時期 |
|---|---|---|
| **事業中心人事** | HRBP（中原淳定義「事業部を元気にする支援者」） | 2025/12/24 |
| **引力経営** | 識学（恐怖駆動・権限ピラミッド） | 2026/04/23 |
| **組織を変えるな。引力を作れ** | 制度先行・組織開発手法 | 2026/04 |
| **「抜く」翻訳者** | 「足す」コーチ／コンサル | 2026/04 |
| **Gravity Shift（80 万・3 ヶ月）** | 1,000 万円規模の組織コンサル | 2026/04/22 |
| **「批判覚悟・尖り路線 4.3」** | 安全路線の 4.7（万人受け） | 2026/04/24 |

肯定だけで作ったコンセプトは「いい話・賛同される・忘れられる」状態になる。**対立軸を明示することで顧客の認知に刻印される**（堀田 4/15 指摘「敵を作らないコピーは届かない」と同根）。

**運用フロー：**

```
[1] 立ち上げたいテーマを置く
   例：「経営者の引力源を組織に翻訳する」
        ↓
[2] そのテーマで世の中で支配的な既存概念を特定
   例：HRBP・組織開発・識学・コーチング流派
        ↓
[3] 既存概念の限界・矛盾・退屈さを言語化（敵化ではなく構造批判）
   例：HRBP は分離前提・繋ぐ機能・経営しない
        ↓
[4] 対立軸として新コンセプトを立てる
   例：「事業中心人事は経営する／繋ぎではない」
        ↓
[5] 既存概念のディスを通じて新概念の輪郭を浮き彫りにする
```

**禁則：**
- ❌ 既存概念に「+α」する形でコンセプトを立てる（差別化が弱い）
- ❌ 個別批判（人格攻撃・特定企業のディス）
- ✅ 構造批判（既存概念の方法論的限界・パラダイムの古さ）

**ガードレール：**
- 対立軸を立てる時は「**自分が共鳴する思想の延長線にあるか**」を確認（識学への対比は GrowthFix の引力経営と接続している）
- 対立軸を作っても本人の主軸（思想・Why・偏愛）からズレない（主軸先行ルールと両立）
- 関連 memory：`feedback_edge_path_43.md`（尖り路線採用の判断軸）／`feedback_embrace_criticism_and_exaggeration.md`（批判受容のメンタル）

### 戦略レイヤー：採用ペイン主軸原則（260429 堀田 4 回目セッション・最重要）

**ルール：GrowthFix の対外メッセージング（LP・セミナー・Note・WP・営業）すべての最終接地点を「採用うはうは・離脱止まる」具体ベネフィットに統一する。組織抽象論はトップランディング（入口）のみ・中身は採用具体に必ず接地させる。**

**Why：** 260429 堀田指摘「経営者の癖なんだろうけど、悪徳以外のことにあんまり興味ない。究極なので、人材獲得の話以外って理を感じにくい」。中堅経営者の真のペイン＝**「優秀な人が雇えない／辞めていく」**。組織課題はこのペインに宿っている。

**判断軸：**
- ✅ そのコピー／タイトル／LP は「採用」「離脱」「優秀な人材」のいずれかに接地しているか？
- ✅ 抽象論（思想・引力・組織変革）が中央に置かれていたら、**必ず採用ペインへの橋渡し**を入れる
- ❌ 「あったらいいな」レベルの抽象的ベネフィット
- ❌ 採用・離脱を裏方扱いにしたコピー

**運用：** memory `feedback_recruitment_pain_first.md`／会社 OS `商品.md`「採用ペイン軸への主軸転換」セクション

### 戦略レイヤー：スピーカー偏差値原則（260429 堀田指摘）

**ルール：セミナー・登壇・対談・コンテンツ企画時、タイトル候補に対して「自分が喋るときに割が良いか」を点検する。同業の有名人 3 人と比較してスピーカー偏差値が高いものだけ採用する。**

**Why：** 260429 堀田指摘「これ全く同じタイトルだったら、安宅さんとかが喋った方がいい。より面白そう。だから安宅さんに比べると、ボブはちょっと割が悪い」。タイトルが思想・抽象寄りだと有名思想家の方が割が良くなる。**石井さんの DMM 50 事業 HRBP / MoonX 人事 16 年が活きる切り口**を選ぶ。

**判断軸（企画時のチェックリスト）：**

```
[1] このタイトルを聞いて、聴衆は誰のセミナーだと思うか？（同業の有名人を 3 人挙げる）
[2] その 3 人と比べて、石井さんが喋る方が「面白そう」「割が良い」と感じるか？
[3] 感じない場合、タイトルを変える（または企画自体を捨てる）
```

**判定例（260429 堀田直接指摘）：**
- ❌ 「仕事ができる経営者ほど、幹部が話を合わせる理由」（安宅さんの方が割が良い）
- ✅ 「ベンチャー社長が採用でやってしまう、三つの気持ち悪いところ」（人事担当者経験フック・石井さんが圧倒的優位）

**運用：** memory `feedback_speaker_index_principle.md`／全セミナータイトル・登壇企画で必ず適用

### 戦略レイヤー：えげつない具体 > 抽象論（260429 堀田指摘・AI 時代の差別化）

**ルール：Note・WP・LP・セミナーすべて「えげつない具体」（言っちゃいけないやつの言葉レベル）を主軸に置く。トップランディング（入口）でのみ抽象 OK・中身は必ず具体。**

**Why：** 260429 堀田指摘「えげつない具体とかが最近は起きると思いますよ。言っちゃいけないやつの言葉ぐらいの話。抽象論の方が手に入っちゃう（AI 量産）」。

**コンテンツの 3 層構造：**
```
[トップランディング層] ← 抽象 OK（思想・興味喚起）
  ↓
[コア層] ← えげつない具体（実装経験から書く・他では聞けない）
  ↓
[CTA 層] ← 具体ベネフィット（採用うはうは等）
```

**判定基準：** 「これは言っちゃいけないやつの言葉だな」と感じるかが基準。
- ✅「面接でこう質問する経営者は採れない（具体的な NG 質問 3 つ）」
- ❌「組織には引力がある」（抽象・誰でも書ける）

**運用：** memory `feedback_egetsunai_specific_over_abstract.md`／`feedback_speaker_index_principle.md` と整合

### 戦略レイヤー：フェーズ判定原則（供給→需要転換・260428 確定）

**ルール：事業の主問題が「供給側（作れているか）」か「需要側（届いているか／買われているか）」かを四半期ごとに判定し、フェーズに応じて時間配分・計測軸・運用を切り替える。**

**Why：** 事業フェーズが変わると最適な行動が逆転する。供給側フェーズでは「作る・磨く」に時間を全力投下することが正解だが、商品が完成した後にも同じ時間配分を続けると**反応・受注が出ないのに「もっと作れば」で逃げる**慢性パターンに陥る。260428 時点で 5 サービス・19 LP・WhitePaper V9 が完成し、需要側フェーズ（マーケ・広報・営業集中期）に移行した。フェーズ転換の認識が遅れると AI レバレッジが効かない領域（人間関係・対話）に時間配分を寄せられず、CF 機会損失が累積する。

**フェーズ判定軸：**

| 観点 | 供給側フェーズ | 需要側フェーズ |
|---|---|---|
| 主問題 | 作れているか／質はどうか | **届いているか／買われているか** |
| 朝の使い方 | 制作（LP・コピー・WP・診断） | **リーチ＋対話（DM・投稿・パイプ更新）** |
| 計測指標 | 完成度・質感 | **反応率（CTR・DL率・返信率・商談化率・受注率）** |
| AI レバレッジ | 8 倍（拡張効く領域） | 2-3 倍（人間関係・対話は拡張効きづらい） |
| 時間配分 | インプット 8：アウトリーチ 2 | **インプット 2：アウトリーチ 8** |

**フェーズ転換時の運用切り替え：**

```
[判定] 四半期ごと or 大きな完成イベント時に主問題を再点検
   ↓
[切替] フェーズが転換したら以下を同時に移行
   ├─ 朝のルーティン（制作 → リーチ）
   ├─ 計測指標（完成度 → 反応率）
   └─ 仕組み（営業.md「フェーズ運用」発火）
```

**禁則：**
- ❌ 反応率が出ないのに「もっと作れば」で供給側に逃げる
- ❌ フェーズ転換を意識せず過去の慣性で動く
- ❌ 11 指標を全部追う（1 点ボトルネックに集中する）

**詳細運用：** `09_会社OS/非公開/機能/営業.md`「フェーズ運用」セクション
**関連 memory：** `feedback_phase_transition_marketing_to_sales.md`

### 運用レイヤー：時間制約セッション・目的決定優先原則（260429 確定）

**ルール：時間制約のあるセッション・会議・対話は、目的を最初に決めてからコンテンツを作る。**

**Why：** 60 分や 90 分の制約があるセッションで「全活動の網羅報告」「全戦略決定の整理」「全課題の棚卸し」をやると、肝心の論点に時間が足りなくなる。**目的を 1 行で言語化してから、それに関連するコンテンツだけを揃える**ことで、限られた時間の集中度を最大化する。

**適用対象：**
- 堀田コーチングセッション（60 分）
- コーチングクライアントセッション（60 分・週次/隔週）
- 対談・取材・登壇前の準備
- 対外的な打ち合わせ・商談（30-60 分）

**運用フロー：**

```
[1] セッションの目的を 1 行で言語化
    例：「新規集客の壁打ちと売上を作っていくステップのアラインメント」
        ↓
[2] 目的に関連する論点を 3-5 件に絞る
        ↓
[3] 各論点に対する数値・現状・問いを構造化
        ↓
[4] それ以外の論点は意図的に削る（別の機会に持ち越す）
```

**禁則（やってはいけない）：**
- ❌ 「全活動を網羅報告したくなる」誘惑に流される
- ❌ タイムライン・全戦略決定・全課題の棚卸しを目的に関係なく書く
- ❌ 問いを 7-10 問詰め込む（1 問あたり 10-15 分の議論時間が必要）
- ❌ 「念のため書いておく」で削れない情報を膨張させる

**ガードレール：**
- 重大決定・問い・指標は **3-5 件**で止める
- 「目的に関連するか？」を各セクションで自問
- 60 分セッションなら**問いは最大 5 問**まで（10-12 分 × 5 問）

**関連テンプレ：** `04_GrowthFix/01_サービス設計/_template_堀田ディスカッション資料.md`（堀田セッション用に最適化済・他セッションにも応用可）

**Why この原則を会社 OS に刻むか：** 60 分という制約は構造的に変わらない。毎回ゼロから「何を持っていくか」を考える運用は非効率かつブレやすい。**「目的決定→コンテンツ設計」の順序を判断基準に固定化**することで、毎回のセッション準備が 60-90 分で完成する仕組みになる。

### 運用レイヤー：壁打ち推奨明示プロトコル

**問いを投げるとき、毎回セットで提示する：**

```

```

**Why：** 判断高速化のため。問いだけだと選択肢の網羅性が分からず、選択肢だけだと推奨がブレる。

### ★ KISS 原則 ／ 情報の削ぎ落とし価値（260503 夜 確立・石井伸幸の傾向自己認識）

**ルール：LP・名刺・コピー・対外発信物は KISS（Keep It Simple, Stupid）原則に従う。発散して選択肢を出し切った後、極力シンプルに無駄を省く。**

#### 中核原理：情報は削ぎ落とされた情報の総量に価値がある

情報理論的洞察として、最終アウトプットの価値は **「載せた情報量」ではなく「載せなかった情報量（削ぎ落とした総量）」**で決まる。

- 多くを検討し尽くすプロセス（発散）= 価値創出の前提
- 極力シンプルに無駄を省く（収束）= 最終価値の実現
- **「発散して選択肢を出し尽くす」と「シンプルに削ぎ落とす」は同じ価値創造の表裏**

#### 石井伸幸の傾向自己認識（260503 言語化）

> **「不安から情報を盛り込みすぎる傾向がある」**

過去事例：
- **名刺**：当初は情報を盛り込みすぎていたが、最終的に大幅削減した結果とても良くなった
- **LP（260501-260503）**：シャープ化哲学で 1 文 SSOT に研ぎ澄ましたが、49 質問詳細・3 軸ハイブリッド・採用コスト試算等を LP に盛り込みすぎ → メンテ工数複利問題
- 一般化：**「載せれば伝わる」「載せないと不安」が罠**。実際は「削げば伝わる」「載せると埋没する」

#### 運用プロトコル（260503 確立・LP / 提案書の役割分担）

```
[1] 発散フェーズ（足し算・既存）
    ↓
    49 質問・3 軸ハイブリッド・採用コスト試算 等で全要素を出し切る
    ↓
[2] 収束フェーズ（引き算・260503 移行）
    ↓
    a. 営業提案書 v1 を「全差別化資産の受け皿」として先に作る
    b. LP は営業提案書から「経営者が 30 秒で判断できる必要最小限」のみ厳選
    c. LP に出さなかった要素 = 営業提案書 + 商談で見せる
    ↓
[3] 信頼供給装置の分離（事例 3 件以上後）
    ↓
    LP = 興味喚起（minimal）
    事例コンテンツ = 信頼供給（量産）
    営業提案書 = クロージング（詳細）
```

#### 判定原則：「LP に載せるか／削るか」の問い 3 つ

1. **30 秒で経営者が「これは自分の話だ」と判断できる要素か？**（Yes → LP 載せる／No → 営業提案書へ）
2. **営業時に説明する内容を LP で先取りしていないか？**（Yes → 削る・LP は興味喚起に専念）
3. **3 ヶ月後に変更する可能性が高い情報か？**（Yes → LP に固定化せず営業資料で運用）

#### 既存原則との関係

- **シャープ化哲学（260501）**：1 文 SSOT への研ぎ澄まし → KISS 原則の前段・素材作り
- **minimal LP 戦略（260503 朝）**：Orbit / Shift（複合）/ Coaching の minimal 化 → KISS 原則の Push/Pull 別実装
- **本原則（260503 夜）**：5/8 モニター後の事例コンテンツ蓄積に伴って **全 LP の minimal 化判断**を行う上位原則

#### 実装ロードマップ（260503 確定・実行は 5/8 後）

| Phase | 時期 | 状態 | アクション |
|---|---|---|---|

### ★ Lean MVP コンセプトテスト原則（260501 夜 確立）

新サービス・新訴求の検証は「実証セッション」より**「3 案 × N サービス = N × 3 VP コンセプトテスト」**で本能的反応を取る方が ROI 高い。

**理由：**
- 1 案だけ持ち込むと「滑るリスク」が高い
- 3 案でベネフィット軸（コスト訴求 / 時間訴求 / 品質訴求）を比較すると経営者の真因痛みが浮かぶ
- N サービスで R/C 等の対称構造を持ち込めば 1 セッションで複数サービス分の改善ヒント取得

**実装パターン（260501 5/8 長谷さんモニター）：**
- CODE 60 分 + 6 VP 提示（5 分 × 6 = 30 分）+ フィードバック収集 30 分 = 計 2 時間 10 分
- マクロ経済 × 経営者個人 × 採用品質の 3 角形 × R/C の 6 マトリクス
- フィードバック収集シート 10 質問（直感選定 / 理由 / 価格感 / 購入意欲 / 不足要素）

詳細：`04_GrowthFix/01_サービス設計/260508_長谷さんモニター_R_A統合_6VPテスト資料一式.md`

### ★ サービス軸性質判定（Push vs Pull・260502 確立）

5 サービスを **funnel 上の役割別に「軸性質」で分類**し、1 文 SSOT の方向性とブラッシュアップ手法を分ける。

| サービス | 1 文軸 | 性質 | 集客戦略 |
|---|---|---|---|
| CODE | 採用接地（B 軸：面接で口説けない）| **Push** | 広告・LP・市場接地最強訴求 |
| Scan | 採用接地（B 軸：採用に払っても辞める）| **Push** | 同上 |
| Shift R | 採用接地（B 軸：吸い寄せられる）| **Push** | 同上 |
| Shift C | 制度疲弊（B 軸：施策疲弊からの解放）| **Push** | 同上 |
| **Coaching** | **演じる構造（A 軸）** | **Pull** | **「売り言葉にしない」原則・紹介と求心力で年 5-10 社** |

**運用ルール：**
- **Push 型サービス**：funnel 入口・市場接地を最大化・採用ペイン具体起点で訴求（「えげつない具体」原則）
- **Pull 型サービス**（Coaching）：能動誘導 NG（「Coaching どうですか？」「Coaching 30 分無料相談」禁則）。CODE → Scan → Shift の王道フローと並走する個人軸として、関係性の中で自然着地

**判定ガイド：**
1. **funnel 上の位置**：入口〜中段は Push、本命終着点は Pull
2. **顧客が能動的に欲しがるか**：能動的に欲しがるサービス → Push可、能動的に欲しがらないサービス → Pull 必須
3. **広告で売れるか**：広告クリエイティブが書けるか・書けないなら Pull 設計
4. **既存クライアントの獲得経路**：紹介・知人ルート優位なら Pull、広告・LP 優位なら Push

詳細：`04_GrowthFix/01_サービス設計/260502_Coaching_シャープ版_1文_5論点確定.md` ／ `memory/feedback_coaching_not_frontline_word.md`

minimal LP 化を判定する時の問い：

1. **顧客ジャーニーは単軸ピン痛で来店するか？**（Yes → minimal 化候補・経営者は「複合パッケージ」を能動検索しない）

### ★ ブラッシュアップの境界線（思想書 vs 採用ペイン具体起点・260502 WhitePaper LP 事故教訓）

LP・WP・コピーのブラッシュアップで「**ブラッシュアップしすぎて魂を失う**」事故を防ぐ境界線判定。

**事故パターン（260502 WhitePaper LP）：**
- Before：「引力経営 ── 社長がいなくても、すべてが回る組織の原理。」（思想書体裁）

- 結果：思想書として読まれていた魂が消失・「具体によりすぎて全く面白味がない」状態に

**境界線判定ルール：**

| LP / WP の役割 | 推奨軸 | 理由 |
|---|---|---|
| **funnel 入口 LP**（CODE/Scan/Shift R/C）| **採用ペイン具体起点** | 市場接地最大・コンバージョン優先 |
| **WhitePaper / 思想書 / Hub LP**（gravity 全体・whitepaper オプトイン）| **思想書体裁** | 「経営の原理を学ぶ」入口・読者の知的興味を起点 |
| **Pull 型サービス LP**（Coaching）| **構造起点（演じる等）** | Push 型と区別・売り言葉ゴリ押し回避 |
| **コーポレート TOP**（/）| **会社思想 + 6 サービス導線** | 信用供給・ブランド構築 |

**判定の問い 3 つ：**
1. 「これは funnel 入口か、それとも本命/思想書か？」── 入口なら Push 軸、本命なら Pull or 思想軸
2. 「読者の脳内モードは『コンバージョン候補』か『学習者』か？」── 学習者モードなら思想書を捨てない
3. 「ブラッシュアップ後、3 ヶ月前のファンが『これじゃない』と言う可能性は？」── 高ければ魂消失リスク、Hero の前後比較で判定

**運用ハーネス：**

### ★ SSOT 更新範囲の判定原則（260502 ファイナンス横断更新セッション後 確立）

「SSOT が変わった時、過去のドキュメントをどこまで更新するか」の判定原則。**過剰更新を避け、Historical Snapshot を保持する**ことで、進化の軌跡が追える状態を維持する。

#### 更新対象 vs 触らない対象

| ファイル種別 | 更新ポリシー | 理由 |
|---|---|---|
| **09_会社OS/** | ✅ **常に最新（SSOT）** | 全社運用のリアルタイム参照源 |
| **05_プロダクト/_共通/SSOT_用語と定義.md** | ✅ **常に最新（用語 SSOT）** | LP・WP・営業会話の SSOT |
| **07_マクロ経済分析/260404_統合タイムライン_*.md** | ✅ **継続参照される計画 SSOT** | 9 年計画として大変更時に反映 |
| **260YYMMDD_「最新」「マスター」明示ファイル** | ✅ **名前通り最新化**（260427_最新統合プロファイル等）| 役割が「最新を保つ」と明示 |
| **memory（feedback / project / reference）** | ✅ **積み重ね型・適宜追記/更新** | Claude memory システムの設計通り |
| **260YYMMDD_ 時系列ドキュメント**（バランスホイール / セッション記録 / WP 旧版 等） | ⏸ **Historical snapshot として原則触らない** | 「いつ何が決まったか」の進化軌跡を保持 |
| **デイリーログ / work_log** | ⏸ **その日のスナップショット・後から修正しない** | 過去の認識を固定する記録装置 |

#### 判定 3 問（運用時）

「このファイルを更新すべきか？」の判定：

1. **このファイルは継続的にリファレンスされる SSOT か？** → Yes なら更新
2. **ファイル名に「最新」「マスター」「SSOT」等が含まれるか？** → Yes なら更新
3. **時系列ドキュメント（260YYMMDD_）か？** → Yes なら原則触らず・必要なら新ファイル化 or 09_会社OS にサブ SSOT を作る

**判定不明時：** 09_会社OS 内に類似 SSOT があれば、そちらを更新して時系列ドキュメントは触らない。なければ 09_会社OS にサブ SSOT MD を新規作成。

#### Why（過剰更新を避ける根本理由）

- **過去ドキュメントの書き換えは「歴史改竄」になる** ── 260404 時点の「金 2 億」を 260502 に「金 5,539g 評価 1.5 億」に書き換えると、「いつ評価額の認識が更新されたか」が消える
- **進化の軌跡が見えなくなると、判断軸の進化も追えなくなる** ── 260415 深掘り → 260427 再統合 → 260502 SSOT 化の流れが、過去ドキュメントを残すからこそ追える
- **メンテ工数が指数的に増える** ── SSOT 1 件変更で N 個の過去ドキュメント更新は持続不能

#### 例外運用（過去ドキュメントを更新する場合）

- **戦略的判断軸が大幅に変わった時**（例：ファイナンス目標の桁が変わる）── 9 年計画の SSOT として継続参照されるファイルのみ更新
- **過去ドキュメントの誤情報が現在の業務に影響する時**（例：廃止用語が LP に残存）── 機械的整合のための部分修正
- **「最新版」明示ファイル**（260427_最新統合プロファイル等）── 名前通り最新化が役割

### ★ git 履歴運用（5 ヶ月放置事故対策・260502 確立）

**事故事実：** 2025-12-26 〜 2026-05-02 の **5 ヶ月間 git commit が一度も実行されておらず**、WhitePaper LP の「昨日のブラッシュアップを巻き戻したい」要望に対して **git からのロールバックが不可能**だった事故。

**ハーネス装置（260502 実装）：**

1. **`auto_commit.sh`（Stop hook）**：Claude セッション終了時に未コミット変更があれば自動 commit
   - secrets 検出ガード組み込み（`{FTP_PASSWORD}` / `sk-ant-` 等検出時は commit 中止）
   - JST タイムスタンプ + 主要編集ディレクトリで commit message 自動生成
   - **自動 push する**（260503 更新・Stop hook は単発発火のため loop 化しない・GitHub Push Protection が secrets 二重防御）／push 失敗時は warning のみ（exit 0 で hook を阻害しない・手動 `git push origin main` でリカバリ）
   - 配置：`06_開発/scripts/hooks/auto_commit.sh`

2. **`.gitignore` 防御パターン拡張**：
   - `.claude/`（Claude Code 設定）
   - `**/config.php`（API キー含む設定）
   - `_archive/` ／ `**/_archive/`（過去構成）
   - `06_開発/開発ツール/FTP情報_メインFTPアカウント.md`（FTP 認証単一情報源）
   - `06_開発/scripts/config_*.json`

3. **GitHub Push Protection 活用**：secrets を含む push は GitHub 側で自動ブロック（保険として優秀）

**運用原則：**
- 自動 commit は装置に任せる・**手動 push は週次 or 重要マイルストーン時**
- secrets を含むファイルは commit 前に必ず .gitignore で防御
- 「ロールバック可能性」がブラッシュアップの心理的安全性を支える ── git なしのブラッシュアップは事故の温床

詳細：`memory/feedback_lp_mobile_audit_required.md`（hook 設計パターン参照）／ `04_GrowthFix/04_デイリーログ/260502_daily.md`

### 良いアウトプットの合格基準（Pass条件）

新規案件・新サービス・大型支出・撤退の意思決定が以下を**全て**通過する：

- [ ] **CF 優先順位**：CF 安定前の流動性低い投資になっていないか
- [ ] **「俺の時間を奪うな」体感**：判断後 1 週間後の予定を見て、自分の高価値時間が削られない設計か
- [ ] **顕在ニーズあるか（260430 夕 Scan リブート学び）**：サービス命名・主訴求が「顧客が顕在的に検索する言葉」になっているか／「○○が欲しい」と顕在的に思う経営者が思い浮かぶか／思い浮かばないなら戻る
