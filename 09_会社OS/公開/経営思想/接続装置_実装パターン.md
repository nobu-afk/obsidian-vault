### ★ 商品名・納品物名の二段運用（260501 確定・260503 サービス名にも拡張）

接続装置の二段運用原則は LP/コピーだけでなく**サービス内の納品物・概念名**、さらに **260503 サービス名そのもの**にも拡張される。

#### 1. サービス名の二段運用（260503 B 案確定）

| 軸 | 内的呼称（社内・SSOT・09_会社OS）| 外的呼称（顧客接点・LP / WP / Note / セミナー / メール / 見積書）|
|---|---|---|
| 採用基盤変革 | Shift R | **Gravity Recruit** |
| 躍動組織変革 | Shift C | **Gravity Cultivate** |
| 採用 × 躍動 複合 | Shift Full | **Gravity Shift**（R+C 複合パッケージ）|
| 承継変革（将来）| Shift S | Gravity Succession |

**Why：** "Shift" は内部設計言語（変革プロセスの符号化）、"Gravity ◯" は顧客接点言語（経営者が動詞で理解できる）。本接続装置原則「外的入口→内的体験」と完全整合。詳細：`memory/project_naming_two_layer_260503.md` ／ SSOT_用語と定義.md §0.5

#### 2. 納品物名の二段運用（260501 確定・既存）

**Shift R/C の Week 1-2 標準納品物**は、外的呼称と内的呼称を分離して運用する：

| レイヤー | 用語 | 使用場面 |
|---|---|---|
| **外的（顧客向け）** | **「3 ヶ月予言の書」**（採用基盤 / 躍動組織）| LP・セミナー・商品名訴求・営業会話・契約書見積書 |
| **内的（社内・参入障壁）** | **「ハーネスエンジニアリング」**（採用ハーネス / 躍動ハーネス）| 09_会社OS・商品.md・Shift 受講中の深掘り説明・業務委託パートナー教育 |

**Why（260501 発見）：**
- 「ハーネスエンジニアリング」は石井独自概念で参入障壁性は高いが、経営者が顕在的に理解できない（顕在ニーズゼロ語）
- 「予言の書」は経営者が「事前に気づきを設計する 80 万円」という理論的根拠を一語で理解できる接地語（RECODE C-2「研修成果の 90% は事前設計」と整合）
- 両者を同じものに統一すると、参入障壁か接地性のどちらかを失う → **二段運用で両立**

**運用ルール：**
- LP・セミナー・営業会話・見積書・契約書・受講中の経営者対話 → **必ず「予言の書」で語る**
- 09_会社OS・商品.md Part 2-3・業務委託パートナー教育材料・社内 SSOT → **「ハーネス（エンジニアリング）」呼称を維持**
- Shift 受講中の経営者説明で「ハーネス」を使うのは禁止（外的領域に内的用語が漏れる）
- 09_会社OS から「ハーネス」が完全に消える編集も禁止（参入障壁を失う）

**詳細：** `memory/project_shift_rc_specs_260501.md` D-F セクション ／ `09_会社OS/公開/ガイドライン/商品.md` Part 2「ハーネス用語の二段運用」

### ★ ハイブリッド配置パターン（260502 確立・/sharpen × 5 サービスで標準化）

シャープ化で確定した 1 文 SSOT を LP に配置するときの **3 軸ハイブリッド配置パターン**を標準化。1 案だけでは捨てる軸が出るため、3 案を LP の異なる場所に配置することで「市場接地 + funnel 整合 + ROI 補強」を全部成立させる。

**標準パターン（CODE / Scan で確立）：**

| 軸 | 配置場所 | 役割 |
|---|---|---|
| **B（採用接地・市場接地最強）★ 主軸** | Hero h1 ＋ description meta | 1 文 SSOT・市場入口の症状起点 |
| **A（既存軸・本質）** | Sub Hero | サービス本質の継承・既存ファン継承 |
| **C（構造起点・連続フロー）** | Pricing 直前 ROI 補強セクション | 「N 万投資で M 万方角確定」のメンタルモデル形成 ＋ **必ず信用装置（ケース）と併記**（260507 確定・TASK-10）|

**例：Scan の場合（260504 全 Hero 刷新後）**

**例：CODE の場合（260504 全 Hero 刷新後）**

- B：Sub Hero「経営者の引力源は、採用・組織・発信のすべてを変える」（採用接地ベネフィット）

### ★ funnel 全体の軸統一原則（260502 確立・採用接地統一 + Pull 軸独立）

5 サービスの 1 文 SSOT は **funnel 上の役割で 2 グループに分離**：

| グループ | サービス（内的呼称）| サービス（外的呼称・LP / 顧客接点）| 1 文軸 | 性質 |
|---|---|---|---|---|
| **採用接地統一群** | CODE → Scan → Shift R / Shift C | Gravity CODE → Gravity Scan → **Gravity Recruit / Gravity Cultivate** | 採用接地（B 軸）| Push 型 |
| **Pull 独立軸** | Coaching | Gravity Coaching | 演じる構造（A 軸）| Pull 型 |

**funnel 全体ストーリー（260504 全 Hero 刷新後・本質 → 実装 → 進化の 3 段階）：**
```
Gravity CODE：自分の引力源を、60 分で丸裸にする（個人軸の本質診断）
  ↓
Gravity Scan：自社の引力タイプを、60 分で丸裸にする（組織軸の本質診断）
  ↓
Gravity Recruit：優秀人材が吸い寄せられる会社を、3 ヶ月で実装する（採用基盤実装）
  ↓
Gravity Cultivate：優秀人材が躍動する土壌を、6 ヶ月で本格実装する（躍動組織実装・260511 拡張）
  ↓
Gravity Shift：優秀人材が吸い寄せられ、躍動し続ける会社を、6 ヶ月で実装する（両軸複合）
  ↓
Gravity Orbit：優秀人材が集まり続け、躍動し続ける会社を、月次で進化させる（継続運用）
```
→ **動詞 3 段階で funnel が言語的に統一**：丸裸にする（診断）／実装する（実装）／進化させる（継続）。各段階の Sub に採用接地ベネフィット（採用・育成・離職対策の変化）を配置することで、funnel 全体で経営者の「自分の話」感を維持。

**Coaching が独立軸である理由：**
- 「Coaching は売り言葉にしない」原則（260429・feedback memory）── 経営者が能動的に「コーチング受けたい」と思わないため
- 既存 3 名運用パターン（紹介・知人ルート）が示す Pull 性質
- Coaching が本命の終着点 ── 広告ゼロ・紹介と求心力で年 5-10 社を狙う構造

**運用ルール：**
- 新サービス追加時、まず「Push か Pull か」を判定（判断基準.md「サービス軸性質判定」参照）
- Push 型なら採用接地統一群に組み込む・1 文 SSOT を採用ペイン具体起点で書く
- Pull 型なら独立軸として設計・売り言葉ゴリ押し回避

### ★ サービス三位一体 SSOT 整合パターン（260507 v5.3.4 確立・CODE 起点）

**定義：** 1 サービスの SSOT 整合は、**LP（外的）／ 診断 UI（接点）／ 生成物・レポート（内的）** の 3 接点で**同一思想軸**を一貫させて初めて完成する。1 接点だけ修正しても、他 2 接点が古ければ顧客体験は分断される。

**CODE 三位一体の実装パターン（v5.3.4 達成）：**

```
[外的・LP]                  [接点・診断 UI]              [内的・レポート]
/gravity-code/              /gravity-code/executive/     YOUR GRAVITY CODE PDF
  Hero 直下「引力とは」  →   STEP 1 接続装置メッセージ  →  Cover Page「引力とは」
                              「採用 / 組織 / 発信 3 軸」     Block C「経営インパクト 3 軸」
                              バッジ                          Before / After カード
```

**3 軸の一貫性（採用 / 組織 / 発信）：**
- LP Hero 直下：3 軸の症状を太字で明示（「採用では…」「組織では…」「発信では…」）
- 診断 UI STEP 1：3 軸バッジ（ゴールドラベル + 症状）
- レポート Block C：採用 / 組織 / 発信 × Before / After で具体ベネフィット

→ 商品.md SSOT 1 文「経営者の引力源は、**採用・組織・発信**のすべてを変える」を **3 接点全てで装置化**。1 接点だけだと「経営者個人の話」で完結してしまうが、3 軸が三位一体で機能して初めて経営インパクトが伝わる。

**運用ルール：**
1. **新サービス追加時**：LP / 診断 UI / 生成物の 3 接点を**同時設計**（後追い反映は SSOT 違反リスク）
2. **既存サービス改修時**：1 接点を変更したら他 2 接点を**24 時間以内に同期改修**
3. **公開語彙統一**（商品.md line 522）：3 接点全てで「動詞→動き」「環境→発揮される場」「根源動機→Why」を機械的にチェック（lint 対象）
4. **三位一体検証**：改修完了時、3 接点の主要訴求文を並列読みして「同じ思想か」を確認

**他サービスへの水平展開対象（260507 引き継ぎ書 v1.0 で別 Node 実行設計済）：**
- Scan：`/gravity-scan/` + `/gravity-scan/diagnose/`（推定）+ Scan レポート（generate.php）
