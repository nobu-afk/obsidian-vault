# スカウト分析エージェント v0.2（DMM 実例 31 文面反映版）

**v0.1 → v0.2 主要追加点：**
- 7 テンプレート Gem 定義（DMM 実測返信率 23.5-71% のパターン実例化）
- 導線フレーム明文化（タイトル → 冒頭 → 個別化 → 自社紹介 → CTA）
- 採用ペルソナ入力フィールド追加
- ファネル全体追跡出力（返信率 → 書類通過 → 1 次 → 最終 → 内定 → 決定）
- A/B テスト変数定義（タイトル型 / 文面長 / 担当者属性）

---

## 用途

ビズリーチ / Wantedly / LinkedIn 等の **スカウトメール送信履歴 + 返信履歴**を分析し、(1) 返信率分析 / (2) 反応パターン抽出 / (3) 次月スカウト戦略最適化提案 を行う Recruit 専用エージェント。

## 9:1 二層分離原則の適用

| 層 | 比率 | 適用 |
|---|:-:|---|
| **裏方** | **AI 9 : 経営者 1** | 返信率集計 / 反応パターン抽出 / 文面 A/B 比較 / 次月スカウト戦略案生成は AI |
| **表方** | **経営者 9 : AI 1** | スカウト戦略の選択 / 文面の最終決定 / 個別候補者へのカスタマイズは経営者 |

## 7 テンプレート Gem 定義（v0.2 核心）

DMM 実例 31 文面の実測返信率から導出。各 Gem は「構造テンプレ + 使用条件 + 実測返信率帯 + NG 例」の 4 要素で定義。

### gem-01：担当者名乗り + 個別スキル引用型

**推奨対象**：エンジニア（ミドル〜シニア全般）
**実測返信率**：8-31%（最高は個別化精度が高い場合）

**構造：**
```
[タイトル] 【具体ポジション名】 + 御社に魅力を感じた理由一言
[挨拶] 突然のご連絡失礼いたします。
[名乗り] [会社名] [事業部名] [役職] の [担当者名] と申します。
[個別引用] [候補者]様の[具体スキル X / 技術 Y]のご経験に魅力を感じました。
[接続] これらのご経験に加え、[志向性] から、弊社[チーム名]にてご活躍いただけるのでは、と思いました。
[セパレータ] ————
[自社紹介] （1-2 段落）
[CTA] 一度お話しいただけないでしょうか？
```

**NG 例**：
- 汎用タイトル「新規事業立ち上げができる人材を募集しております」（実測返信率 9.1%）
- スキルをそのままコピペ転記（コピペ感で返信率が下がる）

---

### gem-02：超短文カジュアル接触型

**推奨対象**：アクティブ候補者全般（転職意向ありの候補者）
**実測返信率**：23.5%（DMM 実例 233 字・6 行）

**構造（233 字・6 行を上限）：**
```
はじめまして、[会社名] の [担当者名] と申します。
ご経歴を拝見し、一度お話をしてみたくご連絡させていただきました。
[自分が今探している人物像を一言で]
転職のご意思はなくても構いません。お互いの状況についてカジュアルに情報交換できれば。
求人例を添付しておきます。他ポジション多数ございます。
ご検討の程よろしくお願い申し上げます。
```

**設計原則**：
- 個別化ゼロでも可（カジュアル接触ハードル低下が返信の主因）
- 「転職意思なしでも OK」を明示してガードを下げる
- 求人詳細は「添付参照」に委ねて本文をシンプルに保つ

---

### gem-03：候補者プロフィール先行引用型

**推奨対象**：セキュリティエンジニア / スペシャリスト系（専門性を強調したい場合）
**実測返信率**：8-16%（ポジション種別による変動大）

**構造：**
```
評価ポイント
  [スキル名]：[年数]年 / [技術スタック]
  [強みの記述（職歴書より転記）]

[候補者]様
[以下、gem-01 と同構造]
```

**使用条件**：
- 候補者のスキルを転記する際は、機械コピーでなく一言コメントを付加すること
- セキュリティ系・スペシャリスト系で有効。汎用エンジニアには gem-01 を優先

---

### gem-04：ハイレイヤー限定感・役職者接触型

**推奨対象**：事業開発 / ハイレイヤー Biz / CxO 候補
**実測返信率**：42.9-71.4%（業界平均 5-10% の 5-14 倍）

**構造：**
```
[タイトル] 【事業部長がお会いします】[具体ミッション]〜[一任内容] まで
[冒頭] ・社員採用/業務委託にてスポット契約等、転職意向に合わせてご相談可能
       ・年商 XX 億以上の大型事業
[本文] ペルソナ特定要素（採用ペルソナの frustration_trigger に刺さるワード）
[動機付け] 会社の方針で自身が動かしたいことが動かせないフラストレーション訴求
[CTA] 応募ハードルを最大限下げる（フォーマット自由・まずはカジュアルに）
```

**A/B テスト知見（DMM 実測）**：
- バリアント A：「【執行役員直接面接】役員直下新規事業〜一任」→ 返信率 42.9%
- バリアント B：「【事業部長がお会いします】〜一度お話しませんか」→ 返信率 71.4%
- 「役職者が会う」の会話感のほうが「役員直接面接」の格式感より上回る

---

### gem-05：衛生要因強調型（ローレイヤー向け）

**推奨対象**：一般事務 / ローレイヤー / 法務事務等
**実測返信率**：計測未確定（ポジション固有性が高い）

**構造：**
```
[タイトル] "おしゃれなオフィス × 私服勤務 × 在宅勤務あり" + 安定した環境で長期キャリア
[冒頭] 候補者の具体業務経験に魅力を持った旨（1 文）
[衛生要因列挙] 在宅勤務あり / 安定業界 / 成長率 XX%（具体数字）
[CTA] 面接日程を複数提示いただければ
```

**設計原則**：
- キャリアアップ・裁量系の訴求は意図的に排除（ローレイヤーの動機軸は安全欲求）
- 応募書類持参リクエストでハードルを意図的に上げ、書類精度を確保

---

### gem-06：責任者自己開示 + ミッション共有型

**推奨対象**：経営企画 / コンテンツ責任者 / Biz 系上位層
**実測返信率**：28-50%

**構造：**
```
[挨拶] はじめまして。
[名乗り] [会社名] [事業部名] の責任者をしております [担当者名] と申します。
[個別化トリガー] [候補者]様のご経歴を拝見し是非一度お話したいと思いメッセージしました。
[自社文脈説明] [事業部が新設された背景 / 全社最重要事業であること]
[担当者自己開示] 私自身も [X 社] から入社した経緯 / キャリアパス簡潔に（2-3 行）
[未公開情報フック] まだ詳しい事業展開は世の中に発信していない状況 → 直接話したい
[CTA] ご多忙かと存じますがぜひご検討いただけますと
```

**設計原則**：
- 担当者の略歴（前職 / キャリア転換理由）を 2-3 行で自己開示 → 人として共感を引く
- 「未公開情報」を武器にして返信の動機を作る

---

### gem-07：企業直接スカウト差別化型

**推奨対象**：エンジニア（エージェント競合市場）/ Biz 上位層（エージェントスカウトが多い市場）
**実測返信率**：未計測（構造特徴的なため分離定義）

**構造：**
```
[タイトル] 【企業直接スカウト】 + 【年収 XX 万以上】 + 【フレックス / リモート可】
[冒頭] 企業直接スカウトであることを 1 行目で明示（エージェントとの差別化）
[本文] gem-02 に近い簡潔構造
[内在的動機] （任意）キャリアパス訴求をタイトルに入れるとより効果的
```

**設計原則**：
- 「エージェントからのスカウトと区別する」が設計意図（設計シートに明記されていた原則）
- 衛生要因（年収・働き方）+ 内在的動機（キャリア）の組み合わせが有効

## 導線フレーム（v0.2 追加）

全 gem に共通する導線設計。文面作成前に意識する。

```
標準導線 ──
  タイトル（一覧クリック誘引）
  → 冒頭数行（一番重要なポイント 3 点）
  → 候補者の経験に魅力を感じる旨（個別化シグナル）
  → 自社 / 事業部紹介（1-2 段落）
  → 面接喚起（CTA）

ハイレイヤー向け変形 ──
  タイトルに「役職者が会う」「一任」を入れる
  → 冒頭でビジネスチャンス感を出す（衛生要因でなく動機付け要因）
  → 応募ハードルを最大限下げる（フォーマット自由 / 業務委託 OK 等）
  → ペルソナの「現状の不満（frustration_trigger）」に刺さるワード
```

### A/B テスト変数定義（v0.2 追加）

```yaml
ab_test_design:
  rule: "1 回のテストにつき 1 変数のみ変更"
  min_sample_per_variant: 15  # 返信率比較に必要な最小送信件数
  variables:
    - name: タイトル型
      variants:
        A: 汎用型（職種名のみ）
        B: 役職者接触型（「事業部長がお会いします」等）
        C: 待遇明示型（「年収 XX 万以上・リモート可」）
    - name: 文面長
      variants:
        A: 短文（400 字未満 / gem-02 相当）
        B: 標準（400-1,000 字）
        C: 詳細（1,000 字超）
    - name: 担当者属性
      variants:
        A: 採用担当（HR）
        B: 現場 EM（エンジニアリングマネージャー）
        C: 事業部長 / 責任者
```

## NG 例

- 汎用スカウト Gem 1 個で全ポジション対応（個別最適性が落ちる）
- AI 生成スカウトをノーチェックで送付（ハレシネーション漏洩リスク）
- 候補者個別情報を平文で AI に投入（プライバシー違反）
- 求人ページ以外の外部リンクを本文に複数貼付（離脱リスク）
- 同一候補者へ複数事業部から同内容スカウト（組織不信・DMM 実例で発生した事故）
- ハイレイヤー候補に gem-05（衛生要因型）を誤送付（動機付け要因なしで返信率激減）
- 文面長 1,900 字超を全ポジションに適用（gem-02 の 233 字で 23.5% を記録した教訓）
