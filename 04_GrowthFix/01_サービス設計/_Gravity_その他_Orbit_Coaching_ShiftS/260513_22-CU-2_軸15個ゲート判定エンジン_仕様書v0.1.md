**バリデーション 5 項目**：
1. `customer_id` / `month` 必須（month は `YYYY-MM` 形式）
2. `axes` 内 15 キー全て存在（欠損時は `null` 許容・該当軸を「未計測（–）」扱い）

## 3. 処理ロジック

### 3.1 system prompt（コア指示・抜粋）

```
あなたは Gravity Orbit の「5 年継続装置」の中核判定エンジンである。

【ロール】
- 顧客の組織引力を 15 軸（集まる 5 / 躍動 5 / 留まる 5）で月次判定する
- 単一軸の閾値判定 → 3 ヶ月連続達成カウンタ更新 → 軸間相関監視 → カテゴリスター判定 → Gravity 全冠判定の 5 段階を順次実行
- ファイブスター演出（★1〜★3）を 22-CU-1 に供給し、月次レポート上のゲーミフィケーション動機を毎月再生産する

【学術裏付け】
- Locke-Latham 1990 目標設定理論（具体的・困難・達成可能な目標）
- Bandura 1977 Self-Efficacy（小さなクリア体験の累積で「次のステージへ」動機を再生産）
- Saks 2006 Engagement（職務 × 組織の 2 軸エンゲージメント）

【絶対ルール】
- 数値閾値は YAML 外出し必須（`orbit_axes_thresholds.yml`）。ハードコード禁則
- 連続達成カウンタは同月二重加算禁則（SQLite で client_id × axis_id × month を一意制約）
- 軸間相関 4 ルールは常に全評価（離職率↑なのに個人引力↑で警告未発火は NG）
- ステータス急変（先月★→今月⚠）には必ず構造的理由を warnings に明示
- 「弊社の独自指標」感の演出は禁則（業界標準語彙を主軸に置く）

【出力フォーマット】
- 月次サマリ JSON（4.1 参照）
- Quarterly 突合点 JSON（四半期末月のみ・4.1 参照）
- 軸 15 個ステータス一覧（cleared / progressing / danger + stars 0-3）

【セルフチェック】
出力前に以下 4 項目を自問：
1. 15 軸全て判定済か（null 軸は「未計測」明示）
2. 連続達成カウンタが過去 12 ヶ月分整合しているか
3. 軸間相関 4 ルールが全評価されているか
4. ファイブスター演出仕様（★1〜★3 + カテゴリスター + Gravity 全冠）が出力に含まれているか
```

### 3.2 内部ステップ（順序遵守）

```
Step 1: 入力 JSON Read + バリデーション（2.2 の 5 項目チェック）
   ↓
Step 2: YAML 閾値定義 Read（業種別 override 適用済の閾値マップ取得）
   ↓
Step 3: 軸ごとに閾値判定（15 軸 ループ）
   - direction（higher_is_better / lower_is_better）に応じて clear / warning / danger 振り分け
   - 結果 = {axis_id, status, value, threshold, distance_to_clear}
   ↓
Step 4: SQLite から過去 12 ヶ月の履歴 Read
   - 顧客 ID × 軸 ID × 月の連続 cleared 月数をカウント
   ↓
Step 5: 連続達成カウンタ更新（同月二重加算チェック）
   - 当月 cleared → 連続数 += 1
   - 当月 progressing → 連続数 = 0 だが「猶予期間 1 ヶ月」中なら維持
   - 当月 danger → 連続数 = 0（猶予なし）
   - 結果 = stars: 0-3（min(連続数, 3)）
   ↓
Step 6: 軸間相関監視 4 ルール適用
   - R-1: ⑪離職率 × ⑩個人引力スコア 逆相関
   - R-2: ②辞退率 × ⑥売上達成率 順相関
   - R-3: ⑨1on1 実施率 × ⑫エンゲージメント 順相関
   - R-4: ⑬ドラムビート × ⑪離職率 先行指標
   - 異常検知 → warnings 配列に追加
   ↓
Step 7: カテゴリスター判定
   - 集まる 5 軸 全て stars==3 → 「集まる軸スター」
   - 躍動 5 軸 全て stars==3 → 「躍動軸スター」
   - 留まる 5 軸 全て stars==3 → 「留まる軸スター」
   ↓
Step 8: Gravity 全冠判定
   - 全 15 軸 stars==3 → 「Gravity 殿堂入り」フラグ true
   ↓
Step 9: 月次サマリ JSON 構築 → stdout 出力 + SQLite 履歴更新
   ↓
Step 10: Quarterly 直前月のみ（metadata.quarterly_end_month == true）
   - 4 型再判定との突合点 JSON 追加出力
   - 集まる軸スコア合計 / 躍動軸スコア合計 / 留まる軸スコア合計 を `orbit_quarterly_review.py` に渡す
   ↓
Step 11: セルフチェック 4 項目（3.1 system prompt 内）を内部実行 → ログ出力
```

### 3.3 永続化設計

| データ | 永続化先 | スキーマ |
|---|---|---|
| 連続達成カウンタ | SQLite（`06_開発/scripts/orbit/orbit_axes_history.sqlite3`）| `axes_history(customer_id, axis_id, month, value, status, consecutive_count, cleared_at)` PRIMARY KEY = (customer_id, axis_id, month) |

### 3.4 連続達成カウンタの猶予ルール（R10 §3.2 準拠）

```
状態遷移：
  1 ヶ月達成 → ★1（「クリア 1 ヶ月目！あと 2 ヶ月で正式クリア」）
  2 ヶ月連続 → ★★（「あと 1 ヶ月で正式クリア」）
  3 ヶ月連続 → ★★★（正式クリア）
  正式クリア後 1 ヶ月のみ未達 → ★★ に降格（猶予期間中・SQLite に `grace_period_active` フラグ）
  正式クリア後 2 ヶ月連続未達 → ★ ゼロ（連続カウンタリセット）
  
猶予期間 SQLite 表現：
  consecutive_count 列 = 3 を維持しつつ
  grace_period_remaining 列 = 1 → 0 で減算
  次月 cleared 回復 → grace_period_remaining リセット
```

**重要決定**：R10 §9 オープン論点⑤「軸ごとに猶予期間が異なる可能性」は v0.1 では**一律 1 ヶ月猶予**で実装。v0.2 で軸別猶予に拡張余地を温存（YAML に `grace_period_months` フィールド追加可能な構造）。

## 5. ハーネス装置

### 5.1 NG 例（やってはいけない）

| # | NG パターン | 理由 | 検知方法 |
|:-:|---|---|---|
| ① | 閾値ハードコード（Python コード内に数値直書き）| YAML 変更時の追従不可・業種別 override 不能 | Grep `clear\s*=\s*[0-9]` で検出 |
| ② | 連続カウンタ同月二重加算 | データ整合性破壊・★3 が虚偽になる | SQLite UNIQUE 制約 `(customer_id, axis_id, month)` |
| ③ | 軸間相関 4 ルールの一部スキップ | 「離職率↑なのに個人引力↑で警告未発火」のような盲点 | self_check.correlation_4_rules_evaluated == false 時にエラー |
| ④ | ステータス急変の理由不明（先月★→今月⚠）| 顧客動機を削ぐ・「再カウントダウン」の演出失敗 | warnings に構造的理由を明示しなければエラー |
| ⑤ | 「弊社の独自指標」感の演出 | 業界標準語彙を主軸に置く R10 §0.3 指針④違反 | message 内 NG ワード `弊社独自` `当社オリジナル` を検出 |
| ⑥ | ★1 個で動機が削がれる演出（「もう少しで…」精神論）| R10 §4.5 OK/NG 演出ルール違反 | message に必ず「あと N pt」「あと N 日」希望表現を含める |
| ⑦ | カテゴリスター達成時の過剰演出 | R10 §4.5 「天才です」NG 例違反 | 「達成」「次の動機接続」のみ・形容詞禁則 |
| ⑧ | 業種別 override の取り違え | デフォルト閾値で判定してしまう / industry 不一致時のフォールバック誤り | input.industry と YAML キーの存在チェック |

### 5.2 セルフチェック項目（出力前に内部実行）

```python
def self_check(output_json: dict) -> dict:
    checks = {
        "all_15_axes_evaluated": len(output_json["axes"]) == 15,
        "consecutive_counter_integrity_ok": _verify_sqlite_consistency(),
        "correlation_4_rules_evaluated": len([w for w in output_json["warnings"]
                                              if w["rule_id"].startswith("R-")]) >= 0,
        "five_star_payload_complete": all(
            "stars" in axis and "status" in axis
            for axis in output_json["axes"]
        ),
        "category_star_flags_present": "category_stars" in output_json,
        "gravity_full_crown_flag_present": "gravity_full_crown" in output_json,
        "next_focus_recommendation_present": "next_focus_recommendation" in output_json,
    }
    return checks
```

**全項目 true でない場合**：エラーを stderr に出力 + 22-CU-1 へ「判定エンジン異常」シグナルを返す（22-CU-1 側で人手介入をトリガー）。
