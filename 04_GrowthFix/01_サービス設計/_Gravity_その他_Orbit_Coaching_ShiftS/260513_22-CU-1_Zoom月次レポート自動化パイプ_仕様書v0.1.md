## 0. agent definition draft（frontmatter 案）

```yaml
---
name: orbit_zoom_report_pipe
description: Orbit 月次運用の中核自動化エージェント。Zoom 録画 → Whisper 文字起こし → Claude Code 分析 → 月次引力レポート A4 3p 自動生成 → 石井 1 推敲ゲート → 顧客納品の一気通貫パイプ。20-30 社並列対応のスケール装置（石井稼働 月 8-10h / 売上月 300 万 / 時給 30-37 万を成立させる装置の中核）
model: opus
tools: Read, Write, Bash, Grep, Edit
---
```

**モデル選定理由（opus 固定）**：
- 月次レポートは「顧客に見せる思想層含む対外納品物」。Sonnet では学者モード化で抽象論に流れ「自社専用の分析」感が出ない（agent definition `CLAUDE.md` モデル選択原則の「思想層を含む深い分析」枠）
- 石井推敲ゲートで人間介入が入る前提でも、AI 草稿の質が低いと推敲時間が 5-10 分 → 30-60 分に膨らみ、石井稼働 月 8-10h 設計が破綻する。Opus 1 ファイル / Sonnet 並列の使い分けは本パイプでは Opus 単発が正
- 例外：Step 1（Whisper 文字起こし起動 + ノイズ除去）は別 sub-agent で Haiku / Sonnet 委譲可能（後述 §3.2 内部ステップ ① で記載）

## 1. 役割スコープ

### 1.1 担当する

- Zoom 録画ファイル自動取込（ローカル `~/Documents/Zoom/` 監視 + クラウド Zoom API 両対応）
- Whisper / Notta 文字起こし起動 + 完了検知（非同期キュー処理）
- 議事録 → ペイン抽出（離職予兆 5 シグナル）→ 月次サマリ草稿生成
- 軸 15 個ゲート判定エンジン（22-CU-2）への入力供給 JSON 生成 + 結果受領
- 月次引力レポート A4 3p PDF 自動生成（既存 `orbit_monthly_report.py` v3.1 拡張ベース）
- 石井推敲待ちキュー管理（Notion DB ステータス「推敲待ち / 推敲済 / 配信済 / 差戻し」）
- 推敲完了通知 → 顧客メール自動配信（PDF 添付 + テンプレ文面）
- プロコーチ事前ブリーフ JSON 生成（R9 連動・翌月セッション準備用）
- 議事録 Markdown を 22-CT-1/2/3（Cultivate AI 武装エージェント群）に連動配信（顧客が Cultivate 同時契約時のみ）

### 1.2 担当しない（メインスレッド or 他 agent に差し戻す）

- Zoom 録画自体の取得設定（Zoom Pro 以上の管理者権限 / 録画クラウド有効化は user / IT で事前設定）
- 軸 15 個ゲートの判定ロジック自体（22-CU-2 が担当・本 agent は IO 接続のみ）
- プロコーチ自身のセッション運営・プロコーチ評価（R9 プロコーチ束ね運用設計の領域）
- 石井推敲の質判断（推敲完了は石井の手動 OK ボタンで判定）
- Quarterly セッション（A4 8p）の自動生成（別 agent 候補・本 agent は月次 A4 3p のみ）
- 顧客契約管理 / 課金処理（カスタマー.md / Notion 顧客マスタ DB 側の領域）

### 3.2 内部ステップ

| Step | 処理 | sub-agent / model | 所要時間 / 1 社 |
|:-:|---|:-:|:-:|
| ① | Zoom 録画 mp4 取込 → Whisper API 文字起こし（or Notta API 委譲）| 別 sub-agent 化候補（Haiku / Sonnet）| 録画 90 分 → 文字起こし 5-10 分 |
| ② | 議事録 Markdown 化 + ノイズ除去（フィラー / 確認句 / 雑談セグメント抽出）| 本 agent（Opus）| 2-3 分 |
| ③ | ペイン抽出（離職予兆 5 シグナル：エンゲージメント / 業務遂行 / 評価不公平感 / キャリア停滞 / 関係性ストレス）| 本 agent（Opus）| 2-3 分 |
| ④ | 軸 15 個ゲート判定エンジン（22-CU-2）に JSON 入力供給 → 結果受領 | 22-CU-2 委譲 | 1-2 分 |
| ⑤ | 月次引力レポート A4 3p 草稿生成（reportlab PDF）| 本 agent（Opus）→ `orbit_monthly_report.py` v3.1 拡張呼出 | 3-5 分 |
| ⑥ | プロコーチ事前ブリーフ JSON 生成（R9 連動・翌月セッション準備）| 本 agent（Opus）| 1-2 分 |
| ⑦ | 石井推敲キューに投入（Notion DB ステータス = 「推敲待ち」+ Slack 通知）| 本 agent（Opus）| 即時 |
| ⑧ | 石井推敲完了通知 → 自動メール送信（顧客 PDF 配信）| 本 agent（Opus）| 即時 |

**自動化合計時間 / 1 社**：15-25 分（②-⑦・石井推敲時間は含めず）
**石井推敲時間 / 1 社**：5-10 分（差分 diff 提示 + p.1 / p.3 重点確認）
**20 社並列時**：自動化 15-25 分 × 20 = 5-8h（バッチ実行・夜間 cron）+ 石井推敲 5-10 分 × 20 = 100-200 分

## 5. ハーネス装置

### 5.1 NG 例（実行時に検出して停止 or 警告）

| # | NG パターン | 検出方法 | アクション |
|:-:|---|---|---|
| 1 | 石井推敲ゲートをスキップして顧客に直接配信 | Notion ステータスが「推敲済」以外で配信トリガー発火 | 配信停止 + Slack エラー通知 |
| 2 | 個人特定情報の不適切露出（センシティブ発言の生披露）| 議事録に「退職」「うつ」「セクハラ」等の機微語含む生引用検知 | 該当箇所マスキング + 石井推敲時要確認フラグ |
| 3 | 軸 15 個ゲートの数値が前月と完全同じ（チェック漏れの可能性）| 全 15 軸の数値が前月 JSON と完全一致 | 警告 + 石井推敲時に「データ取込み確認」フラグ |
| 4 | レポート文体が前月コピペ感（個別差分が弱い）| 前月レポートとの cosine similarity > 0.85 | 草稿再生成 + 「差別化不足」フラグ |
| 5 | 「Orbit のコーチング」「Orbit 内 Gravity Coaching」など語彙混同 | 正規表現 grep `Orbit.*コーチング` / `Orbit.*Coaching` | 自動置換 + 石井推敲時要確認 |
| 6 | 学術論文名 / 被引用数を顧客向け本文に露出 | grep `Saks 2006` `Edmondson` `被引用` 等 | 自動マスキング（裏に温存・表に出さない） |
| 7 | プロコーチ評価コメントが顧客向け本文に混入 | grep `プロコーチ.*評価` `プロコーチ.*課題` | 自動削除 + ブリーフ JSON 側へ移動 |
| 8 | フレンズ価格 / 値引き示唆が混入 | grep `特別` `今だけ` `値引き` 等 | 自動削除（商品.md 中核原則 8 認知価格設計違反） |

### 5.2 セルフチェック項目（草稿生成後・石井推敲前に自動チェック）

| # | チェック項目 | 合格条件 |
|:-:|---|---|
| 1 | p.1 / p.2 / p.3 の 3 ページ構造が崩れていないか | PDF ページ数 = 3 |
| 2 | 軸 15 個 × ★/☆/⚠ アイコンが全 15 軸表示されているか | アイコン数 = 15（欠損なし）|
| 3 | 学術武装の隠し方が適切か（裏で論文・表で日常言語）| 顧客向け本文に学術用語 0 件 |
| 4 | Coaching 主軸不変の語彙ルール遵守（「Orbit セッション」呼称統一）| `Coaching` / `コーチング` を Orbit 文脈で使用 0 件 |
| 5 | 前月比の差分明示（伸びた軸 / 落ちた軸の Before/After 表現）| 差分グラフ含有 + 差分テキスト 100 字以上 |
| 6 | プロコーチ事前ブリーフ JSON に重点軸 3 個 + 問い 5-7 個が含まれるか | JSON schema 検証 PASS |
| 7 | 議事録 Markdown のノイズ除去率 | 元ファイル比 30-50% 圧縮（過剰圧縮 = 情報欠落の懸念） |
| 8 | 4 型現在地の更新（前月 = 今月）| 4 型変化があれば理由 50 字以上付与 |

### 5.3 合格基準（最終納品物としての品質ゲート）

| # | 合格基準 | 測定方法 |
|:-:|---|---|
| 1 | 石井推敲 5-10 分 / 1 レポートで完了する粒度 | 推敲ログから経過時間集計（中央値 < 10 分）|
| 2 | 顧客側で「自社専用の分析」と感じる個別性 | 顧客フィードバック 5 段階 ≥ 4 / 5（NPS 別軸測定）|
| 3 | プロコーチ側で「翌セッションすぐ使える」具体度 | プロコーチ NPS ≥ 4 / 5・「ブリーフのみで準備完了」率 ≥ 80% |
| 4 | 軸 15 個進捗が前月と差分 1 軸以上ある | 完全同一 = データ取込み失敗の蓋然性高 |
| 5 | 「絶対買う確信フィルター」（接続装置.md 中核原則 0）通過 | 石井の主観判定（「これを毎月もらえるなら 5 年継続したい」確信）|
