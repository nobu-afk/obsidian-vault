## 2. 動的位相モデルの正式定義

### 2.1 「Dynamic」の意味

**Rousseau et al. 2018 Abstract の核命題（Crossref / OpenAlex 抽出）：**

> The paper presents a phase-based model of PC processes (intraphase and interphase) wherein the functions of key variables (e.g., promises, inducements, contributions, and obligations) change over time and context.

**「Dynamic」の正式定義：**

> **心理的契約は静的合意ではなく、約束 / 誘因 / 貢献 / 義務などの主要変数の機能（function）が時間と文脈に応じて変化する動的プロセスである。**

→ 従来の Rousseau 1989/1995 は心理的契約を「合意」として記述したが、2018 は **「プロセス」「機能の文脈依存的変化」**として再定義。Wevox の "engagement スコア" のような静的測定では捉えきれない次元を理論化。

### 2.2 「Phase」の意味

**4 phases の正式名称（確定）：**

1. **Creation phase（契約形成位相）**
2. **Maintenance phase（契約維持位相）**
3. **Renegotiation phase（再交渉位相）**
4. **Repair phase（修復位相）**

**各 phase の核命題（既知知識 + Abstract 抽出ベース推論）：**

| Phase | 中心活動 | 主要変数の機能 | Orbit 対応装置 |
|---|---|---|---|
| **Creation** | 採用時・入社初期の期待形成 | 約束（promise）が中心 | Shift R / Shift Full の「躍動 OS 編成」フェーズ |
| **Maintenance** | 日常業務での約束 - 履行の継続的照合 | 誘因（inducement）と貢献（contribution）のバランス | **Orbit 月次セッション**（90 分・月 1）= Maintenance phase の制度装置 |
| **Renegotiation** | 環境変化・約束変更時の合意再形成 | 義務（obligation）の再定義 | **Orbit Quarterly Page 6**（90 分・3 ヶ月毎）= Renegotiation phase の中核制度 |
| **Repair** | Breach/Violation 発生後の修復努力 | 全変数の修復的再設計 | **Orbit Quarterly Page 6（緊急発動時）+ 半期年合宿** = Repair phase の制度化 |

### 2.3 「Multidimensional」の意味（intraphase / interphase）

**Abstract の決定的記述：**

> ...intraphase and interphase wherein the functions of key variables change over time and context

**正式定義：**

| 次元 | 定義 | 含意 |
|---|---|---|
| **Intraphase dynamics**（位相内ダイナミクス）| **同一 phase 内**で主要変数の機能や強度が時間経過で変化する | 同じ Maintenance phase 内でも、入社 6 ヶ月時点と 3 年時点では誘因 / 貢献のバランスは違う |
| **Interphase dynamics**（位相間ダイナミクス）| **phase 間の遷移**で主要変数の機能が質的に変化する | Maintenance → Renegotiation 遷移時、約束の意味づけが「日常確認」から「再合意対象」に質変化 |

→ **「Multidimensional = intraphase + interphase の 2 軸 + 4 主要変数（promises / inducements / contributions / obligations）の組み合わせ」**

これは Phase 14-A で推測した「cognitive / affective / behavioral の 3 次元分解」とは異なる構造。**2018 の multidimensional は「時間軸 × 変数軸」の構造**であり、2015 の self-regulation（cognitive / affective）はその下位構造として包摂される。

## 3. Tomprou 2015 との理論的接続

### 3.1 2015 と 2018 の関係

| 項目 | Tomprou-Rousseau-Hansen 2015 | Rousseau-Hansen-Tomprou 2018 |
|---|---|---|
| 第一著者 | Tomprou（博士課程後）| **Rousseau（心理的契約論開祖）が引き継ぎ**|
| 研究範囲 | Violation 後の被害者プロセス（局所）| 心理的契約プロセス全体（全局） |
| モデル種別 | **結果分類**（4 outcomes：thriving / reactivation / impairment / dissolution）| **位相モデル**（4 phases：creation / maintenance / renegotiation / repair）|
| 中心構成概念 | VRBs（Violation Resolution Beliefs）+ self-regulation | Phase × 主要変数 × intra/inter phase dynamics |
| 時間軸の扱い | 単一 violation event 後の self-regulation period（数ヶ月）| 雇用関係全体のライフサイクル（数年）|
| 介入対象 | Repair phase 内の cognitive regulation | **Maintenance phase での予防介入 + Renegotiation phase の制度化 + Repair phase の質確保** |

### 3.2 2018 で新規追加された理論的貢献

1. **時間軸の完全組み込み**：2015 は violation event 起点の局所モデルだったが、2018 は雇用関係全体のライフサイクルに拡張
2. **Maintenance phase の理論化**：Breach 発生前の「日常維持期」を独立 phase として位置付け。これは予防介入の理論的根拠
3. **Renegotiation phase の独立化**：従来 Repair に含まれていた renegotiation を独立 phase として分離。Breach がなくても環境変化で renegotiation が必要という認識
4. **Intraphase / Interphase の二層ダイナミクス**：従来の「合意の静的状態」観から「機能の動的変化」観への根本的転換
5. **Self-regulatory mechanisms の各 phase への適用**：2015 では violation 後にのみ self-regulation を扱ったが、2018 では各 phase に固有の self-regulation を理論化

### 3.3 2018 が解消する 2015 の理論的限界

| 2015 の限界 | 2018 での解消 |
|---|---|
| Violation event 起点モデル = 「壊れた後」しか扱えない | 4 phase 循環 = 壊れる前・壊れた後・修復後の全局面を扱う |
| Repair / Renegotiation の混合 | Repair（Violation 後）と Renegotiation（環境変化対応）を分離 |
| 単一 violation 想定 | 複数 violation の累積・循環を扱える |
| 介入機会が violation 後に限定 | **Maintenance phase での予防介入を理論化** |
| Cognitive vs Affective の二分 | 4 主要変数 × intra/inter phase の多次元ダイナミクス |

### 3.4 セット引用の必然性

書籍『引力経営』第 6 章では、**2015（局所詳細モデル）+ 2018（全局位相モデル）のセット引用が学術的に必須**：

- **2015 単独引用** = 「violation 後の対処法」しか語れない（Orbit 月次セッションの根拠が弱い）
- **2018 単独引用** = phases の名称羅列のみで、各 phase 内の自己調整プロセス詳細が不足
- **2015 + 2018 セット** = **「Maintenance phase で予防（月次）→ Renegotiation phase で再交渉（Quarterly）→ Repair phase で修復（緊急 Quarterly + 合宿）」の Orbit 3 層構造の完全な理論的裏打ち**

## 4. 位相転移の動的プロセス

### 4.1 4 位相と循環構造

```
                  ┌──────────────────┐
                  │                  │
                  ↓                  │
            Creation phase           │
              （契約形成）            │
                  │                  │
                  ↓                  │
            Maintenance phase ────┐  │
              （契約維持）         │  │
                  │               │  │
        Breach    │   Environment │  │
         発生 ↓   │      変化 ↓   │  │
                  │               │  │
            Repair phase ←  Renegotiation phase
              （修復）  →     （再交渉）
                  │               │
                  └───────────────┘
                  Maintenance に復帰 or Dissolution
```

**循環構造の含意：**

- Maintenance phase は **「default state（既定状態）」**：通常時はここに長期滞在
- Breach 発生 → Repair phase に強制遷移（Violation outcome 4 分岐の場）
- 環境変化（昇進・組織変更・外的衝撃）→ Renegotiation phase に主体的遷移
- Repair / Renegotiation 後 → Maintenance に復帰 or Dissolution（Tomprou 2015 の dissolution outcome）

### 4.2 位相転移のトリガー

| 遷移 | トリガー | Orbit 検出装置 |
|---|---|---|
| Creation → Maintenance | 入社後 90-180 日経過・心理的契約の安定化 | Shift R/Full 完了 → Orbit 移行（90 日後） |
| Maintenance → Renegotiation | 組織変更・昇進・役割変化・環境的衝撃 | **月次セッション**で兆候検出 → Quarterly Page 6 で正式交渉 |
| Maintenance → Repair | Breach / Violation の発生 | **月次セッション**で early violation 検出 → 緊急 Quarterly 発動 |
| Renegotiation → Maintenance | 新合意成立・契約条件再確定 | Quarterly Page 6 [5] 新合意書 → 次 Maintenance phase 開始 |
| Repair → Maintenance | Thriving / Reactivation outcome（Tomprou 2015）| Quarterly Page 6 VRBs 高水準確認 |
| Repair → Dissolution | Dissolution outcome（Tomprou 2015）| VRBs 極低検出 → 石井登壇緊急 Quarterly |

### 4.3 位相間遷移の決定要因（既知知識ベース）

Rousseau et al. 2018 が示唆する遷移決定要因：

1. **Self-regulatory capacity**（自己調整能力）：高いほど Maintenance → Repair → Maintenance の修復ループに復帰しやすい
2. **VRBs（2015 概念を継承）**：高いほど Repair phase で Maintenance 復帰、低いほど Dissolution
3. **Organizational support**（POS）：Renegotiation phase での新合意質を決定
4. **Past phase transition history**（過去の位相遷移履歴）：成功体験が次の遷移を促進
5. **Environmental volatility**（環境変動度）：高いほど Renegotiation 頻度上昇

## 5. 多次元構造（Multidimensional）詳細

### 5.1 4 主要変数 × 4 phases マトリクス

**Rousseau et al. 2018 の核命題：4 主要変数（promises / inducements / contributions / obligations）の機能が各 phase で変化する**

| 変数 | Creation | Maintenance | Renegotiation | Repair |
|---|---|---|---|---|
| **Promises（約束）** | 形成・明示 | 確認・想起 | 再定義・拡張 | 明示的再確認 + 補償 |
| **Inducements（誘因）** | 期待形成 | 履行確認 | 再交渉対象 | 修復的追加 |
| **Contributions（貢献）** | 期待形成 | 履行表示 | 再定義 | 一時的減退 |
| **Obligations（義務）** | 暗黙的形成 | 暗黙的継続 | 明示化・再合意 | 修復的再合意 |

→ **同じ変数（例：promises）でも phase により機能が質的に違う**。Maintenance での promise は「想起確認の対象」、Renegotiation での promise は「再定義の対象」、Repair での promise は「明示的再確認 + 補償の対象」。

### 5.2 Intraphase dynamics の具体例

**Maintenance phase 内の dynamics：**

- 入社 6 ヶ月時：promises の想起頻度高・inducements / contributions のバランス確認頻度高
- 入社 3 年時：promises の想起頻度低・obligations の暗黙的継続が中心
- 入社 5 年時：promises の薄れ・新規 promises の自然形成（隠れた Renegotiation）

→ **Orbit 月次セッションは Maintenance phase 内の intraphase dynamics を制度的に可視化する装置**。放置すると 3 年時点で promises が薄れ、5 年時点で隠れた Renegotiation が無自覚に進行する。

### 5.3 Interphase dynamics の具体例

**Maintenance → Renegotiation 遷移時の dynamics：**

- 昇進前：promises が「日常的に確認される対象」だった
- 昇進辞令直後：promises が「再定義の対象」に質変化
- 昇進後 30 日：obligations の暗黙的拡張が新規 promises として再合意される

→ **Orbit Quarterly Page 6 は Maintenance → Renegotiation 遷移を制度化された場で行う装置**。放置すると暗黙的拡張が「人材側の主観的 violation」を引き起こす。

## 6. Self-Regulatory Mechanisms（各 phase での自己調整）

### 6.1 2018 が拡張した self-regulation 概念

Tomprou 2015 では Repair phase（violation 後）の self-regulation のみを扱ったが、**2018 では各 phase に固有の self-regulation を理論化**：

| Phase | 主たる self-regulation メカニズム |
|---|---|
| **Creation** | Expectation calibration（期待の調整）：自己と組織の期待値ギャップを縮小 |
| **Maintenance** | **Monitoring & adjustment**（監視と微調整）：日常的な約束 - 履行照合 |
| **Renegotiation** | **Goal reformulation**（目標再形成）：自己目標と組織目標の再アライメント |
| **Repair** | **Cognitive vs Affective regulation**（Tomprou 2015 継承）：VRBs を介した 4 outcome 分岐 |

### 6.2 Orbit Quarterly Page 6 への翻訳

**Page 6 は Renegotiation phase の制度装置**であるが、Repair phase 緊急発動時にも兼用：

- **通常時**（Renegotiation 主導）：Goal reformulation を制度的に行う場
- **緊急時**（Repair 主導）：Cognitive regulation を強制発動させ Thriving/Reactivation outcome に誘導

→ **Page 6 v1.1 では「通常版 Renegotiation 版」と「緊急版 Repair 版」の 2 トラック構造化が必要**

## 7. VRBs の 2018 での発展

### 7.1 2018 における VRBs の位置付け

**2015 では VRBs は Repair phase 内の中核構成概念だった**が、**2018 では VRBs を含む信念体系（beliefs）が各 phase での self-regulation の媒介変数として一般化された**：

| Phase | 該当する beliefs |
|---|---|
| Creation | Promise fulfillment expectations（履行期待）|
| Maintenance | **Maintenance beliefs**（維持信念）：「約束は今後も履行される」確率信念 |
| Renegotiation | **Renegotiation efficacy beliefs**（再交渉効力感）：「再交渉で良い合意に到達できる」信念 |
| Repair | **VRBs**（Tomprou 2015 概念）：「violation は解決される」信念 |

→ **2015 VRBs は 2018 における Repair phase 固有の beliefs であり、他 phase には独自の beliefs 系がある**

### 7.2 VRBs 3 次元（Probability / Timing / Agency）の各 phase 拡張

| 次元 | Repair phase（VRBs）| Maintenance phase（拡張）| Renegotiation phase（拡張）|
|---|---|---|---|
| **Probability** | violation 解決可能性 | promises 履行確率 | 新合意到達確率 |
| **Timing** | 解決までの時間 | 履行までの時間 | 合意成立までの時間 |
| **Agency** | 誰が解決するか | 誰が履行するか | 誰が交渉主導するか |

### 11.5 位相転移検出ロジック（参謀 / AI 自動化）

```python
# 月次セッションログ → Quarterly Track 判定

def determine_quarterly_track(monthly_session_logs):
    """
    Args:
        monthly_session_logs: 過去 3 ヶ月の月次セッションログ
    Returns:
        ('A', 'B', or 'A+B') = Renegotiation 版 / Repair 版 / 複合
    """
    breach_detected = any(log['breach_signal'] for log in monthly_session_logs)
    vrbs_low = any(log['simple_vrbs'] <= 2 for log in monthly_session_logs)
    env_change = any(log['environmental_change'] for log in monthly_session_logs)
    maintenance_belief_drop = (
        monthly_session_logs[0]['maintenance_belief']
        - monthly_session_logs[-1]['maintenance_belief']
    ) >= 1

    if breach_detected or vrbs_low or maintenance_belief_drop:
        track = 'B'  # Repair 版
    elif env_change:
        track = 'A'  # Renegotiation 版
    else:
        track = 'A'  # 標準 Renegotiation 版（軽微）

    if env_change and (breach_detected or vrbs_low):
        track = 'A+B'  # 複合対応

    return track
```
