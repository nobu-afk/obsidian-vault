## 2. 新規事業開発の心得（pptx）構造抽出

### 2-1. コンテンツ概要

- **実施主体：** プレイド社内事業組織（Accel 3）でのノウハウ共有セッション
- **構成：** インプット講義（前半・スライド 1〜27）+ グループワーク（後半・スライド 28〜40）
- **目的：** 新規事業開発の基礎思考とアイデア検証のフレーム習得
- **形式：** 90 分グループワーク + 30 分発表（1 チーム 5 分発表 + 5 分 Q&A）

### 2-2. 新規事業開発の方法論（コア構造）

**「事業とは何か」の定義から入るアプローチ：**

```
事業 = 「課題 × 解決方法」の組み合わせで収益を生み出すこと
新規事業 = 「課題 × 解決方法」の組み合わせが「新しい」こと
新規事業と既存事業の最大の違い = お金を払ってくれる顧客の有無
人は何にお金を払うか = ありたい姿と現状のギャップを埋めるもの（＝課題の解決）
```

→ 抽象定義から入り「なぜ課題設定が最重要か」を論理的に導く構成。

**「良い課題 × 良い解決策」の 6 要素：**

| 軸 | 要素 | 核心 |
|---|---|---|
| 課題 | 1. インパクト大 | 強度 × 頻度。「少数が泣いて喜ぶ課題 > 多数がほどほど困る課題」 |
| 課題 | 2. 解像度 MAX | 個に迫る。具体的な 1 人の課題から掘る（抽象課題から掘らない） |
| 課題 | 3. 構造理解 | 課題発生のメカニズム把握。なぜ現在解決されていないかを問う |
| 解決策 | 4. 課題を十分に解決 | クリティカルイシューを完全解決できる最低ラインを定義する |
| 解決策 | 5. シンプル | 何かを尖らせる・トレードオフで何かを捨てる・デメリットを割り切って宣言 |
| 解決策 | 6. 他より優れている | A 軸で負けても B 軸で圧倒的に優れている軸を持つ |

**アイデア設計の総括原則：**

```
・広く薄い課題ではなく、狭く深い課題を選ぶ
・極限まで解像度を高める、個に迫る
・メカニズムを捉える
・クリティカルイシューを解決する
・トレードオフで捨てる、尖りを作る
・他の選択肢より何が良いかを明確にする
```

**「意外と大事なこと」（マインドセット層）：**

- アイデアのデメリット部分は包み隠さず言う（粗探しを誘導してあげる→安心感が増す）
- 何が分かっていないかを隠さず言う（次ステップに進めやすくなる）
- 自分・自分の会社がいらないものは誰もいらない（自己当事者性）
- 知り尽くしていない誰かを妄想して検討しない

**「一番大事なこと」（スライド 25）：**

```
「ワクワクするか？」
```

→ ロジック構造を積み上げた末に「感情・内発性」に着地させる設計。プレイドの思想層が見える。

### 2-3. リーンキャンバスの活用方法

グループワークのアウトプット形式として「リーンキャンバス」を使用。各セルの記入指針を明示：

| セル | 記入内容 |
|---|---|
| 課題 | クリティカルイシューを見極めた上でのメカニズム把握。既存代替品も記入可 |
| 解決策 | クリティカルイシューへの具体的解決方法 |
| 主要指標 | 進捗把握のための重要指標 |
| 独自の価値提案 | クリティカルイシューを解決できる顧客への明確で説得力のあるウリポイント |
| チャネル | 顧客との接点経路 |
| 顧客セグメント | お金を払ってでも解決したい課題を抱える具体的な顧客属性（1 枚に複数顧客を書かない） |
| コスト構造 | 変動費と固定費 |
| 収益の流れ | マネタイズモデル・単価 |

### 2-4. ワーク設計の構造

**グループワーク設計：**

1. 各自が事前に記入した「探索シート」を共有
2. 発表者を決める
3. テーマを 1 つに絞る
4. 新規事業アイデアをディスカッション
5. リーンキャンバスにまとめる
6. 発表（5 分）+ Q&A（5 分）

**チーム別成果物の特徴（ワーク実施後スライドから）：**

各チームのアイデアを「企業の魅力発信・ミスマッチ解消」「RPO 比較マッチング」「研究室×企業マッチングプラットフォーム」「組織開発 SaaS」「学校法人集客支援」「中小企業 M&A マッチング」と多様に発展させた点が特徴的。

→ 参加者が「自分ごと」として当事者性のある課題を持ち込むことで、アイデアの解像度が自然と上がる設計になっている。

## 3. ユーザー体験の設計とは：マリオの話（pptx）構造抽出

### 3-1. コンテンツ概要

- **作成者：** Takahiro Ueda（UEPON）/ PLAID STUDIO ZERO
- **日付：** 2024 年 4 月 1 日
- **形式：** ナレッジ WS 資料（前半 8 スライドが実コンテンツ・後半 15 スライドは再利用可能テンプレート）

### 3-2. 教育 WS のフレーム

**中核概念（スライド 2）：**

```
ユーザー体験の設計とは？
= ユーザーになってもらいたい状態から逆算して、サービス体験を設計すること
```

→「ゴール状態（なってもらいたい状態）」から逆算して設計するという原則を先に定義してから、例を通じて説明する構成。

**例 1：スーパーマリオのゲーム体験設計（スライド 3〜5）：**

参照書籍：『ビジネスを変える"ゲームニクス"』（サイトウ・アキヒロ）

マリオの最初の 30 秒の体験設計を「ゲームバランス基本曲線」で説明：
- 縦軸：報酬（快感・達成感）
- 横軸：時間
- ゲームは「少し難しい → 試みる → 成功 → 報酬 → 次の難しさ」のサイクルで設計

マリオの初期体験設計の具体例（UX 設計の解析）：
1. マリオが左端・キャラが右向き → 右に進むことを直感的に伝える
2. 画面内に明らかに目立つ「？」ブロック → アクションを誘発
3. 心地よい音とともにコインを入手（報酬）
4. 同じ「？」を叩くと何かを得られる法則が定着
5. 見慣れぬキノコが進行方向に流れてきて巨大化（報酬）

→ **「ルールを教えずにプレイしながら学ぶ」設計**。これが UX 設計の本質として提示される。

**例 2：食材宅配 EC サイトの体験設計（スライド 6〜8）：**

Oisix の定期宅配を題材に、カスタマージャーニー全体像 → EC サイト体験の位置づけ → ナビゲーション設計の順で説明：

**「なってもらいたい状態 × やってもらいたい流れ」の設計例：**

| なってもらいたい状態 | やってもらいたいこと |
|---|---|
| 木曜日に注文開始に気づく | 通知 |
| 初期セット内容を編集する | 今週のセット画面で不要なものを外す |
| 未知の商品に出会い追加する | 特集ページでワクワクして追加 |
| 決めていた商品を素早く追加 | カテゴリ・検索で能動的に探す |
| 次回以降も自動カートに入れる | リスト登録アクション |

**ナビゲーション設計の構造：**
- サイトの遷移は「定期セット → おすすめ → 探す → カート」の順番で設計
- 「おすすめ TO おすすめのループ」に乗せ、必ず新しい商品か追加商品を発見させる
- ユーザーは「押しつけ」でなく「自然な流れ」として追加する

### 3-3. 教育コンテンツとしての学び設計

このスライドが示す教育設計の原則：

1. **抽象定義を先に置く**（「ユーザーになってもらいたい状態から逆算する」）
2. **誰もが知る具体例で直感的に理解させる**（マリオ）
3. **ビジネス文脈の具体例に接続する**（Oisix EC）
4. **テンプレートスライドを再利用可能な形で後半に配置**（自社・顧客の事例に差し替えて使える）

→ 「理論 → なじみのある例 → ビジネス例 → 自己適用」の 4 段構造。

**「ゲームニクス」フレームとしての活用：**

- 報酬曲線（達成感の出し方）
- ルールを教えずに体験させる（直感的 UX）
- 段階的な難易度設計（フロー状態の維持）

## 4. 3 ファイル横断パターン（プレイド型「伴走ナレッジ」の構造）

### 4-1. プレイド伴走型支援の通底構造

3 ファイルを横断すると、以下の一貫したパターンが見える：

**[パターン 1] 「定義から入る」設計**

- 旅行業 E 社：「コーポマーケとは何か・なぜ必要か」を課題認識スライドで定義してから提案に入る
- Accel 3：「事業とは何か → 新規事業とは何か → 人は何にお金を払うか」と基礎定義から順番に積み上げる
- マリオ：「ユーザー体験の設計とは？」を最初のスライドで 1 文で定義する

→ **「定義を共有することで思考の前提をそろえる」という伴走哲学**。

**[パターン 2] 「具体事例 → 汎用化」の知識移転設計**

- 旅行業 E 社：リクルート・三井不動産の他社事例で「他社はどう解決したか」を先に示す
- Accel 3：QBハウス・自動車保険等の具体例で原則を説明
- マリオ：マリオ（日常文脈） → Oisix（ビジネス文脈）で抽象化を段階的に促す

→ **「具体 → 抽象 → 自己適用」の 3 ステップ知識移転モデル**。

**[パターン 3] 「当事者性・ワクワク」を最後に置く**

- Accel 3：6 要素のロジック構造を積んだ末に「一番大事なことは ワクワクするか？」で締める
- マリオ：UX 理論を積み上げた後に余談スライドで終わる（おそらく「自分ごと化」の場）
- 旅行業 E 社：戦略策定の先に「人材育成・組織設立・行動基準策定」まで見据える姿勢

→ **ロジックで根拠を積み上げ、最後に感情・内発性に着地させる設計**。

**[パターン 4] フェーズを「何をする月か」ではなく「何が完成する月か」で定義**

旅行業 E 社の月次設計で顕著：

- 単なる「インタビュー期間」ではなく「情報収集完了 / ROI 算定情報蓄積」
- 単なる「戦略策定期間」ではなく「コーポマーケ戦略骨子 / ROI 算定基準骨子」

→ **アウトプット定義でフェーズを区切る伴走設計**。クライアントが「今月何が手に入るか」を常に可視できる。

**[パターン 5] 「mustとnice-to-have」の優先度設計**

ヒアリング項目に must フラグを付与し、必須収集情報と補足情報を分離。

→ **時間制約の中でも核心情報は必ず収集する設計原則**。

### 4-2. プレイド型伴走の特徴的な弱点・限界

- マリオ WS の後半 15 スライドがテンプレートのみで実コンテンツなし → **WSコンテンツの汎用化・再利用が未成熟**
- 旅行業 E 社の提案書は「支援体制」の個人名が多数登場 → **人依存型の提案構造**（スケールしにくい）
- Accel 3 のワーク成果物（チーム別アイデア）のフィードバック・評価軸が明示されていない → **「いいアイデアかどうか」の判定基準が不明**

**結論：** プレイド STUDIO ZERO の 3 ファイルから「定義から入る → 具体事例で汎用化 → 感情・内発性に着地」という一貫した伴走設計哲学を抽出。Coaching / Shift C / Orbit の設計強化に直接転用できる P0 候補が 8 件（SC: 3 / OR: 2 / CO: 3）、営業資料組込候補が 5 件確認された。

1. **「アウトプット定義でフェーズを区切る」伴走設計**（旅行業 E 社）：「今月何が手に入るか」をクライアントと合意してからフェーズ着手する構造。Orbit 月次レポートと Shift C フェーズ設計に即適用可能（P0 OR-01 / SC-01）

3. **「なってもらいたい状態から逆算する」UX 設計原則**（マリオ WS）：プロダクト設計だけでなくコーチング設計・セッション設計にも適用可能な汎用原則。「クライアントになってもらいたい状態」から逆算してセッション構造を組む（P0 CO-01）
