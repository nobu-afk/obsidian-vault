## 3. 賞与パック（評価会議運営ガイド + 評定決定計算シート）の構造

**ソース：** xlsx①（評価会議運営ガイド・2シート）+ xlsx②（評定決定・賞与額計算シート・6シート）

### 3-1. 賞与運用フロー（全8ステップ）

**評価会議ガイド（①）が規定する前半：**
1. システム内で評価者による全評価が完了していることを確認
2. システムから評価済みデータをダウンロード
3. ダウンロードデータを「評価調整・賞与計算シート」へ貼付け、シートを完成させる
4. 評価会議にて協議（事前に「賞与原資」「評価調整方針」を準備・決定）
5. 評価調整、最終評価決定
6. 賞与原資の決定（全体一本 or 部門別分割）
7. 最終評定を元にポイント集計
8. 1次集計（賞与額計算）→ 必要に応じて2次集計（調整）→ 支給額決定

**賞与計算後の後半：**
9. 支給額決定、支給手続き（社会保険賞与支払い届の作成・提出）
10. 本人宛通知
11. 評価面談（フィードバック）

### 3-2. 賞与額計算シート（②）の6シート構造

| シート | 内容 | AI 自動化可能性 |
|---|---|---|
| **0_所属役職マスタ** | 所属/役職/等級/評価者の4マスタ定義（5所属×5役職×6等級） | 高：初期設定後は静的 |
| **0_社員マスタ** | 社員番号・所属・役職・等級・評価点のマッピング（50名規模デモ） | 高：HR システム連携で自動更新 |
| **1_評定決定・調整シート** | 評価点数 → 評語（SS/S/A/B+/B/C/D）変換、評価者間分布確認、ピッチ調整 | 中：調整判断は人間、計算は自動化 |
| **2_支給計算・個人別** | 等級×評語 → ポイント換算 → 賞与額計算 | 高：ポイント換算表が固定なら全自動 |
| **3_参考・評価者別** | 評価者間の評価点散布図（評価者バイアス可視化） | 高：可視化自動生成可能 |
| **4_賞与結果サマリ** | 全体結果のサマリ（空シート・ユーザーカスタム領域） | 高：自動集計 |

### 3-3. 評定決定ロジックの構造

**点数 → 評語変換テーブル（例）：**

| 評価点範囲 | 評語 |
|---|---|
| 0〜35点 | D |
| 36〜40点 | C |
| 41〜45点 | B |
| 46〜60点 | B+ |
| 61〜70点 | A |
| 71〜75点 | S |
| 76〜80点 | SS |

**2段階調整メカニズム：**
- 1次集計：ポイント換算表で賞与額計算（等級×評語の交差テーブル）
- 2次集計：賞与原資との総枠調整（超過時は按分調整）
- 調整方法：最低評価点の設定 / 評定間ピッチの変更（ピッチ拡大=人数増・縮小=人数減）

## 4. Gravity Orbit AI 武装サービス v0.4 への直接反映候補

### [反映候補 1] 「年間12ヶ月 × 評価フェーズ別」運用タイムライン提案書テンプレート

**根拠：** ミナジン標準提案書が「Q別×施策種別の年間ガントチャート」を提案書の中核に据えている。Orbit 契約時の提案書でも同構造が有効。  
**Orbit v0.4 実装案：** 「Orbit 年間運用ロードマップ（12ヶ月 × 引力軸別アクション）」として標準提案書フォーマット化。Month 1-2（引力環境整備）/ Month 3-6（PDCA 月次）/ Month 7-12（自律運用・AI 自走）の3フェーズを視覚化。  
**差分（ミナジン vs Orbit）：** ミナジンは「評価制度の運用伴走」に特化 → Orbit は「組織の引力軸全体（Scan × Coaching × Shift 横断）の伴走」に拡張。

### [反映候補 2] 3ヶ月に1度の「評価制度運用状況チェック」→ 「引力スコア定期診断（QSI）」設計

**根拠：** ミナジン 22-2「評価制度運用状況チェック（3ヶ月に1度）」が、伴走の「ゲート」として機能している。  
**Orbit v0.4 実装案：** QSI（Quarterly Satisfaction & Influence score）として、Q ごとに CODE スコア・SCAN スコアの変化を定量測定し、Orbit 継続の判断基準と連動させる。軸15個ゲート設計（Orbit v3.1）の「中間チェックポイント」として実装可能。  
**AI 武装ポイント：** サーベイ自動送信 → 集計 → スコア変化レポート生成をエージェント化すれば、コンサル工数ゼロで QSI 実施可能。

### [反映候補 3] 目標添削サポートの「良い例/悪い例フレームワーク」→ AI 目標添削ボット設計

**根拠：** ミナジン 21-4「目標添削サポート」が運用フォローの中で最も工数がかかる施策の一つ。Orbit 月次伴走でも目標設定品質の問題は頻出。  
**Orbit v0.4 実装案：** 「Orbit 目標添削 AI」として、Good/Bad 例を学習させた LLM プロンプトで目標文の品質チェックを自動化。SMARTゴール基準 + Gravity 引力軸との整合確認を自動レポート化。  
**AI 武装ポイント：** 人事担当が目標文をテキスト貼付 → 即時フィードバック（5段階スコア + 改善提案）→ 添削工数を 90% 削減。

### [反映候補 4] 評価者バイアス可視化（評価者間散布図）→ 「評価者キャリブレーション AI ダッシュボード」

**根拠：** xlsx② シート「3_参考・評価者別」が評価者間の点数散布図を可視化している。中小企業では評価者バイアス（甘辛・中心化傾向）が組織不公平感の最大要因の一つ。  
**Orbit v0.4 実装案：** Gravity Scan の評価データ + Orbit 月次 1on1 データから「評価者バイアスヒートマップ」を自動生成。特定の評価者への「厳しすぎ/甘すぎ」アラートを HR 担当に自動通知。  
**差分（ミナジン vs Orbit）：** ミナジンは手動 Excel → Orbit は API 自動取得 + リアルタイム可視化。

### [反映候補 5] 評定決定「2次集計・賞与原資調整」ロジック → 「賞与原資 AI 試算エンジン」

**根拠：** xlsx② の2段階集計（1次：ポイント換算 → 2次：原資調整）は SMB 社長が毎半期最も時間を使う意思決定の一つ。「全員 A なのに原資が足りない」調整作業が属人化・ブラックボックス化している。  
**Orbit v0.4 実装案：** 評定データをインプットするだけで「原資別・等級別・評語別の賞与額シミュレーション」を即時出力する AI 試算ツール。「原資を X% 削減した場合の個人別影響額」も自動算出。  
**AI 武装ポイント：** 社長の意思決定を「試算1回30分 → 5秒」に短縮。

### [反映候補 6] 考課者研修「期首/期中/期末 × 力点設計」→ Orbit 管理職コーチングプログラム改訂

**根拠：** ミナジン研修設計が「期首=目標設定（最重要）/ 期中=進捗管理 / 期末=評価フィードバック」と明示的に力点を分けている。現在の Orbit 月次伴走では管理職スキル開発の力点設計が不明確。  
**Orbit v0.4 実装案：** 「Orbit 管理職ラーニングパス（12ヶ月）」として、期首Month（目標設定スキル）/ 期中Month（1on1・進捗管理）/ 期末Month（評価・フィードバック）の3サイクルで管理職育成コンテンツを構造化。  
**差分（ミナジン vs Orbit）：** ミナジンは研修単発 → Orbit は月次 1on1 伴走と統合した継続学習。

### [反映候補 7] 「制度設計（Shift） → 運用伴走（Orbit）」の2フェーズ接続設計の明文化

**根拠：** ミナジンの「設計 → 導入 → 運用」フェーズ構造が、GrowthFix の「Shift（制度設計） → Orbit（運用伴走）」の2サービス接続そのものと対応している。ミナジンは1社で一貫提供するが、GrowthFix は Shift R/C（設計）+ Orbit（運用）として分離提供している。  
**Orbit v0.4 実装案：** Shift 完了クライアント向けの「Orbit スムーズ移行プロトコル」として、Shift 最終成果物（人事制度設計書・等級/評価設計）を Orbit Month 1-2 の引力環境整備インプットとして自動連携する標準 SOP を策定。  
**差分（ミナジン vs Orbit）：** ミナジンは単一サービス内完結 → GrowthFix は Shift → Orbit のクロスセル設計として商業的に重要。Orbit 提案書に「Shift 卒業者特典：Month 1の引力環境整備コンサル無料」等の差別化オプション設計も可能。

## 5. 仮説 E v0.2 営業資料 v0.4 への組込候補

### [組込候補 1] 「車の両輪」メタファーの引力経営的翻訳

**ミナジン原文：** 「評価制度の『構築』と『運用』は車の両輪、どちらかが欠けても組織はドライブしていきません。」

**引力経営版：** 「組織の引力を作る（Shift）と、引力を継続させる（Orbit）は車の両輪。引力がなければ人は集まらず、継続しなければ引力は薄れる。」

**営業資料 v0.4 への組込：** Shift → Orbit のクロスセル訴求の核心コピーとして活用。SMB 社長が「制度作るだけじゃ機能しない」という経験的直感を持っていることを前提にした訴求。

### [組込候補 2] SMB 3大課題の明示（「運用できるか不安 / 専門家がいない / 確認がしたい」）

**ミナジン原文 3 課題：** 
1. 評価制度を導入したが運用できるか不安
2. 社内に専門知識を持ったスタッフがいない
3. 定期的に上手くいっているか確認がしたい

**引力経営版（Orbit 向け訴求）：**
1. 組織の引力を設計したが、実際に機能させられるか不安
2. 社内に引力経営を実践できる管理職がいない
3. 引力スコアが上がっているか定期的に確認したい

**営業資料 v0.4 への組込：** Orbit の「誰に効くか」セクションでの課題共感コピーとして活用。SMB 社長向け訴求の冒頭「3つの不安に答えます」フレームで展開。

## 7. 新規 AI エージェント候補

### [候補 1] orbit-bonus-calculator（賞与原資 AI 試算エンジン）

**機能：**
- 入力：社員マスタ（等級/評語）+ 賞与原資総額
- 処理：等級×評語のポイント換算 → 原資按分計算 → 1次/2次集計シミュレーション
- 出力：個人別賞与額 + 原資シミュレーション（±X% の感度分析付き）

**実装難易度：** 低（xlsxの計算ロジックをPython化するだけ）  
**業務インパクト：** 社長の賞与決定作業を30分→5秒に短縮。Orbit 月次伴走の付加価値ツールとして提供。  
**既存候補との関係：** Phase 3 で挙がった「pm-feedback-scorer」（プレイド由来）と補完関係。プレイドは大手向け PM フィードバック → ミナジンは SMB 向け賞与計算と対象規模が異なる。
