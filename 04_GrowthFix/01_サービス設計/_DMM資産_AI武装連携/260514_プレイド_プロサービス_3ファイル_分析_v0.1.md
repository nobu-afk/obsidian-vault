## 1. PLAID ALPHA（プロフェッショナルサービス紹介）構造抽出

### 1-1. サービスポジショニングの構造

**ミッション文言（Slide 2）：** 「顧客中心経営を実現する一番のパートナーでありたい」

これは「プロダクト（KARTE）の売り先」ではなく「顧客経営の変革パートナー」として自己定義している。プロダクト企業がプロフェッショナルサービスを立ち上げる際の典型的な自己再定義パターン。

**サービス体系（4 層構造）：**

```
コンサルティング（1）
  └ 顧客理解・顧客戦略/ロードマップ立案
  └ 顧客育成戦略、UI/UX戦略立案
  └ 顧客分析（新規・ロイヤル）
  └ データマート設計・KPI可視化

インテグレーション（2）
  └ システム計画策定〜導入の推進・管理
  └ プロジェクトプランニング
  └ 要件定義/設計/実装/テスト
  └ 体制構築/トレーニング

グロース（3）
  └ グロース支援（PDCA運用支援）
  └ テクニカルサポート（実装支援）
  └ 内製化支援/トレーニング

ソリューション（4）
  └ KARTE for Sales（営業強化・効率化）
  └ チラシ DX（QR計測→行動分析）
  └ イベント DX（会場行動データ統合）
```

**コアコンピタンス（Slide 4, 12 等）：** CX プラットフォーム / KARTE 導入・運用メソッド / 業界・業種別ベストプラクティス / リファレンスアーキテクチャ / 標準プロセス・テンプレート / CX・マーケ施策事例 / 標準イベント・データ活用策 / ソリューション企画・開発・導入・運用

**構造的特徴：** プロダクト（KARTE）という「武器」を前提に、コンサルティング→インテグレーション→グロースという「一気通貫」を提供することで、顧客の依存関係を深め、単価・期間を最大化する設計。

### 1-2. 提供形態の設計

**チーム構成（Slide 8-9）：**
- 専門ドメイン：マーケティング / データアナリティクス / デジタルテクノロジー / クラウドアーキテクチャ / プロジェクトマネジメント / グロースハック
- 20 名以上のプロフェッショナル
- リーダー陣自らプロジェクトにディープダイブする（=リーダー常駐モデル）

**グロースサービスの特徴（Slide 36, 38）：**
- 顧客オフィスに週 2 日で常駐
- 業務端末・アカウントをクライアントから貸与された状態で社内 MTG にも同席
- 「チームのメンバーとして」という表現（外部委託ではなく内部化感覚）
- 最短契約期間：3 ヶ月単位

**施策プロセス（Slide 33）：** 企画→要件定義→実装→振り返り（共通）の 4 フェーズ。各フェーズで ALPHA が担当する具体タスクを細分化。

### 1-3. 価格設計の構造（金額除外・比率構造のみ）

- コンサルティング：スコープに応じた個別見積（数百万円〜の表記があるが金額は除外）
- インテグレーション：プランニング / 導入（インプリ）/ 運用準備の 3 フェーズ別見積
- グロース：3 ヶ月単位の継続契約
- ソリューション：スポット型プロダクト別（チラシ DX / イベント DX）

**価格設計の哲学（Slide 13 の課題設計から読解）：** 「数百万円〜」という開放型を使い、スコープ・体制・期間で最大化する準委任契約型が基本形。

### 1-4. 実績の見せ方（マスキング済み）

**事例構造（各支援事例スライドより）：**

**インテグレーション事例（Slide 26-28）の構造抽象化：**

- 事例 1（CS サービス系）：次世代 CS モデル構想→アクションプラン策定→社員主体の自走体制確立
- 事例 2（会員数大規模アプリ系）：複数 MA・CDP統合→KARTEを主軸としたアーキテクチャ移行→ライセンス・運用コスト大幅削減

**グロース事例（Slide 36-37）の構造抽象化：**

- 事例 1（デジタルエンタメ系）：週 2 日常駐→KARTEを使った施策実装伴走→スピーディーかつ安全な実装
- 事例 2（複数プロダクト活用大手）：複数プロダクト同時並行→ユーザー ID 軸でデータ統合→全チャネルの成果最大化

### 1-5. プロダクト × ヒトのハイブリッドモデル（Slide 64）

「エンタープライズ企業群への展開を強化するため『プロダクト×ヒト』のハイブリッドモデルにシフト」

## 2. STUDIO ZERO ホワイトペーパー（pptx）構造抽出

### 2-1. サービスの哲学・ミッション

**STUDIO ZERO のミッション（Slide 4）：**

「各産業のフラッグシップとなる事例を創出する事業家集団」

「日本を代表する大企業や地域経済を支える中小企業、スタートアップ企業、行政・公的機関などのパートナーと共に、データを活用した顧客視点での新規事業創出や既存事業の変革を目指しています。」

**注目点：** 「フラッグシップとなる事例」という表現。顧客数を最大化するのではなく、産業ごとに「先行旗印事例」を 1 件つくることを目的とする。スケール戦略ではなく「シンボル事例創出」戦略。

### 2-2. 業界での立ち位置（Slide 5-7）

**4 象限マップによる自己定義：**

```
縦軸：論理的 ←→ 創造的
横軸：主観的 ←→ 客観的

既存プレイヤーの位置づけ：
  広告代理店        → クリエイティブアプローチ（創造的 × 主観的）
  経営顧問・アドバイザリー → エグゼクティブアプローチ（創造的 × 主観的寄り）
  ビジネスコンサル   → マーケティングアプローチ（論理的 × 客観的）
  デザインコンサル   → デザイン/CXアプローチ（創造的 × 客観的寄り）

STUDIO ZERO のポジション：
  「各アプローチを得意とするメンバーをアサインしPJ特性に合わせてリードする」
  → 事業的アプローチ + 経営的アプローチに拡大中
```

**内部カンペ的記述（Slide 5-7 の「カンペ」テキストより）：** 「ZERO で一定の顧客経営層から支持を受けた VP 以上は、より経営のパートナー的な立ち位置へ進出しようとしている」という内部的野心が明示されている。これは ZERO が「CX コンサル」から「経営パートナー」への格上げを狙う戦略的動きであることを示す。

### 2-3. 差別化軸（Slide 8-9）

**「実行力支援」の哲学（Slide 8）：**

| 一般的なコンサル | ZERO のアプローチ |
|---|---|
| 戦略設計→実行はクライアントに一任 | 戦略設計＋実行のケイパビリティまで支援 |
| 戦略と実行の間に認識・行動のズレが発生 | 半常駐型・プレイング型のコミットで業務実行支援 |
| 「絵に描いた餅」になりがち | 成果が出るまで伴走 |

**「内製化支援」の哲学（Slide 9）：**

| 外部委託の問題 | ZERO のアプローチ |
|---|---|
| ナレッジが企業内に残らない | 人が育ち、ナレッジが蓄積される仕組み・組織化を支援 |
| 変革人材が育たない | 伴走を通じた内部人材の成長 |
| 主導権が外部に流れる | クライアント主体の自走体制確立 |

事例として東京ガス様（[マスク対象のため「エネルギー供給事業者 D 社」]）のスクラム開発内製化支援を掲載。

### 2-4. 事業ポートフォリオ（Slide 10）

```
伴走型 新規事業開発サービス  → 爆速で新規事業開発を伴走する
伴走型 事業変革サービス     → 顧客基点で事業の勝ち筋を探し事業成長に貢献
伴走型 組織変革サービス     → 事業変革をリードする人材を育成し組織の変革を加速
伴走型 行政経営DXサービス   → 住民・地域事業者・観光客基点の伴走支援
伴走型 物流価値創造サービス  → 物流に価値を与え適切なインフラ構築を目指す
```

全て「伴走型」をプレフィックスに使用している。「伴走」が ZERO のブランドワードとして機能している。

### 2-5. 顧客事例（Slide 11-13）のパターン

**事例 1：大手たばこ事業会社（[マスク対象：大手消費財メーカーF 社]）— 組織変革**

- 声：「コーチングという手法は思考を洗練させるという意味で非常に意味のある取組」

**事例 2：エネルギー供給事業者 D 社 — 事業変革**
- 課題：電気・ガス自由化で顧客との関係性向上が重要になった

**事例 3：旅行業 E 社 — 新規事業開発**
- 課題：コロナ後の事業士気向上と新収益機会創出
- アプローチ：公募型新規事業推進プロジェクトの立ち上げ・運用
- プロジェクトの考え方：「審査・起案段階で失敗しがちな起案・進め方を事前に見極め事業案を磨き込む」「PoC段階でPDCAの回転速度を上げる」

## 3. STUDIO ZERO 提案書テンプレート（pptx）構造抽出

### 3-1. 提案書の哲学（Slide 1 の内部コメントより）

提案書 Slide 1 には通常の提案内容の前に、ZERO が自社の提案担当者向けに残した 3 つの哲学が記載されている：

1. **「与えられた課題（提案内容）に限定せず、クライアントの事業成長に必要な本質的な課題を解決する提案をする」**
   → RFP の制約を超えて経営レベルの課題まで踏み込む。クライアントの「いった範囲」ではなく「あるべき範囲」で設計する姿勢。

2. **「単にプロジェクトの進め方・作業を説明する提案には意味がなく、提案時点でしっかりとした仮説を持つ（たとえ間違っていても）」**
   → 提案書は「作業説明書」ではなく「仮説プレゼン」である。仮説の正誤より仮説を持つこと自体が信頼の源泉。

3. **「Why ZERO?を明確にする（他ではなくを選ぶ合理的な理由）」**
   → 差別化の根拠を提案書内で論証する。

### 3-2. 提案フレームワーク（アジェンダ構造）

**標準章立て（Slide 3, 7, 10, 12, 16, 19, 21, 24, 26 のアジェンダスライドより）：**

```
1. プロジェクトの背景と目的
2. 目指すべき将来像
3. プロジェクト成功の要諦・弊社提供価値
4. プロジェクトアプローチ
5. 推進スケジュール
6. 実施タスク・想定アウトプット
7. ご支援体制
8. お見積り
Appendix
```

**各章の内部設計メモ（ZERO 担当者向け注記より抽出）：**

| 章 | 重要設計指針 |
|---|---|
| 1. 背景と目的 | 「RFPの内容に捕らわれすぎず、本来達成すべきゴールを顧客目線・企業価値目線で定義。提案前に最低3回は打合せを実施して探索」 |
| 2. 将来像 | 「初期仮説はプロジェクトを通じて方向転換になることもあるが、提案時点で想定している実現したい世界勘を俯瞰的な概念図で示す」 |
| 3. 要諦・提供価値 | 「単にやることの列挙ではコンペでの差別化ができないので、自社固有のアセット（方法論・ノウハウ・アイデア・プロダクト・データ・専門人材等）をどのように活かすかを簡潔かつ大胆にアピール」 |
| 4. アプローチ | 「ZEROは答え（仮説）を持っているということを提案段階で示す」 |
| 5. スケジュール | 「1〜2週程度で1タスク」「3つ以上の同時並行タスクがあるとデリバラビリティ怪しい」 |
| 6. タスク・アウトプット | 「成果物という記載は請負契約を連想させるので、作成支援物またはアウトプット（曖昧な表現）とすることが多い」|
| 7. 支援体制 | 「アドバイザーをつけることでデリバリー品質の維持・向上への期待値と安心感を高める。アドバイザーは見積に含めないことが多い」 |
| 8. 見積り | 「複数フェーズに分かれる案件では、次以降のフェーズの概算を示すと良い（先回りで予算確保の動きを取ってもらいやすい）」「値引き交渉では単価を下げるのではなく、担うタスク自体を削る（スコープ縮小による金額低下と見せる）」 |

### 3-3. WBS 構造

**Lv1 タスク体系（Slide 13, 17, 20 より）：**

```
フェーズ 1：現状把握・仮説立案
  A. 初期仮説策定
  B. 経営層インタビュー
  C. 顧客調査

フェーズ 2：改革方針策定
  D. 目指すべき体験設計
  E. 顧客受容性評価（PMF）
  F. ビジネスモデル定義
  G. 実現に向けた施策定義
  H. ビジネスケース策定

フェーズ 3：実行計画策定
  I. 実行計画策定
```

**スケジュール設計原則（Slide 17-18 より）：**

- 週単位で定義（月単位は月の中のタイミングが曖昧になる）
- 1〜2 週間で 1 タスクを目安（1 ヶ月以上続くタスクは間延び・遅延リスク）
- タスク間の Input/Output の連関を明示
- 3 つ以上の同時並行タスクはデリバラビリティに問題が出る

### 3-4. 会議体設計（Slide 18）

**標準会議体の構造：**

| 会議体 | 頻度 | 出席者（ZERO 側） | 主目的 |
|---|---|---|---|
| 週次進捗 | 週次 1 時間 | PM + メンバー | WBS 進捗共有・課題対応 |
| 経営層インタビュー | PJ 2-3 週目（1 時間/人） | 責任者 + PM | 仮説に基づく経営課題討議 |
| 中間報告① | X 月 X 週目 1 時間 | 責任者 + PM + メンバー | 目指すべき体験・ビジネスモデル討議 |
| 中間報告② | X 月 X 週目 1 時間 | 責任者 + PM + メンバー | PMF 評価結果・改善方針共有 |
| 最終報告 | X 月 X 週目 2 時間 | 責任者 + PM + メンバー | 事業強化方針・ビジネスケース・次フェーズアクション |

**中間報告の位置づけ（内部メモより）：** 「単にプロジェクトのちょうど真ん中ではなく、意味のあるアウトプットが完成するタイミングで決定する（Point of No Return）」

### 3-5. 支援体制の設計（Slide 22-23）

**標準体制図の役割定義：**

```
ZERO 側：
  プロジェクト責任者（MD/P：パートタイム）
  プロジェクトマネジャー（F：フルタイム）
  デリバリーメンバー（F：フルタイム × 複数）
  アドバイザー（P：パートタイム × 有識者）

クライアント側：
```

### 4-2. プロサービス設計の 6 構造的特徴

**特徴 1：「一気通貫」と「フェーズ分割」の両立**
- ALPHA：コンサルティング→インテグレーション→グロースを一気通貫で提供するが、各サービスは単独でも購入可能
- ZERO：「現状把握・仮説立案→改革方針策定→実行計画策定」の 3 フェーズが標準構成で、フェーズ単位で契約を区切る
- GrowthFix への示唆：CODE→Scan→Shift（R/C/Full）→Orbit という動線設計は「一気通貫の提示 + フェーズ購入の現実」として同じ構造

**特徴 2：「実行力」を差別化源泉にする**
- ZERO の最重要 KW が「実行力支援」：戦略だけ渡して終わりではなく、実行まで伴走する
- ALPHA のグロースサービスが「週 2 日常駐」「業務端末貸与」「社内 MTG 同席」という形で最も強い実行サポートを提供
- GrowthFix への示唆：Shift C（躍動組織・6 ヶ月）の差別化も「実装まで伴走する」で訴求できる

**特徴 3：「内製化支援」でロックインではなく「離脱 OK」という逆説的ポジション**
- ZERO は「ナレッジが外部に流れる問題」を逆手に取り「内製化させるためにいる」と訴求
- 結果として顧客の「依存したくない」という感情バリアを消している
- GrowthFix への示唆：Orbit の「AI 自動化 + 内製化」ラインは同じ設計ができる（Orbit を卒業できる状態を目指す、が実は継続の最強動機になる）

**特徴 4：「半常駐・プレイング型」という新しいエンゲージメントモデル**
- ZERO は「半常駐型・プレイング型などのコミットで業務の実行支援」を差別化として明示
- ALPHA のグロースも「週 2 日常駐」「チームのメンバーとして」と表現
- GrowthFix への示唆：Shift C / Orbit で「石井が顧問として定期常駐」ではなく「AI エージェントが常時稼働 + 石井が月次でプレイング参加」というハイブリッドモデルを設計できる

**特徴 5：「仮説先行型」提案スタイル**
- ZERO の提案哲学「提案時点でしっかりとした仮説を持つ（たとえ間違っていても）」

**特徴 6：「スコープ縮小→金額低下」という値引き対応の型**
- ZERO の内部メモ「値引き交渉では単価を下げるのではなく、担うタスク自体を削る（スコープ縮小による金額低下と見せる）」

- GrowthFix への示唆：見積交渉で「Scan だけ」「Week 1-2 だけ」というスコープ分割を当初から設計しておくことで、「削り交渉」に対して事前防衛できる
