### 1-5. 職能評価の 2 軸構造

**職能 1（主軸）＋ 職能 2（副軸）** の 2 次元マトリクス設計：
- 職能 1 = 本人の主要スキル領域
- 職能 2 = 副専門・兼任スキル領域（なしの場合もあり）
- 例：「サーバーサイドエンジニア（職能 1）× プロジェクトマネジメント（職能 2）」のように、T 字型人材の可視化を狙った設計

### 1-6. 集計・分析フロー設計

ファイルの構造から読み取れる運用フロー：

```
1. import_基本情報シートに HRシステムから社員マスタを一括インポート
2. data シートで個人別に職種・職能・等級を付与
3. 集計シートで本部×部室×等級×職種の 4 軸クロス集計
4. グラフシートで等級分布の可視化
5. 職種パターンシートで職能 1・2 の組み合わせを管理
6. ピボットテーブルで経営層向けレポート出力
```

**backup シートの存在（2018/12→2020/03 の 2 時点比較）**：
- 集計_backup = 2018/12/11 時点の集計（古い版）
- 集計 = 2020/03/04 時点（最新版）
- 2 時点で比較することで、等級分布の変化・成長を追跡する設計思想が確認できる

### 2-5. 手当設計の全体構造（給与体系における「手当」の位置づけ）

com シートのヘッダー構造から読み取れる給与体系：
```
年俸設定
  ├── ベース給（本体）
  └── 加算給（オプション）
＋手当 12 種類（月次支給）
  ├── 法定必須系：残業・深夜・土日
  ├── 固定付加系：役職・業務・職種・住宅
  ├── 一時金系：特別手当（四半期）
  └── 実費精算系：通勤（課）・通勤（非）
```

年俸がベース給＋加算給に分割されており、加算給は成果・評価に連動する可変部分と推測。手当は年俸とは別立てで月次支給する設計。

## 4. 3 ファイル横断パターン（制度設計の構造的特徴）

### 4-1. 横断パターン 7 選

**パターン 1：「制度の分離設計」原則**

就業規則は骨格のみを定め、詳細は「賃金規程」「育児介護休業規程」「テレワーク規程」等の別規程に委任する構造。就業規則本文は 88 条で完結しているが、実際の制度詳細は複数の別規程に分散している。スキル評価も「人事制度規程」等の別ドキュメントがあるはずだが今回のファイルには含まれていない。

→ **大企業 HRBP の制度設計原則**：変更頻度の高い部分（賃金金額・手当金額）を就業規則から分離し、法改正時の届出コストを最小化する。

**パターン 2：「2 時点スナップショット比較」による制度効果測定**

スキル評価 xlsx は 2018/12 と 2020/03 の 2 時点分の集計を保持（集計 vs 集計_backup）。手当 xlsx は 12 ヶ月の時系列データを保持。これにより「制度導入前後の変化」「人材構成の変化」を数値で追跡できる設計。

→ **時系列比較が制度 PDCA の前提**。点のデータではなく線のデータとして蓄積する。

**パターン 3：「法人格別」の制度並走設計**

手当 xlsx は COM（合同会社 DMM.com）と EXNOA（合同会社 EXNOA）の 2 法人のデータを同一ファイルで並走管理。就業規則は子会社 A 社（株式会社ハッシャダイ）が独立した規則を持ちながら DMM グループの制度設計思想を踏襲。

→ **グループ企業管理**：法人格の独立性を保ちながら、制度設計の SSOT（就業規則の基本骨格・手当種類の定義等）は本社 HRBP が主導。

**パターン 4：「職種特化型」スキル評価の先行実施**

スキル評価ファイルはエンジニアとデザイナー専用。全社員を対象にした評価制度ではなく、「まず技術職から始める」段階的アプローチ。ピボットテーブルでは「ディレクター」等の他職種も等級分布が確認できるが、職能評価は技術職のみ。

→ **評価制度の段階展開**：難易度の高い職種から先行し、横展開のたたき台とする。

**パターン 5：「等級不明者」が 25% 存在する課題開示**

等級不明者がエンジニア 103 名・デザイナー 22 名（合計 125 名規模・全体の約 25%）存在。これは「評価制度が整備されても実際の等級付与が追いついていない」ことを示す生データ。

→ **制度整備と運用実態のギャップ**が数値化されて見える。「制度があっても機能していない」ゾーンの可視化。

**パターン 6：「手当の月次変動」で人員変動と労働実態を同時モニタリング**

通勤手当の緩やかな増減は入退社の実態を反映。特別手当の季節性は賞与支給実態を反映。残業手当の月次変動は繁閑差を反映。手当データは「コスト管理」だけでなく「組織の健康状態モニタリング」として機能する。

→ **手当データは先行指標**として使える。残業手当が特定部門で急増すれば燃え尽き予兆として検知可能。

**パターン 7：「子会社への就業規則展開」のコストとリスク**

子会社 A 社は自社名義の就業規則を持ち、労働基準監督署に届出。PMI（子会社統合）時に就業規則の整合が必要になる。MOON X 分析（260514_MOONX）と合わせると、PMI 時の制度統合は「就業規則の書き換え」「手当体系の統一」「等級制度の統合」という 3 層の作業が必要であることが浮かぶ。
