## 3. 機密情報マスキング方針（最重要・必須遵守）

### 3.1 絶対除外（出力に含めない）

- 個人氏名（漢字 / カタカナ / 英字）
- メールアドレス
- 電話番号
- 給与・退職金・賞与金額
- 住所・生年月日
- 顧客社名・取引先名（事業領域だけ抽象化）
- DMM / MOON-X / プレイドの売上数字・顧客名
- 内部 URL / パスワード / Slack 通知設定値

### 3.2 抽象化処理（業務構造のみ抽出）

- 「○○本部」→「事業部 A」
- 「年商 ◯ 億」→「◯ 億規模」
- 「正社員 ◯ 名」→「正社員 ◯◯ 名規模」
- 「DMM 戦略 ◯◯」→「業務プロセスの構造として抽出」

### 3.3 抽出 OK（公開情報）

- 業務プロセス・ワークフロー
- 評価指標・KPI 設計の構造
- ナレッジ・テンプレート
- 質問項目体系（カテゴリレベル）
- 公開論文 / 標準フレームワーク

### 3.4 機密マスキングログ（必須）

各分析 MD の末尾に「機密マスキングログ」セクション（カテゴリ別件数）を必須記載。

## 7. リスク・注意事項

### 7.1 Dropbox トークン

- 約 60 日有効（next 失効目安：2026-07-12 頃）
- 失効時の修復手順：`06_開発/scripts/config/dropbox_token.json` の `"access_token"` 値の閉じクォート確認（260514 朝の修復例参照）

### 7.2 並列度

- 3 並列以下推奨（260513 教訓）
- 4 並列以上は stall リスク高（特に大きいファイル分析時）

### 7.3 機密マスキング

- **絶対除外項目（§3.1）を 1 件でも出力に含めると重大コンプラ違反**
- 各 Agent prompt で機密マスキング方針を明示必須
- 統合 MD でも機密マスキングログ累計を必ず最終確認

### 7.4 規模感

- 約 100 ファイル / 推定 200-300 MB 級の処理
- analysis-runner Sonnet 並列実行で実時間 3-5 時間想定
- メイン Node の context 容量に注意（個別分析結果は MD 保存 + 統合 MD で集約）

### 7.5 既存深掘り済との重複回避

- DMM HRBP Phase 1+2 で深掘り済 9 ファイルは再分析しない（§2.2 で参照のみ）
- 同じ業界横断パターンが既に v0.3 に記録されている場合は §1 で「v0.3 既発見 + MOON-X / プレイドで追加検証」と注記
