## 1. クライアントA社（ソフトウェア・人事制度説明資料）の構造

### 1-1. 資料目的・対象
- 2019年4月運用開始の新人事制度を、2020年4月追記版として社員全員に説明するための資料
- 等級制度・報酬制度・評価制度の3本柱を1冊に統合した「社員向け完成版SSOT」

### 1-2. 等級制度の設計

| 等級 | 名称 | 対応役職・レベル |
|---|---|---|
| 5等級 | EX | 本部長（高い専門性で組織業績に大貢献） |
| 4等級 | EM/SM | 課長・部長（専門性を活かしてチーム業績に責任） |
| 3等級 | E3/S3 | リーダー（工夫改善で業務に価値を創出） |
| 2等級 | E2/S2 | 担当職（定性業務を工夫改善） |
| 1等級 | E1/S1 | 一般職（定性業務を遂行） |

- **2コース体系**：技術職系（E）と営業・事務職系（S）で等級名を分離
- 昇格は「評価基準ノミネート → 役員・人事協議」の2段階ゲート
- 降格ルール：連続C評価以下 or 重大過失 or 指摘改善なしの3条件
- 最低滞留年数：1→2等は2年 / 2→3等は2年 / 3→4等は3年 / 4→5等は5年

### 1-3. 報酬制度の設計

- 基本給は等級別バンド制（標準値±上下限幅）
- 役職手当：リーダーI〜部長IIIの6段階（役割の難易度・期待役割の大きさで総合判断）
- 昇給テーブル：S評価 2,500〜10,000円 / A評価 2,000〜5,000円 / B評価 1,500〜4,000円 / C評価 0円 / D評価 降給
- **新卒・未経験技術職特別ルール**：入社後3年間は一律昇給額（B評価10,000円・A評価15,000円・S評価20,000円）で固定、4年目から通常テーブル適用

### 1-4. 評価制度の設計

| 評価区分 | ウェイト | 評価対象 |
|---|---|---|
| 重点課題評価（目標管理） | 30% | 定量的成果・個人ミッション達成度 |
| 職務行動評価（基準対比） | 70% | 等級別期待行動・日常業務遂行レベル |

- 評価サイクル：年1回（通期評価）、上期・下期の2フェーズで進捗管理
- 評価プロセス：目標設定面談 → 中間面談 → 自己評価 → 一次評価 → FB面談 → 最終確定
- 評価者体制：一般社員＝リーダー・課長が一次評価者、社長が最終評価者
- 年間評価得点：S評価85点以上 / A評価65〜84点 / B評価50〜64点 / C評価35〜49点 / D評価34点以下

### 1-5. 技術職1→2等級昇格要件（詳細スキルマップ付き）

- **必須項目**：ヒューマンスキル（コミュニケーション・タスク管理・問題相談）+ ビジネススキル（Office活用・議事録・PDCA）+ 業界知識（チーム構成・商流理解）
- **選択項目**：エンジニアスキル15項目中 5割以上でB評価・8割以上でS評価
- 技術スキルのみならず「ビジネスマナー・業界理解・対人スキル」を昇格要件に組み込んだ点が特徴的

### 1-6. SMB人事制度設計の「完成形パターン」として示す論点

- 「等級定義 → 昇降格ルール → 報酬テーブル → 評価体系 → 目標設定手順」を1資料で完結させた構造は、SMB向け人事制度のSSOT設計例として完成度が高い
- 新卒・未経験者向け特別ルールで「入社初期の不安除去 × 4年目以降の通常運用への移行」を設計した点は、採用競争力にも直結

## 2. クライアントB社（音楽イベント・人事制度構築MTG議事録）の構造

### 2-1. MTG基本情報
- 日時：令和元年7月（第1回MTG）
- 参加者：先方経営陣（代表・人事担当・経理担当）、ミナジンコンサルタント2名
- 議題：インタビュー結果共有・定量分析結果確認・人材ポートフォリオ導入

### 2-2. 現状課題の詳細（議事録から抽出）

**課題1：評価軸の不適合**
- 現状の評価3軸（個人目標30% / 能力評価40% / 態度評価30%）がスペシャリスト人材の仕事実態に合っていない
- アーティストマネジメントという専門職性の高い業務がマネジメント能力評価で埋もれている
- 「マネジメント9：スペシャリティ1」の業務配分なのに、評価軸にスペシャリティが反映されていない

**課題2：年齢構成の歪みによる組織硬直**
- バブル期採用のA層（勤続23〜28年・50代）と後輩世代（B層）の間に中間層が薄い「くびれ型」構造
- A層が組織ヒエラルキーに馴染まず、自律的・自己判断型の働き方が定着
- 30〜40代の中堅層が台頭してきた際に「公平性への疑問」が噴出するリスク

**課題3：報酬の逆転現象と管理職忌避**
- 年齢給・能力給が統合された年俸制に移行したことで、昇給管理が複雑化
- 管理職手当が低水準で、部長職の年収が一般社員と逆転するケースが発生
- アーティストマネジメントを志望して入社している社員が多く、管理職を目指したがらない
- 「部長たちが一般社員と同じ実務をやっている」ナイまぜ状態

**課題4：プロフェッショナル人材の流出リスク**
- プロフェッショナル人材（自立型アーティストマネージャー・プロデューサー）は社内に少数しかいない
- 対価バランスを求める層であるが、社歴・年齢優先の報酬設計で正当評価されず流出傾向
- 外部採用も難しい業種特性（アーティスト業界の特殊スキル要求）

**課題5：50代人材の活用設計不在**
- 57歳以降5%減・60歳以降8%減の年齢別減額ルールがあるが、成果・貢献と紐づいていない
- 「減額するとモチベーション低下」vs「減額しないと原資不足・若手が報われない」の構造的ジレンマ
- 「育成・開発ではなく活用」という視点での50代向け個別設計が未着手

### 2-3. コンサルタントの提示方向性（議事録から抽出）

- 「A層は育成でも開発でもなく活用」という枠組みで個別設計する方向
- 管理職要件の明確化 → 役割に見合った報酬設計 → 管理職インセンティブの整備
- プロフェッショナル人材はアドベンチャー制度（社内公募）または外部採用で確保
- 報酬体系全体を見直す際は手当も含めた包括的議論が必要
- 10年後の人件費シミュレーションを作成して判断材料とする方針

### 2-4. SMB診断フェーズの典型論点

- 人材ポートフォリオ（4象限）を使った「組織の引力タイプ分析」に近い議論が自然発生していた
- 「マネジメントかスペシャリストか」という二軸の明確化要求は、SMB経営者に普遍的な課題
- 「部長になりたくない」問題の根本：報酬逆転 + 専門職志向 + 責任過重 = 管理職の引力消失

## 3. クライアントC社（屋外広告・現行分析確認事項テンプレ）の構造

### 3-1. 資料目的・位置づけ
- 「フロント商材開発用」というラベルが付いており、ミナジンが新規営業・見込み顧客向けの初回分析テンプレとして汎用化した資料
- 実クライアントのデータを使ったサンプル構成（「M社（サンプル）御中」という表記あり）
- 初回分析提案で使う「現状分析 → 課題提示 → 人事ポリシー設計 → サービス案内」の流れを体現

### 3-2. 分析コンテンツの構成

**レイヤー1：労務・人員構成分析**
- 年齢別従業員分布図（現在・5年後・10年後の3時点予測）
- 4つの構成型パターン解説（ピラミッド型 / 釣鐘型 / 逆ピラミッド型 / くびれ型）を使った現状診断

**レイヤー2：等級・報酬の課題特定**
- 現行の各等級別人数分布と管理職比率（世間水準15%・部長のみ6%との比較）
- 年齢別年収分布（25〜45歳の昇給ルールと45歳以上の未整備を可視化）
- 等級（役職）別年収分布（一般社員とマネージャー層の賃金逆転現象の指摘）

**レイヤー3：退職金・将来原資の課題**
- 10年以内の退職者集中による退職金原資不足リスク
- 「特定年齢層引き止め効果なし」「上位者と下位者のメリハリなし」「勤続一律インセンティブ」の3欠陥

**レイヤー4：人事ポリシー設計の必要性提示**
- ハーズバーグの動機付け・衛生要因を使った「給与は衛生要因（必要最低条件）にすぎない」の根拠づけ
- 事業方針 × 人事ポリシー5領域（採用・労務・人材育成・組織風土・人事制度）の連動設計フレーム
- 働き方改革関連法（有給義務化・残業上限・同一労働同一賃金等）の適用スケジュール

**レイヤー5：ワークショップ設計**
- パイロット評価単位（2〜3部門）を選定して制度試行する方法論
- 成果指標 → 応用行動 → 基本行動の3層評価項目設定ワーク
- ミナジンはファシリテーター役・たたき台提示・他社事例紹介を担当

### 3-3. 提案サービスラインナップ（C社提案書より）
- 人事ポリシー設計ワークショップ：採用・育成ポリシー設計 / 労務ポリシー設計（金額は機密マスキング除外）
- 人事制度改定コンサルティング / 規定コンサルティング / 勤怠管理システム / 給与計算アウトソーシング

### 3-4. 「フロント商材」として設計された分析テンプレの特徴

- 「現行分析でいくつの課題が出るか」を予測し、各課題に対応するサービスがカタログとして提示される設計

## 4. クライアントD社（ITサービス・操作説明資料）の構造

### 4-1. 資料目的・位置づけ
- MINAGINEシステム（人事評価SaaS）を導入済みの先方に対し、操作手順を標準化するオンボーディング資料
- 人事部（Administrator）と一次評価者・二次評価者・各社員の役割分担を明示

### 4-2. 評価フローの全体像

```
期首：評価シート配布（管理者） → 目標入力（社員） → 目標レビュー・承認（管理者）
         ↓
期末：自己評価入力（社員） → 一次評価入力（一次評価者） → 二次評価入力（二次評価者）
         ↓
     最終評価確定（処遇入力・等級確定）（管理者） → CSV出力（賞与シート連携）
```

### 4-3. SMBが躓く典型ポイント（操作説明から逆引き）

- **評価シート配布前の最終確認**：社員番号・氏名・部署・評価者・等級・メールアドレスが固定されるため、割り当て前に社員情報の確認が必須
- **ウェイト合計100の強制**：目標入力時のウェイト合計が100でないと提出エラー → SMBの現場では担当者が混乱しやすい
- **下書き保存と提出の誤操作**：「提出」後は修正不可。誤提出対応の5ケース別フローを専用スライドで解説
- **二次評価者の事前追加**：評価シート割り当て時点では一次評価者しか設定されないため、二次評価前に追加操作が必要
- **評価者変更の理由**：産休・異動等で評価者が変わった場合の変更履歴を残すための手順

### 4-4. 評価点数テーブル（D社設定）

| 評価 | 総合評価得点 |
|---|---|
| S | 89〜100点 |
| A | 79〜88点 |
| B+ | 63〜78点 |
| B- | 51〜62点 |
| C | 43〜50点 |
| D | 〜42点 |

- A社の5段階評価（S/A/B/C/D）に対し、D社は6段階（B+/B-の分割）で設定。同じMINAGINEシステム上で顧客別カスタマイズが可能

### 4-5. システム定着フェーズのコンサル論点

- 操作説明資料が必要になる = 制度設計完了後に「使い続ける」フェーズの最初の壁
- SMBの人事担当は評価システムの管理者を兼任していることが多く、手順の標準化が离職リスク対策になる
- 「誤操作対応フロー」の整備がシステム定着を左右する（何かあったらどこに聞けばいいかが明確でないと使われなくなる）

## 5. 4クライアント横断パターン（SMB人事制度導入の典型4フェーズ抽出）

### フェーズ1：現状診断・課題可視化（B社・C社が該当）

**何が起きているか：**
- 経営者が感覚的に「何かおかしい」と思っているが、どこの問題かが言語化されていない
- 定量分析（年齢別分布・等級別人数・年収分布）を見て初めて「自社がどのタイプか」を認識

**典型課題パターン：**
- 人材ポートフォリオの歪み（逆ピラミッド型 / くびれ型）
- 賃金逆転現象（管理職 < 高年齢一般職）
- マネジメント不全（管理職がプレイヤー業務も担っているナイまぜ状態）
- 50代活用設計の不在

**コンサルタントの役割：** ヒアリング + 定量データ分析 + 他社事例の提示 → 「うちはこの型です」の認識合わせ

---

### フェーズ2：人事ポリシー設計（C社資料が明示）

**何が起きているか：**
- 制度設計に入る前に「何のための制度か」が経営陣の中で統一されていない
- 「採用・育成・評価・報酬・退職」の各領域で方針バラバラのまま制度だけ作ると使われなくなる

**典型課題パターン：**
- 事業方針と人事ポリシーの不連動
- 「褒めるべき行動・報いる仕組み」の言語化不足
- スペシャリストコースの有無（管理職にしかインセンティブがない問題）

**コンサルタントの役割：** ワークショップ形式でファシリテーション → 経営陣が「自分たちで決めた」感を醸成

---

### フェーズ3：制度設計・社内展開（A社資料が完成形を示す）

**何が起きているか：**
- 等級・報酬・評価の3本柱を連動させた制度を設計し、社員向けに説明する段階
- 「制度を知らない」「なぜそのルールなのかわからない」が現場混乱の主因

**典型課題パターン：**
- 昇格ルールの透明性不足（どうすれば上がれるかが不明）
- 評価区分のウェイト設定（何を重視するか）の未決
- 新卒・未経験者向けの特別ルール設計

**コンサルタントの役割：** 制度設計の文書化 + 社員説明会資料作成 + 年間スケジュール設計

---

### フェーズ4：システム定着・運用支援（D社資料が体現）

**何が起きているか：**
- 制度は決まったが、評価システムの操作が定着せず「使いづらい」「誰に聞けばいいか不明」状態
- 年2回の評価サイクルのたびに管理者が手順を忘れる

**典型課題パターン：**
- 誤操作による評価データ損失リスク
- 評価者交代時の引き継ぎ漏れ
- CSV出力と賞与計算の連携ミス

**コンサルタントの役割：** オペレーションマニュアル整備 + ヘルプデスク設置 + 年度初めのセットアップ代行

## 6. R/C/Orbit AI武装サービスへの反映候補（3〜5件）

### 反映候補1：Gravity Shift C（組織引力診断）への「人材ポートフォリオ自動生成」
