### 2-3. HRBP の社内営業「提供価値訴求」の構造

**訴求の 3 層設計**：

**層 1：Why（存在意義）の共有**
- 「DMMの次の成長を創る」← 全社視点のミッション
- 「人・組織で事業に貢献する」← HRBP のミッション
- 「ひとり一人が意欲・能力を最大発揮する」← 人材開発ポリシー
- → **Why を 3 段階で階層化**し、事業部側が「自分ごと化」できる粒度まで落とす設計

**層 2：守備範囲の可視化**（Slide 3 が核心）
- 認知 → 興味 → 応募 → 選考 → オファー → 内定 → 承諾 → 入社 → オンボーディング → 活躍 → ロイヤリティ → アルムナイ（退職後再入社）
- **入社前（採用ファネル）と入社後（エンゲージメント）を一枚の図で可視化**
- HRBPの守備範囲 = このライフサイクル全体

**層 3：具体事例による実績訴求**
- 組織改善実例（DMM.make AKIBA 等・事業部名は抽象化）
- 事業縮小・撤退時のアウトマネジメント事例（「BPチームが現場に入り込み、ピープルマネジメントや本部間調整を引き受け実施」）

### 2-4. 事業部 ⇔ HRBP 役割分担の設計

Slide 7 が社内営業の核心設計：

| 役割 | 事業部側 | HRBP 側 |
|---|---|---|
| スタンス | パートナーとして協働 | パートナーとして協働 |
| 具体 | 活発な意見提示 / 必要な情報の提供 / 建設的な議論 / 社内調整 | **議論の進捗管理 / 仮説（たたき台）の提示 / 検討用ツールの提供 / 他グループ事例の紹介 / 検討内容の整理** |

**HRBP の価値提供 8 項目（Slide 7 から抜粋・抽象化）**：
1. 目標設定の書き方へのフィードバック（MBO 設計支援）
2. 採用ペルソナ設定・採用活動サポート（全レイヤー対応）
3. エンゲージメント改善 + 人事課題の定期レポート支援
4. 他部署からの異動調整（課題解決型人材の移動）
5. 新卒〜マネジメント直前までの育成施策提案・実施
6. 事業部長幹部向け MVV（ミッション・ビジョン・バリュー）策定支援
7. 事業責任者・幹部向け認知科学コーチング支援
8. 攻めの異動調整・優秀人材の積極採用・事業部提案

→ **「たたき台の提示」と「他事例の紹介」が HRBP の中核付加価値として明示されている**。

### 2-5. 採用支援のサービス設計（Slide 8-9）

**採用戦略立案の 6 ステップ確認事項**：

```
ゴールの確認：なぜ採用が必要か・採用後どんな仕事を任せるか
戦略の確認：ビジョンとの整合性
戦術の確認：採用ルート・採用状況・かけられるコスト
従業員の現状：入社理由・退職理由
採用状況：うまくいっているか・公開求人票は何か
採用背景：採用必要性の根拠・採用後ミッション
```

→ **「なぜ採用が必要か」を最低 3 回聞く構造**（戦略・戦術・背景）。採用を「ヘッドカウント補充」でなく「事業課題の解決手段」として位置づける。

**採用支援の 7 フェーズ**：
1. 採用ペルソナ設定支援
2. 採用手法選定・提案
3. オペレーション（求人作成・面接調整・応募者対応）
4. 選考（書類・人事面談・面接実施）
5. モニタリング / 要件・手法の見直し
6. クロージング支援（シナリオ作成・面談調整・エージェント連携）
7. 入社後フォロー（ガイダンス・採用担当者面談）

## 3. 面接官研修（基礎）（pptx）構造抽出

### 3-1. マネジメント育成版との差分（既分析との比較）

| 観点 | 基礎版（本 MD・初級 TL 向け）| マネジメント育成版（Phase 1 既分析）|
|---|---|---|
| 対象者 | 初級チームリーダー・初めて面接官になる人 | 採用に責任を持つマネジメント層 |
| 作成年 | 2020 年 11 月（研修実施日）| 不明（推定 2021〜22 年）|

| ロープレ構成 | 3 セット × 18 分（面接官役/求職者役/オブザーバー役） | 同構成 |
| バイアス教育 | **4 種詳細**（確証 / ハロー / 類似性 / メラビアン） | 同内容 |

## 4. 3 ファイル横断パターン

### 4-1. DMM HRBP の採用フローと社内浸透の構造的特徴

3 ファイルを横断して浮かび上がる **DMM HRBP の設計哲学 4 つ**：

**哲学 1：「採用 = 事業課題の解決」という再定義を徹底**
- 社内営業資料：採用戦略立案で「なぜ採用が必要か」を 3 段階確認
- 面接官研修：「選んでもらう」マインドセットを最初 15 スライドで徹底
- KPI 管理：本部別辞退者数まで管理 → 採用後の定着を採用の成否として追跡

**哲学 2：「HRBP = たたき台提供者」のポジショニング**
- 社内営業 Slide 7：HRBP の主役割が「仮説（たたき台）の提示」と「他グループ事例の紹介」として明示
- = 答えを持って来る専門家ではなく、「共に考えるパートナー」として設計
- → 事業部の主体性を奪わない・意思決定権を事業部に残す

**哲学 3：データ可視化が説得の主武器**
- KPI xlsx：採用費用・採用単価・チャネル別応募数を月次で全本部に開示
- 社内営業 Slide 10：エンゲージメントスコア / 1on1 実施状況 / プロジェクト進行を DAG と連携してレポート化
- = 感覚ではなくデータで事業部を動かす設計

**哲学 4：「面接官 = 候補者体験の設計者」への再定義**
- 面接官研修：面接のサブゴール = 「会社のファンを増やす」と明示（Slide 29）
- 面接の目的 3 つ = 情報提供 / ファン獲得 / 見極め（この順序）
- = 見極め専念型の面接観から脱却し、採用ブランド構築の場として面接を捉える

### 4-2. KPI 管理と社内営業の接続構造

- KPI xlsx の「本部別辞退者数」管理が、社内営業資料の「エンゲージメント改善サポート」への自然な接続になっている
- 辞退率の高い本部 = HRBP 提案の優先ターゲットとして機能する設計

## 6. 仮説 E v0.2 営業資料への組込候補（3-5 件）

### 組込候補 1（推奨度 ★★★）：「採用 KPI の 3 大盲点」スライド

**提案内容**：営業資料に「SMB の採用管理が抜けている 3 軸」として提示。

```
盲点 1：採用単価を「職種 × 経路」の 2 軸で見ていない
  → 同じ「エンジニア採用」でもエージェント経由と DS では単価が 2〜3 倍変わる

盲点 2：辞退率を「本部別 / チーム別」で管理していない
  → 採用数は達成しているのに離職・辞退が続く本部の存在に気づけない
  → これが「渇望型」組織（Scan 4 型）の定量的シグナル

盲点 3：採用費用の上期/下期の季節性を把握していない
  → 1〜2 月に採用決定が落ちる構造（年末年始効果）を計画に織り込まない
```

**活用場面**：Scan 提案時の「現状可視化が足りていない」の証拠として。

### 組込候補 2（推奨度 ★★★）：「HRBP 型提案の 4 哲学」フレーム

**提案内容**：石井が Coaching / Shift C で「どんな姿勢で入るか」を説明する際のフレーム。

```
GrowthFix の伴走設計（= HRBP 設計の石井版）：
1. 採用・組織課題を「ヘッドカウント補充」でなく「事業課題の解決」として再定義する
2. 答えを持って来ない。「たたき台 + 他社事例 + 選択肢」を提示して経営者が決める
3. データで動かす（感覚ではなく、可視化した指標で議論する）
4. 面接 = 採用ブランド構築の場として設計する（見極め専念型からの脱却）
```

### 組込候補 3（推奨度 ★★）：「面接官 = ファン獲得者」の採用観

**提案内容**：Shift R の採用プロセス設計で「なぜ面接官研修が必要か」の文脈として使用。

```
採用の失敗パターンの真因：
「面接官が見極め専念で、候補者体験を設計していない」

面接の目的（正しい順序）：
1. 情報提供の場（候補者に会社を正確に理解してもらう）
2. ファン獲得の場（不採用でも会社のファンになって帰る）
3. 見極めの場（評価は 3 番目）

→ 採用数が少ない SMB こそ、1 回の面接で候補者のファンを作れるかどうかが採用ブランドを決める
```

### 組込候補 4（推奨度 ★★）：「採用ライフサイクル一気通貫管理」の図

**提案内容**：HRBP 社内営業資料 Slide 3 の「認知 → アルムナイ」一覧図をシンプル化。

Orbit の提案時に「GrowthFix は採用前から離職後まで一気通貫で組織の引力を高める」の視覚説明として使用できる。認知 → 入社 → 活躍 → ロイヤリティ → アルムナイの 5 ステージを組織引力の視点で再解釈する。

### 組込候補 5（推奨度 ★）：「エージェント依存構造の可視化」

**提案内容**：応募の 7 割超がエージェント経由という構造を Shift R の課題提起として使用。

```
SMB の採用現状：
「選択肢を持てず、エージェントに振り回されている」

→ 採用ペルソナの明文化 + ダイレクトソーシング + リファラルの整備が
   エージェント依存からの脱却を実現する
→ これが Shift R が解く課題の定量的証拠
```
