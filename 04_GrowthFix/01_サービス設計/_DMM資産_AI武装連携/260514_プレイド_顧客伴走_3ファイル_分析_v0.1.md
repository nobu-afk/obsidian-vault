**構造的特徴（Gravity Orbit / Shift C 抽象化）：**

1. **検討ステップ可視化**：WS ① → インタビュー ② → 仮説修正 ③ の 3 ステップを明示し、「どんな検討プロセスを踏んだか」を報告書冒頭で示す。顧客が「このプロジェクトは科学的に進めた」と感じられる設計。
2. **「これまで」vs「これから」対比**：守り（CSアンケート年 1 回）→ 攻め（ワーカー直接接点・データドリブン PDCA）の二項対比で変化の価値を明確化。
3. **KGI 設計の明示**：「テナント企業エンゲージメントとサービス満足度の相関」を KGI として設定し、プロセス指標（利用率）→ KPI → KGI の 3 層で成果を定義。
4. **6 シーン利用体験**：顧客ペルソナ 2 名（30 代・40 代）を設定し、SCENE 01〜06 で利用体験をストーリー化。機能説明ではなく「生活の変化」で価値を伝える設計。
5. **外部インタビュー結果（11 社）で受容性エビデンス化**：競合企業の担当者の声を引用し、課題の普遍性（出社率・エンゲージメント・メンタルヘルス・コミュニケーション）を 4 軸で整理。
6. **コスト正当化フレーム**：「デジタル投資はハード投資に比べ他物件展開コストが低く、物件数増加でランニングコスト比率が逓減する」という展開ロジックで、継続・拡張の経済合理性を示す。

### 1-2. ペルソナ × 課題 × サービスの結合構造

```
ペルソナ（2 名）
  ↓
```

この構造は「顧客の状態 → 課題の言語化 → 概念設計 → 機能設計 → 運営設計」という **5 段下降設計** になっており、最終報告書の説得力の軸となっている。

### 1-3. ビル運営 PDCA の設計（Orbit の月次設計への参照）

不動産 C 社向け報告では「ビル運営 PDCA」として以下の 5 ループが設計されている：

- **混雑緩和**：アプリ混雑情報表示 → 人流最適化
- **清掃・設備維持**：清掃リクエスト / 設備不具合 110 番 → 混雑情報の清掃周期反映
- **問い合わせ**：アプリ上の窓口表示 → ビル運営ルール周知
- **ビル内施策**：イベント集客率・来店頻度可視化 → 施策の検証 → 次回企画

**注目点**：データ蓄積 → 実施 → 検証 → 企画 の 4 段ループは「Orbit 月次伴走の設計原理」として直接転用可能。Orbit では「軸 15 個クリアゲート」が進捗指標、月次レポートが PDCA の体裁になる。

### 1-4. 「監視」ではなく「自己管理」の設計（AI 武装への重要示唆）

> 「勤務に関する時間や場所データは可視化しない。サーベイは自己診断ツールとして位置付け、企業へデータ提供を行わないことを明示する。」（スライド 64）

この「監視感ゼロの設計」は、Gravity Orbit が AI 一気通貫月次レポートを展開する際の重要原則として転用できる。エンゲージメントデータを「管理ツール」ではなく「経営者と組織の自己認識ツール」として設計する。

### 2-5. 支援事例に見る「大手企業が外部伴走者に求めるもの」

以下 3 事例から抽出できる共通構造：

**大手インフラ系（新規事業・技術者オンリーのチーム）：**
- 課題：進め方が手探り・スピードが出ない・迷いながら進む
- 支援：タスク設計のたたき台を外部が作り、主体はクライアント。「半歩リード」スタイル。
- 成果：仮説検証スピードが圧倒的に向上・テストリリースを計画より半年前倒し

**医療テック系（ステークホルダー多・プロジェクトマネジメント難易度高）：**
- 課題：一歩進んでは二歩下がる状態・社内リソースに限界
- 支援：経営陣も巻き込み半常駐スタイル・ロードマップ具体化・日々の課題優先度付け
- 成果：経営層との認識ズレ調整・小さな成功体験の積み上げ・組織雰囲気の変化

**鉄道系（OMO 構想・有益性・事業性が曖昧）：**
- 課題：デジタル活用の事業性が曖昧・社内リソースだけでは非効率
- 支援：顧客起点でビジネスモデルと具体的サービス企画立案を支援
- 成果：短期施策から中長期成長モデルまで顧客起点で絵を描き、アクションプランを作成

**共通構造（3 社横断）：**
1. クライアント内部で「主体はクライアント」を維持しながら外部が「半歩リード」
2. 「現場に入る・自ら手本を示す」= ハンズオン、ではなく「クライアントがやれるようになること」を最終ゴールとして設計
3. 経営層と現場の「ズレの調整」を外部伴走者が担う

## 3. 顧客フィードバック SaaS E 社応募図書（73 スライド）構造抽出

### 3-1. 応募図書（行政向け提案書）のフレーム

顧客フィードバック SaaS E 社（エンゲージメント SaaS）が行政機関（兵庫県）のエンゲージメント向上委託業務に応募した提案書。B2G 案件の提案書フレームとして抽出する。

| セクション | 内容 |
|---|---|
| 会社概要・受賞歴 | 信頼性の担保（HRアワード / HRテクノロジー大賞 / ISMS 認証） |
| サービス体系 | クラウドサービス / プロフェッショナルサービス / レポートサービスの 3 層 |
| 特徴 1：設問設計 | 目的・状況・スタンスに合わせたカスタマイズ設問 |
| 特徴 2：分析 | JM（重要度分析）特許技術・影響度分析・属性別分析 |
| 特徴 3：改善アクション支援 | 経営層 → マネージャー → 現場の 3 層同時支援 |
| 活用事例（3 社） | 金融・施設・飲食の業種多様性で汎用性を示す |
| 料金体系 | クラウド 200 万〜 / PS 530 万〜 / レポート 170 万〜 |

### 3-2. eNPS 指標の設計と活用パターン

**eNPS（employee Net Promoter Score）：**
- 設問：「現在の職場を親しい友人や知人にどの程度すすめますか？」
- 算出：推奨者（9-10 点）の割合 - 批判者（0-6 点）の割合
- 意義：従来の ES（従業員満足度）と異なり「職場への貢献意欲」を計測

**活用パターン 3 種：**
1. **採用・退職への示唆**：eNPS と離職率の相関検証
2. **収益指標との連動**：eNPS と店舗 NPS / 売上成長率の相関
3. **人的資本開示**：統合報告書での公開（大企業の ESG トレンド）

**Gravity への翻訳**：Gravity Scan の「引力タイプ 4 型」と eNPS を組み合わせることで、「組織の引力状態 × 従業員エンゲージメント」の複合診断として Orbit の月次指標設計に活用できる。

### 3-3. プロフェッショナルサービス（PS）の 3 フェーズ構造

```
Phase 1：課題発見 / KPI 設定（約 3〜4 ヶ月）
  - EX デザイン（ヒアリング・WS で従業員体験の全体像整理）
  - 調査設計（KGI・KPI・仮説整理）
  - アンケート実施（eNPS + EX 設問）
  - 分析 & 報告会（詳細分析レポート + コンサルタントによる施策ディスカッション）
  - 成果物：調査票 + 詳細分析レポート

Phase 2：日々の改善アクション（約 3〜4 ヶ月）
  - Phase 1 課題の真因特定・優先改善事項の明確化
  - 調査実施（2 回目）・結果分析
  - アクションプラン策定・実行
  - 組織帳票（マネージャー向けレポート）運用

Phase 3（自走フェーズ）：モニタリング
  - Phase 1・2 終了後の改善施策担当者アサイン
  - 改善アクション実行体制 & 継続モニタリング体制整備
  - クライアントが「自走できる状態」を目標とする
```

**所要期間感**：約 1 ヶ月（EX デザイン）+ 約 2 週間（調査実施）+ 約 1 ヶ月（分析・報告）= 単一調査サイクルが約 2.5〜3 ヶ月

### 3-4. 競合状況での提案書設計（B2G 入札の訴求構造）

2. **第三者認証**：HRアワード最優秀賞・HRテクノロジー大賞・ISMS 認証・Pマーク

**Gravity Scan への参照**：Gravity Scan の診断 UI では「4 型診断結果」と「組織の声の自由記述分析」を組み合わせることで、Scan → Shift の転換率向上が期待できる。

## 4. 3 ファイル横断パターン

### 4-1. プレイド型「大手伴走プロセス」の 6 構造的特徴

**特徴 1：フェーズゲート設計（段階承認型の進行管理）**

3 ファイル共通で「Phase 1 → Phase 2 → Phase 3」という段階構造を採用している。各 Phase の「ゴール状態の言語定義」が共通して存在し、「この状態に達したら次 Phase へ」というゲート承認型の設計。単純な期間区切りではなく「何が達成されれば次へ進むか」が事前合意される。

Gravity Orbit が採用している「軸 15 個クリアゲート（5 年継続装置）」は、このフェーズゲート設計と構造的に同じ発想。

**特徴 2：「主体はクライアント」原則の明示**

電力 D 社向け提案書・不動産 C 社報告書共通で、「外部が半歩リードし、主体はクライアント」という役割設計が明文化されている。支援者が「代わりにやる」のではなく「できるようになるまで一緒にやる」設計。

> 「A 社メンバーに主体的にタスクを遂行いただくことを前提に、当社が半歩リードするスタイルで業務をリードする形で、やり方を見て学んでいただく。」

これは GrowthFix の「参謀としての役割」「経営者自身が引力を持つことを最終ゴールとする」という Gravity 思想と完全一致している。

**特徴 3：ワークショップを「プロセスの証拠」として報告書に盛り込む**

不動産 C 社報告（WS 15 回実施の様子）・E 社提案書（改善ワークショップ・勉強会の設計）とも、「ワークショップをやった」という事実を報告書に収録している。これは「外部コンサルが頭で考えたアイデアではなく、クライアントと共に作り上げたプロセス」を可視化するための設計。

Gravity Shift の「週 1 WS 体制」や Orbit の「月 1 石井登壇 Quarterly」も、このプロセスの証拠化という機能を担っている。

**特徴 4：N1 インタビューによる仮説検証ループの内蔵**

電力 D 社向け提案の「仮説立案 → N1 インタビュー → 仮説修正」ループと、不動産 C 社報告の「11 社企業インタビューによる受容性検証」は同じ構造。「仮説は最初から正しくない」という前提のもと、顧客インタビューを設計の中核に置いている。

**特徴 5：「データで語る＋クライアントの声で語る」の二重構造**

E 社提案書の事例掲載（金融・施設・飲食の 3 社）と電力 D 社提案書のクライアント声（JTB・西武 HD・ブラザー工業）は、「定量データ + 定性インサイト」で成果を伝える設計。数値だけでは「本当に効いたか」が伝わらず、声だけでは再現性が見えないため、二重構造が採用されている。

**特徴 6：拡張ロジックの内蔵（継続・横展開の経済合理性）**

不動産 C 社報告の「デジタル投資はハード投資と異なり展開コストが逓減する」という論理展開は、単一案件での完結ではなく「横展開が前提の設計」を最初から組み込んでいる。E 社の「クラウド → PS → レポート」の 3 層料金体系も、導入後の上位プランへの移行を設計している。

Orbit の「20〜30 社並列対応スケール装置化」は、この拡張ロジックの内蔵と同じ発想。

## 6. 仮説 E v0.2 営業資料への組込候補（4 件）

**E-1：「半歩リードで主体はクライアント」という伴走原則の明文化**
