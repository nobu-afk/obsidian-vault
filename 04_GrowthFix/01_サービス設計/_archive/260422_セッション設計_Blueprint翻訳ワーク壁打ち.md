## 前提：Blueprint v5.3 のコンセプト確定事項

```
CODE     = 個人引力の解剖（型診断）
Blueprint = 個人引力 → 組織への翻訳ワーク（設計図）
Scan     = 設計図を CEO+幹部 で立体化
Shift    = 設計図を組織に実装
```

**Blueprint の本質：** 経営者個人を**深掘り**するのではない。経営者個人の引力を**組織の構造（制度・会議・評価軸・役割分担）に翻訳・埋め込む**設計図を描く。

**取り扱う論点：**
1. 翻訳ワークの最適設計
2. CODE結果からの引き継ぎ
3. 出力レポート構造
4. 動線・サービス連携

**追加検討論点（Q5で）：** 260421 の5問（怒り/動機/宣言/傷/未来）はBlueprintから完全排除するか、CODEに統合するか、Blueprintで補助的に残すか

## Block A：翻訳ワークの核心（Q1-Q5）

### Q1. Blueprint の核心機能：「個人→組織への翻訳」とは何を翻訳するか

**問い：** Blueprint で「翻訳」する対象は何か？

**選択肢：**
- **A. 個人引力の3要素全て（Why × 動詞 × 環境）→ 組織の3要素に翻訳**
- **B. 動詞のみ（個人の動詞3連鎖→組織の機能に対応付け）**
- **C. キャラ名と4型判定 → 組織における役割設計**
- **D. CODEの結果全て（12要素）→ 組織への影響として再構築**

**推奨：A（3要素全てを組織に翻訳）**

**理由：** 部分翻訳では組織の引力構造が見えない。3要素セットでの翻訳が最も体系的。

---

### Q2. CODE結果からの引き継ぎ要素（A1案の再検証）

**問い：** Blueprint で CODE結果のどの要素を引き継ぐか？

**選択肢：**
- **A. A1案踏襲（Why + キャラ名 + 4型判定 の3点）**
- **B. A1 + 動詞3連鎖（4点）**
- **C. A1 + 動詞 + 環境条件（5点）**
- **D. CODE結果全て（12要素）を入力に使う**
- **E. 引力の核（源泉=死角の同一動作）も追加**

**推奨：C（5点 = Why + キャラ名 + 4型 + 動詞 + 環境）**

**理由：** Q1で「3要素全てを翻訳」と決めるなら、3要素データ全てが必要。CODE結果からWhy・動詞・環境の3要素＋キャラ名（出発点として）＋4型判定（焦点絞り）の5点が最小完全セット。

---

### Q3. 60分対話の主題構成：5つの翻訳ワークの妥当性

**問い：** 60分の対話を「5つの翻訳ワーク」で構成する案は最適か？

**選択肢：**
- **A. 5つの翻訳ワーク（役割／ボトルネック／構造／役割分担／3ヶ月後）で進行**
- **B. 3つに圧縮（翻訳マップ／ボトルネック／設計図）**
- **C. 7つに展開（個別の組織機能ごと）**
- **D. 自由対話（コーチが状況に応じて誘導）**

**推奨：B（3つに圧縮）**

**理由：** 5つは多い。60分で深掘りには3つが妥当。「翻訳マップ・ボトルネック・設計図」の3層が、レポート3層構造とも整合する。

---

### Q4. 「組織の引力」の操作可能な定義

**問い：** Blueprint が扱う「組織の引力」を、どの粒度で定義するか？

**選択肢：**
- **A. 制度・会議・評価軸・役割分担の4軸**
- **B. プラス文化・採用・評価サイクル等を加えた7-8軸**
- **C. 抽象的な「組織の場」として定義（具体軸は対話で発見）**
- **D. 4軸を基本としつつ、Shiftで詳細展開（Blueprintは骨組み提示）**

**推奨：D（4軸を基本＋Shiftで詳細）**

**理由：** Blueprintは設計図、Shiftは実装。Blueprintで詳細化しすぎると Shift の領域に踏み込む。骨組みに留めて、Shiftへの動線を残す。

---

### Q5. ★追加論点：5問（怒り/動機/宣言/傷/未来）の扱い

**問い：** 260421 の5問（怒り/動機/宣言/傷/未来）はどう扱うか？

**選択肢：**
- **A. Blueprint から完全排除**（個人深掘りはCODE領域）
- **B. CODEに統合**（CODEの対話深掘りツールとして活用）
- **C. Blueprintで補助的に1-2問残す**（特に「傷」「未来」は組織翻訳の参考）
- **D. Coachingに移行**（6ヶ月の継続深掘りツールとして）
- **E. CODEとCoachingに分散**（CODEの対面深掘り＋Coachingでの月次活用）

**推奨：E（CODEとCoachingに分散）**

**理由：**
- CODE対面では既に5問の一部を深掘り技法として活用中（運用ガイド260422に反映済み）
- Blueprint は組織翻訳に純化
- Coaching の6ヶ月は5問を継続的に掘る場として最適
- 5問はCODE→Blueprint→Coaching の各フェーズで違う使い方ができる

★この決定で「CODEに5問深掘りを統合する微修正」が必要になる場合あり

## Block B：出力構造（Q6-Q10）

### Q6. 3層レポートの最適構造

**問い：** Blueprint レポートの3ブロック構造は何が最適か？

**選択肢：**
- **A. 翻訳マップ／ボトルネック／設計図**（v5.3初稿）
- **B. 翻訳マップ／設計図／実装ロードマップ**
- **C. 現状診断／翻訳マップ／設計図**
- **D. 引力の伝播マップ／漏れの構造／引力強化の設計図**

**推奨：A（翻訳マップ／ボトルネック／設計図）**

**理由：** Q3でB案（3つに圧縮）を採用するなら、対話の3パートとレポート3層が一致するA案が最も自然。

---

### Q7. 翻訳マップ（gravity-map-box）の構造

**問い：** Block A の翻訳マップをどう視覚化するか？

**選択肢：**
- **A. 表形式（個人3要素 → 組織3要素の対応表）**
- **B. 矢印図（個人 → 翻訳 → 組織 の流れ）**
- **C. 三角形（個人引力の三角形 と 組織引力の三角形を並列）**
- **D. テキスト中心（box内の段落で記述）**

**推奨：A（表形式）**

**理由：** 「個人の何が、組織の何に翻訳されるか」を一目で見せるなら表が最強。LinkedInやSlackでもスクショされやすい構造。

---

### Q8. ボトルネック診断の粒度

**問い：** Block B のボトルネック診断は何個・どの深さで提示するか？

**選択肢：**
- **A. 3個程度（CEO関与度の過剰領域 上位3）**
- **B. 1個に絞る（最大ボトルネックのみ深掘り）**
- **C. 5-7個並列（網羅的）**
- **D. 2-3個＋優先順位付け**

**推奨：D（2-3個＋優先順位付け）**

**理由：** 1個では薄い、5個以上は読者が選べない。2-3個で「最優先」を明示するのが意思決定支援として最適。

---

### Q9. 設計図の具体度（制度・会議・評価軸・役割のどれを優先）

**問い：** Block C の設計図で、どの軸を優先的に提示するか？

**選択肢：**
- **A. 4軸全て同等に提示**
- **B. 役割分担を最優先（CEOが何を任せるか）**
- **C. 会議体を最優先（どの会議を変えるか）**
- **D. 評価軸を最優先（KPI・採用基準）**
- **E. 4型判定に応じて重点を変える**

**推奨：B（役割分担を最優先）→ E（4型連動も検討）**

**理由：** 経営者が最初に動かせる変数は「役割分担」。会議や評価軸は変更コストが高い。役割分担から始めて、必要に応じて他軸へ波及させる方が現実的。

---

### Q10. 3ヶ月後シナリオ（future-box）の描き方

**問い：** 3ヶ月後の組織像をどう描くか？

**選択肢：**
- **A. 経営者の1日の変化（朝起きてから寝るまで）**
- **B. 組織の数値変化（離職率・新規採用・売上等）**
- **C. 組織内の風景（会議の質・雑談・自発的提案）**
- **D. CEOの役割の変化（何を手放し、何に集中しているか）**
- **E. A+C+D の組み合わせ**

**推奨：E（A+C+D 組み合わせ）**

**理由：** 数値（B）は実装後の話。経営者が腹落ちするのは「自分の1日の変化」「組織の風景」「役割の変化」のセット。

## Block C：動線・サービス連携（Q11-Q14）

### Q11. 4型別の動線（CODE 4型 × Blueprint 推奨）

**問い：** CODE の4型判定 × Blueprint の推奨サービスをどう連動させるか？

**選択肢：**
- **A. 4型全てがBlueprint推奨**（CODE→Blueprintを必須化）
- **B. 整合型と環境ズレ型のみBlueprint推奨**（Whyズレ・動詞ズレはCoaching先）
- **C. 経営者の自由意志（Blueprint LP で4型解説、判断は本人）**
- **D. 4型 × 組織規模 × 組織課題 の3次元で動線決定**

**推奨：B（整合型と環境ズレ型のみBlueprint推奨）**

**理由：** 個人引力が立っていない（Whyズレ・動詞ズレ）状態で組織翻訳を始めても効果薄い。先にCoachingで個人を整えるべき。

---

### Q12. Blueprint → Scan/Shift の移行論理

**問い：** Blueprint 後の Scan/Shift への動線をどう設計するか？

**選択肢：**
- **A. Scan必須（CEOが組織変更を決断する前に幹部視点を入れる）**
- **B. Scan推奨（CEO単独でShiftに進む選択肢も残す）**
- **C. 組織規模で分岐（10名以下→Shift、それ以上→Scan）**
- **D. CEOの確信度で分岐（迷い大→Scan、確信あり→Shift）**

**推奨：C（組織規模で分岐）**

**理由：** 小規模組織ではScanの幹部多視点があまり機能しない。10名以下はBlueprint→直接Shift、それ以上はScanで多視点取って合意形成してからShift、が合理的。

### Q13. コーチセッションの流れ（60分構成）

**問い：** Blueprint 60分セッションをどう構成するか？

**選択肢：**
- **A. 翻訳マップ20分／ボトルネック20分／設計図15分／読み合わせ5分**
- **B. アイスブレーク5分／3パート対話45分／読み合わせ10分**
- **C. CODE結果確認5分／3パート対話40分／設計図共同制作10分／クロージング5分**
- **D. 自由対話50分／レポート生成読み合わせ10分**

**推奨：C（CODE結果確認5分／3パート対話40分／設計図共同制作10分／クロージング5分）**

**理由：** CODE結果確認で5分使うのが効率的（コーチも経営者も前提揃う）。設計図の共同制作10分が重要（経営者を巻き込んだ意思決定）。

---

### Q14. モニター・改善サイクル

**問い：** Blueprint のモニター・改善サイクルをどう設計するか？

**選択肢：**
- **A. CODEと同じ（3件ごと→月次→四半期の段階式）**
- **B. CODEより高頻度（個別案件ごとレビュー）**
- **C. CODEより低頻度（月次から開始）**
- **D. CODE と Blueprint をペアでレビュー（同じ経営者で両方の効果測定）**

**推奨：D（ペアレビュー）**

**理由：** 同じ経営者がCODE→Blueprintを受けたとき、各レポートが連続して機能しているかを測る方が、サービス全体の品質改善に直結する。

## 統合された Blueprint v5.3 設計

### コンセプト
> CODE で言語化された個人引力の6要素を起点に、60分対話の3パート（翻訳マップ／ボトルネック／設計図）で**組織の4軸（制度・会議・評価軸・役割分担）に翻訳**。役割分担を最優先に、Shiftで詳細実装する設計図を描く。

### サービス連携
- **CODE → Blueprint**：整合型・環境ズレ型のみ推奨（Whyズレ/動詞ズレはCoaching先）
- **Blueprint → Scan**：11名以上または確信度低
- **Blueprint → Shift**：10名以下＋確信度高
- **Blueprint → Coaching**：確信度低（小規模で迷い大）

### 5問の再配置
- ❌ Blueprint：5問廃止（個人深掘り→CODEとCoaching領域）
- ✅ CODE 対面：5問の一部を深掘り技法として活用（運用ガイド済）
- ✅ Coaching 6ヶ月：5問を月次テーマに（月1×6回で1問ずつ深掘り）

### 60分セッション内訳（v5.3 確定）

| 時間 | パート | 内容 |
|---|---|---|
| 0-3分 | CODE結果確認 | キャラ名・4型・引力の核を共有 |
| 3-18分 | 翻訳マップ（15分） | 個人3要素 → 組織3要素の対応付け |
| 18-38分 | ボトルネック診断（20分・最重要） | CEO関与過剰領域・蓋外し・矛盾突き |
| 38-53分 | 設計図（15分） | 役割分担主軸＋他3軸補完 |
| 53-60分 | 読み合わせ＋クロージング | レポート確認＋次の一手合意 |

### CODE微修正必要事項（Q11連動）
