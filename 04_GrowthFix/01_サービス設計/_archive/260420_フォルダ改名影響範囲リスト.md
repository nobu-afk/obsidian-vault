## ⚠️ 重大な発見：`診断_本番/` は CODE 側にも存在

**課題：**
`診断_本番/` というフォルダ名は `GravityScan/` と `GravityCode/` の両方に存在する：
- `GravityScan/診断_本番/` → Blueprint（旧Scan・個人CEO）診断
- `GravityCode/診断_本番/` → CODE（個人の引力タイプ）診断

**リスク：**
- GravityScan側だけ `Blueprint_本番/` に改名すると**片方だけ命名ルール変更**で非対称になる
- 逆にGravityCode側も `CODE_本番/` に改名するとCODEの別バリエーション（`診断_executive_本番/`）との整合を要再検討

## 💡 改名方針の提案（棚卸しB A案実装の具体）

### Pattern 1：Scan側のみ改名（最小変更）

| 改名前 | 改名後 |
|---|---|
| `GravityScan/診断_本番/` | `GravityScan/Blueprint_本番/` |
| `GravityScan/DEEP_本番/` | `GravityScan/Scan_本番/` |
| `GravityScan/DEEP_LP/` | `GravityScan/Scan_LP/` |
| `GravityCode/診断_本番/` | **改名なし（現状維持）** |

- **メリット：** 最小変更・リスク最小・CODE側は触らない
- **デメリット：** 命名ルール非対称（Scan側は英名・CODE側は「診断」）

### Pattern 2：両サービスとも改名（統一命名）

| 改名前 | 改名後 |
|---|---|
| `GravityScan/診断_本番/` | `GravityScan/Blueprint_本番/` |
| `GravityScan/DEEP_本番/` | `GravityScan/Scan_本番/` |
| `GravityScan/DEEP_LP/` | `GravityScan/Scan_LP/` |
| `GravityCode/診断_本番/` | `GravityCode/CODE_本番/` |
| `GravityCode/診断_executive_本番/` | `GravityCode/CODE-Executive_本番/` |

- **メリット：** 命名完全統一・見通し良好
- **デメリット：** 変更範囲大・テスト工数2倍

### 💡 推奨：Pattern 1（最小変更）

理由：
1. **棚卸しBの目的は Scan改名**（Blueprint/Scan命名）。CODE側は本来の対象外
2. 最小変更でリスク最小
3. 将来 CODE側も整理したくなった時に Pattern 2 へ段階移行可能
4. CODE側は LP も本番稼働中なので下手に触らない方が安全

## ⚠️ 見落としリスク・注意点

### 1. HTMLファイル内のパス参照
- 各HTMLファイルは `diagnose/` 等の**サーバー側URL**を参照している想定
- ローカルフォルダ名 `診断_本番/` を直接参照するHTMLはないはず（要確認）

### 2. PHPファイル内のパス参照
- `generate.php` が `reports/` などの相対パスを使用している場合、フォルダ改名の影響あり
- デプロイ後のサーバー側ディレクトリ名は変わらないので、サーバー側では影響なし

### 3. `診断_executive_本番/` の扱い
- CODE Executive用・現状維持推奨
- 本作業では対象外

### 4. `ヒアリング_本番/` は既に `_archive/` に退避済
- 重複作業なし
