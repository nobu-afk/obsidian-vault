### 自己理解 - 独自軸 TOP 10

**1. 編集者型思考の識別** （山田20, 長谷部4, 進藤2・26回・独立度3）
   - 定義: 独自のアイデアではなく「他者のアイデアや構造を組み替えて最適化する」ことに熱が出る人を「編集者型」と識別し、そのポジションで勝負させる
   - Gravity転用: CODE「強み」の診断で「生成型」vs「編集型」という軸として使える。GrowthFixの参謀型（既存ビジネスを組み替えて最大化する）というポジションとの共鳴が強い

**2. 言い訳にならない選択原則** （長谷部6, 山田4・10回・独立度3）
   - 定義: 「子供のため」「家族のため」を理由に自分の理想を諦めた場合、大人になった子供から「あの時こうしてほしかった」と言われることがある。最善の選択をし続ける姿勢が長期的に家族全体にとっても良い。
   - Gravity転用: Gravity の経営者コーチング（Coaching）で「家族・社員のためという名の自己犠牲決断」を問い直す文脈で使えるフレーム

**3. 加担構造の自己診断** （長谷部2, 山田1・3回・独立度3）
   - 定義: パートナーや周囲のネガティブな状態（不安・停滞）に自分が「加担している」可能性を自己評価させる問い
   - Gravity転用: Gravity Coaching での経営者の組織問題診断で「社員のエンゲージメント低下に経営者自身がどう加担しているか」を掘り下げる問いとして転用可。長谷部流の心理深掘りの核心的問い。

**4. 本音全開の「4 回目効果」** （山田3・3回・独立度3）
   - 定義: 関係性において「全部言う」体験の積み重ねが相手の変容を促す。最初の 1〜2 回ではなく、繰り返すことで相手の行動変容が定着する段階がある。
   - Gravity転用: GrowthFix の Coaching クライアントへの「1 on 1 設計」に転用。「何回目の本音か」を問うことで変容のフェーズを診断する軸として。

**5. エネルギー状態と季節性の切り分け** （山田2, 長谷部1・3回・独立度2）
   - 定義: 「やる気がない・動けない」を精神論で片付けず、季節・環境・体調という物理的制約として切り分け、変えられる範囲と変えられない範囲を整理する
   - Gravity転用: Orbit月次サーベイで季節性エネルギー変動を観測する設計に転用。Cultivateのドラムビート設計で季節ごとの施策強度調整に使える

**6. やりたいのにやれてないこと・1個体験実験** （長谷部2, 山田1・3回・独立度2）
   - 定義: 「やりたいけどやれてない」ことが続く人に、交渉でいいから1個だけ実際にやってもらう。「やれるんだ」という体感を積み重ねる。
   - Gravity転用: Cultivate（組織開発）での「エフィカシー（遂行能力感）」の形成プロセスとして「小さな成功体験の積み重ね」を設計する際の根拠フレームとして使える。

**7. 先々楽しくなる未来像の欠如診断** （山田3・3回・独立度2）
   - 定義: 「安定はできてきたが先々楽しくなるイメージが湧かない」という状態は、現在の仕事設計が「維持型」になっており「成長・探索・没頭」の要素が欠けているサインである。
   - Gravity転用: GrowthFix の CODE セッション中盤でよく出る「安定したが退屈」という状態の診断軸として。理想の未来像が描けないクライアントへの問い。

**8. 最低思考（51点定義法）** （長谷部1, 山田1・2回・独立度3）
   - 定義: 100点主義の強い人のために「合格ライン（51点）とは何か」を先に言語化し、全力投球の発動をコントロールする思考ツール。
   - Gravity転用: Cultivate や Coaching で「頑張りすぎて続かない経営者・マネージャー」に対し、「組織運営の51点ラインは何か？」を定義させる問いとして使える。堀田フレームの「やりきる装置」の逆算版として機能。

**9. アウトプットまで行かないモヤモヤの構造** （山田1, 長谷部1・2回・独立度3）
   - 定義: アイデアが頭に浮かぶが形にする前に止まる（アウトプット遮断型）人には、スピードではなく「壁打ちのラリー数を増やす」ことで出口を作る
   - Gravity転用: Coaching対象者の典型プロファイル（思考が速く行動が詰まる経営者）のコーチングアプローチとして使える

**10. 緩みとだらつきの違い** （山田1, 長谷部1・2回・独立度3）
   - 定義: 「緩む」は意図的に力を抜く状態。「だらつく」は無目的に流れる状態。緩みはエネルギーを回復させるが、だらつきはエネルギーを消耗させる。自分がどちらの状態にいるかを区別することが自己管理の核心。
   - Gravity転用: Gravity のコーチングで経営者の「疲れているのに休んでも回復しない」というケースの分析に使える。「休み方」の設計（緩み vs だらつき）は経営者コーチングの重要テーマ

### モニター - 独自軸 TOP 9

**1. 二軸サービス分離検証設計** （山田2・2回・独立度2）
   - 定義: 「A×B」という複合サービスを最初から融合して打ち出すより、AとBを別々に先に検証し、各々の手ごたえが出てから融合するという段階的設計
   - Gravity転用: GrowthFix の Recruit × Cultivate × Orbit の三本柱設計で「最初から全部」ではなく「まずどれで成功事例を作るか」の優先設計に参照

**2. 問題解決先行クローズ** （山田2・2回・独立度0）
   - 定義: 継続提案の入口を「セッション」ではなく「問題解決できるかどうか」に変える営業構造
   - Gravity転用: CODE 診断セッション後の Coaching / Gravity プログラム提案でそのまま使える。「CODE の続きをやりませんか」ではなく「あなたが今感じている〇〇の問題、継続することで解消できます」という構造に変換するフレーム。Gravity で言えば「引力設計プログラムを買ってほしい」ではな

**3. 課題データ = 顧客リサーチ装置** （山田1・1回・独立度3）
   - 定義: チャレンジ参加者の課題提出内容を読むことで、インタビューなしに顧客の本音ニーズを把握しオファーをカスタマイズできる。
   - Gravity転用: CODE 4 型診断の設問設計・Web 診断回答データの活用方法として参照可。「診断 = リサーチ装置」という概念と接続。

**4. 自信欠如→モニター帰還サイクル** （山田1・1回・独立度3）
   - 定義: 提案・行動に踏み出せない状態は「そのサービスが自分に売れていない状態」と同義。毎回モニターに帰って実施し直すことで自信が回復する
   - Gravity転用: Gravity サービスで新しいプログラム設計後、社内で「自分がクライアントなら買うか」を問うモニター型検証サイクルに応用できる

**5. アーリーアダプター感度補正** （山田1・1回・独立度3）
   - 定義: モニター参加者の感度（意識の高さ）が平均より高い場合、その反応を一般市場の反応として採用しない補正を行う
   - Gravity転用: GrowthFix の CODE 診断や無料Web診断のモニター評価で、参加者が「自己理解に強い関心がある人」に偏った場合のフィードバック解釈に応用。「感度補正なしで商品設計すると一般市場でコケる」パターンの予防フレーム。

**6. 見込み客段階では欲しいものにフォーカスの原則** （山田1・1回・独立度3）
   - 定義: 信頼関係が未形成の見込み客に対しては、相手が「今段階で手に入れたい」と思っているものにフォーカスする。「本当の願い」や「フェイクの願い」を扱うのはサービス開始後。
   - Gravity転用: Gravity の無料Web診断→個別相談の同線設計で「何を伝える順番か」を設計する際の基本原則として活用。「先出し診断」のコンテンツ設計にも転用。

**7. アンケート設問の「受ける前後ズレ確認」型設計** （山田1・1回・独立度3）
   - 定義: アンケートで顧客満足度を測る際に「受ける前のイメージ」と「実際受けてみた印象の差（ズレ）」を明示的に聞くことで、コミュニケーションのギャップを特定できる
   - Gravity転用: GrowthFix の Shift R/C 卒業後アンケート・採用モニター後アンケートの設問設計に転用。「期待していたこととの差」を問う設問の追加

**8. リピート意欲3択で次回動機を可視化** （山田1・1回・独立度3）
   - 定義: アンケート最後に「次の機会があれば受けたい / 違うテーマなら受けたい / 今回で満足」の3択を置くことで、リピートの動機の質（テーマ依存 vs 満足度依存）が分かる

   - Gravity転用: Gravity がいずれ「引力コーチ認定」や「アンバサダー育成」を考える際の設計根拠として使える。「文化ごと広げる」という設計は天プロモデル自体がその実例

**8. 撤退基準の根拠確認** （山田1・1回・独立度3）
   - 定義: 「X件未達で撤退」という基準が設定されていても、なぜその数字かの根拠がない場合は本当に撤退する気がない可能性がある。根拠を確認することでコミットメントの本気度が分かる
   - Gravity転用: GrowthFix サービスのKPI設定（制約件数・継続率など）に「なぜその数字か」の根拠を持つことの重要性を確認する際に使える

**9. 「立ち上がりフェーズ→チームフェーズ」の移行ポイント** （山田1・1回・独立度3）
   - 定義: 事業立ち上げ期は各自が自由に動く個人プレイが効果的だが、スケール期に入るとチームとしての役割分担・KPI共有が必要になる。この移行ポイントを見極めること
   - Gravity転用: Recruit・Cultivateのサービス設計で「フェーズ診断（立ち上げ期/組織化期）」に基づくサービス適用判断の根拠として使える

**10. LINEやり取り型サービスの「分散エネルギー消費」問題** （山田1・1回・独立度3）
   - 定義: LINEでの常時対応は「1回の返信時間が短くても、意識がそこに向き続ける」ことでエネルギーを消費する。「時間枠の中だけ意識を向ける」形に変えることが解決策
   - Gravity転用: Orbit の月次サービスでの「チャット相談対応」設計に直接適用。「月X時間枠内に集約」という運用ルールへ
