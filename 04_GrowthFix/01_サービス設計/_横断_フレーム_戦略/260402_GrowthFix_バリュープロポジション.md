## コアコンピタンス

> **「事業の言語」と「組織の言語」の翻訳力**

経営者が「人の問題」だと思っていることを、「事業の構造」に翻訳する。

| 経営者が言う | 普通のコンサルの反応 | 石井の翻訳 |
|---|---|---|
| 「営業力が足りない」 | 営業研修を提案 | 「案件見極め基準が属人的」 |
| 「人が辞める」 | エンゲージメントサーベイを提案 | 「評価基準が不透明で何を頑張ればいいかわからない」 |
| 「任せられない」 | 権限移譲研修を提案 | 「判断基準が社長の頭の中にしかない」 |
| 「AIを入れたのに変わらない」 | AI活用研修を提案 | 「施策が事業の勝ち筋ループに接続されていない」 |

**普通のコンサルは「言われたことに対して施策を提案する」。石井は「言われたことの裏にある構造を翻訳する」。**

### 翻訳力の源泉

1. **5パターン全てに「中の人」として入った経験**
   - 単循環型：ミナジン コンサル＋BPO（事業立ち上げ＋執行役員）
   - ストック型：ミナジン 勤怠SaaS（事業部長）
   - M&A型：MOON-X（1人目人事責任者・30→120人）
   - PF型：DMM（50事業HRBP統括）
   - エキスパート型：GrowthFix（創業者）

2. **50社の臨床知**（同じ構造が繰り返されるパターン認識）

3. **事業側と組織側の両方の言語を持っている**（事業開発＋人事の両方を経験）

### AIには代替できない理由

- AIは経営者の「行間」を読めない
- AIは50社の「あのケースとこのケースは同じ構造だ」というパターンマッチングができない
- AIは「営業力が足りない」を額面通りに受け取る。石井は「それは営業力ではなく…」と翻訳する

宋さんの証言：「毎日Claudeに相談してるけど、こんな綺麗にならない。**組織全体の構造が見えたのは初めて**」

## 統一バリュープロポジション

> **社長の限界を翻訳して、仕組みに変える会社**

- 「翻訳」がコアコンピタンス（経営者の言葉→事業の構造に変換）
- 「仕組みに変える」がデリバリー（Gravity 6ヶ月）
- 「社長の限界」がターゲットのペイン

---

## タグライン

> **社長の限界を、事業の限界にしない**

## 各サービスのバリュープロポジション

### 翻訳の深さで並べると

```
翻訳の深さ：浅い ──────────────────────── 深い
            CODE → Scan → Scan DEEP → Gravity → Orbit
            知る    特定    丸裸       変える    維持
```

| サービス | 何を翻訳するか | 経営者が得るもの | 一言VP |
|---|---|---|---|
| **CODE** | 経営者自身の「表の強み」→「隠れた才能」 | 自分の使い方の取扱説明書 | **自分の限界がどこにあるかがわかる** |
| **Scan** | 「人の問題」→「事業の構造」 | 勝ち筋ループのどこが詰まっているかのレポート | **事業の天井がどこにあるかがわかる** |
| **Scan DEEP** | 「経営者の認識」→「幹部との認識ギャップ」 | 経営者が見えていない盲点のレポート | **自分が見えていないものが見える** |
| **Gravity** | 「社長がやっていること」→「仕組み」 | 社長がいなくても伸び続ける状態 | **事業の天井を外す** |
| **Coaching** | 「こうあるべき」→「本当はどうしたいか」 | 経営スタイルの判断軸 | **自分自身の天井を外す** |
| **Orbit** | 「仕組み」→「進化する仕組み」 | 社長がいなくても仕組みが更新される | **外した天井を維持する** |

### 「限界の種類」で分類すると

| 限界の種類 | サービス |
|---|---|
| **自分自身の限界**（自分の才能・使い方） | CODE → Coaching |
| **事業の構造の限界**（勝ち筋ループの社長依存） | Scan → Gravity |
| **認識の限界**（見えていないもの） | Scan DEEP |
| **維持の限界**（仕組みが劣化する） | Orbit |

## 経営者のジャーニー

```
「自分はできてるはず」
  ↓ CODE
「え、隠れた才能があるの？自分の使い方、間違ってた？」
  ↓ Scan
「事業の天井、ここだったのか。社長依存の構造が見えた」
  ↓ Scan DEEP（必要な場合）
「幹部とのズレ、こんなにあったのか」
  ↓ Gravity
「社長がやっていたことが仕組みに変わった。自分は次の仕掛けに集中できる」
  ↓ Coaching（並行）
「自分自身のhave toも手放せた。経営スタイルが変わった」
  ↓ Orbit
「仕組みが自走してる。事業が伸び続けてる」
```

## 3フレームとの接続

GrowthFixの翻訳力は3つのフレームで体系化されている：

| フレーム | 翻訳の機能 | 使うタイミング |
|---|---|---|
| **勝ち筋ループ** | 事業の言語で構造を描く → 社長依存を特定 | セミナー（チラ見せ）→ Gravity |
| **4象限** | 「どこにいるか」を翻訳する → 自己認識のギャップを埋める | セミナー（メイン）→ Scan |
| **建物4フロア** | 組織の言語で構造を診断する → 何が欠けているかを可視化 | Scan（メイン）→ Gravity |

```
セミナー（集客）: 4象限で「自分はどこにいるか」を翻訳
  → Scan（診断）: 建物4フロアで「何が欠けているか」を翻訳
    → Gravity（実行）: 勝ち筋ループで「何を仕組みに変えるか」を翻訳
      → Orbit（維持）: 仕組みの進化を継続
```

---

## Gravityの一言定義

> **「社長がやっていること」を、仕組みに変える6ヶ月**

具体的には：
1. 事業の勝ち筋ループを描く
2. ループの中の「社長依存」を特定する
3. 経営者の言葉を「必要な力」に翻訳する
4. 力 × 担い手のギャップを可視化する
5. 人 × AI × 仕組みで再設計する

## 事業の勝ち筋ループ 5パターン

| # | パターン | 拡大の核 | 石井の実体験 |
|---|---|---|---|
| 1 | **単循環型** | 1件ずつの信頼で回す | ミナジン コンサル＋BPO |
| 2 | **ストック型** | 顧客が積み上がる（SaaS/D2C/サブスク） | ミナジン 勤怠SaaS |
| 3 | **M&A型** | 買収で外部成長 | MOON-X |
| 4 | **PF型** | 面×回遊 | DMM |
| 5 | **エキスパート型** | 専門知識のレバレッジ | GrowthFix |

### パターン別「経営者の言葉→石井の翻訳」

| パターン | 経営者が言う | 石井が翻訳する「力」 | 処方箋 |
|---|---|---|---|
| 単循環 | 「営業力が足りない」 | 案件見極め基準が属人的 | 基準書の言語化＋移転 |
| 単循環 | 「任せられない」 | 判断基準が社長の頭の中にしかない | 権限設計＋判断基準の明文化 |
| 単循環 | 「採っても辞める」 | オンボーディングの仕組みがない | 90日立ち上げプログラム設計 |
| ストック | 「解約が多い」 | 初期価値体験が設計されていない | オンボーディング＋CSプロセス |
| ストック | 「MRRが伸びない」 | アップセルの導線がない | CS→拡張の導線設計 |
| M&A | 「PMIがうまくいかない」 | 異文化を統合するBASEがない | 共通の評価基準＋オンボーディング |
| M&A | 「買収先が機能しない」 | COREの勝ち筋を買収先に翻訳する力がない | 勝ち筋ループの標準化 |
| PF | 「面は取れたが伸びない」 | 事業間の回遊導線が設計されていない | 認知/収益の役割定義＋回遊の仕組み化 |
| エキスパート | 「自分の時間が足りない」 | デリバリーが属人的 | ウニ丼モデル（パートナー委譲） |
| エキスパート | 「スケールしない」 | 知識が仕組み化されていない | コンテンツ資産化＋レバレッジ設計 |
