# GrowthFix フレームワーク体系

> GrowthFixが保有する全フレームワークの定義・構造・使用場所を1ファイルで管理する。
> Skills・プロンプト・セミナー設計からの参照先として機能する。
>
> 最終更新: 2026-04-05

---

## 全体マップ

```
【診断フレーム】経営者の現状を構造化する
  ├ 事業の伸ばし方4象限 ─── セミナーの入口
  ├ 組織の組織の4レイヤー ───── Scanの診断軸
  ├ 勝ち筋ループ ──────── Gravityの設計起点
  ├ 勝ち筋ループ5パターン ─ ヒアリング冒頭の分類
  ├ 経営者の停滞パターン5類型 ─ Scanレポートの類型判定
  ├ 状態の6段階 ──────── Coachingの介入方針
  └ 組織の停滞度診断（12問）── Webフォームの簡易診断

【設計フレーム】課題を解決する手順・ツール
  ├ 翻訳テーブル ──────── Scanレポートの「最初の一手」
  └ 3フレーム統合実装 ──── Gravity 6ヶ月のデリバリー骨格

【事業戦略フレーム】GrowthFix自身の経営
  ├ Gravityマップ ─────── サービス体系の全体像
  └ 点数積み上げモデル ─── ファネル設計

【個人フレーム】
  ├ ソースコード解析v5 ─── CODE/自己分析
  └ バランスホイール8領域 ─ 個人ビジョン設計

【外部理論の活用】
  ├ 堀田5原則 ──────── LP/セミナー/FB投稿の品質基準
  └ 知的スマイルカーブ ─── セミナーの意外性装置
```

## A. 診断フレーム

### A-1. Gravityマップ ── 事業の伸ばし方は4パターンしかない

**GrowthFixの2大看板フレームワークの1つ。** 外部に全部見せる。セミナーで配り、口コミで広がることが目的。

**使う場面：** セミナー（メインコンテンツ）、Scan（現在地の把握）、Gravity（ゴール設定）、Kindle出版

```
              仕組みで回す
                  │
        ❸         │        ❹
   やったが変わらない │   自分は次のステージへ
                  │
  ─────────────────┼─────────────────
                  │
        ❶         │        ❷
   忙しい。全部自分  │   人は増えた。でも…
                  │
              人頼み
```

| 象限 | 典型的な状態 | GrowthFixの打ち手 |
|------|------------|-----------------|
| ❶ | 社長が全部やっている。24時間が天井 | Scan → Gravity（仕組み化） |
| ❷ | 採用したが機能しない。びっくり退職 | Scan → Gravity（制度設計） |
| ❸ | 施策を入れたが変わらない | Scan（構造診断）→ Gravity |
| ❹ | 仕組みが回り始め、社長が次に行ける | Orbit（月次モニタリング） |

**詳細：** [[260401_一発芸スライド_事業の伸ばし方4象限]]

### A-2. 組織の4レイヤー

**使う場面：** Scanレポート（空洞レイヤーの特定・ビジュアル図解付き）、Gravity（6ヶ月の着手順序）、セミナー（チラ見せ）

```
┌─────────────────────────────────────────┐
│ 事業拡大層（Layer 4）                    │ ← 社長が本来いるべき層
│   新規事業・M&A・資金調達                │
├─────────────────────────────────────────┤
│ 人材・制度層（Layer 3）                  │ ← 採用→配置→評価→育成→退職
│   採用基準・評価制度・報酬設計           │
├─────────────────────────────────────────┤
│ 権限・判断層（Layer 2）                  │ ← 判断基準・権限移譲・意思決定構造
│   「社長に確認します」が止まるか         │
├─────────────────────────────────────────┤
│ 経営チーム層（Layer 1）                  │ ← 右腕・経営会議体
│   何でも相談できるNo.2がいるか           │
└─────────────────────────────────────────┘
  ↑ 下から積む。上の層は下が整わないと機能しない
```

**診断の問い：** 「あなたの組織は、どのレイヤーが空洞ですか？」
- 経営チーム層が空洞 → 全ての判断が社長に集中（停滞パターン①）
- 権限・判断層が空洞 → 権限移譲が進まない（停滞パターン①④）
- 人材・制度層が空洞 → 評価・採用が属人的（停滞パターン③④）
- 事業拡大層が空洞 → 社長が下の層の仕事を代行している（停滞パターン②）

**詳細：** [[260402_フレーム02_組織の組織の4レイヤーモデル]]

### A-3. 勝ち筋ループ ── 社長がどこに詰まっているかが見える

**GrowthFixの2大看板フレームワークの2つ目。** Gravityマップと合わせて外部に全部見せる。

**使う場面：** Gravity（事業から逆算した組織設計の起点）、WS（参加者が自社ループを描く）、Scanレポート（事業パターンの特定）、セミナー、Kindle出版

事業が「何が → どう回ると → 売上が伸びるか」を1枚の図にしたもの。各要素に必要な「力」を書き出し、「社長」とマーカーした要素＝事業の天井。

```
[要素A] →（力X）→ [要素B] →（力Y）→ [要素C] →（力Z）→ [要素A]...
   ↑                                                    
  社長 ← ここが天井
```

**使い方：**
1. 自社のループを描く
2. 各要素に必要な「力」を書く
3. 「社長」が担っている力にマーカー → そこが事業の天井
4. 天井を外す打ち手（採用・AI・仕組み化）を配置する

**詳細：** [[260402_フレーム03_事業の勝ち筋ループ]]

### A-4. 勝ち筋ループ5パターン

**使う場面：** Scanヒアリング冒頭（事業パターンの特定）、WS（パターン選択→ループ描画）、セミナー（「どのパターン？」の問いかけ）

| # | パターン | 典型事業 | 社長依存しやすい要素 | 天井の構造 |
|---|---------|--------|------------------|----------|
| 1 | **単循環型** | 受託・小売・飲食 | 全て（特に営業） | 社長の24時間＝回転数上限 |
| 2 | **ストック型** | SaaS・D2C・サブスク | FS（成約）とCS（解約防止） | ファネルのボトルネック |
| 3 | **M&A型** | 多角化・ホールディングス | MA候補発掘＋ファイナンス | BASEが買収スピードに追いつかない |
| 4 | **PF型** | プラットフォーム | 新クラスタ拡張＋事業間導線 | 面は取れたが回遊しない |
| 5 | **エキスパート型** | コンサル・士業・教育 | 発信＋デリバリー | 社長の稼働時間＝売上上限 |

**石井の独自性：** 5パターン＋ハイブリッドの全てを「中の人」として経験済み（ミナジン/MOON-X/DMM/GrowthFix）

**4象限との接続：**

| パターン | ❶に多い | ❹に行ける条件 |
|---------|--------|-------------|
| 単循環型 | ◎ 最多 | 営業の仕組み化 |
| ストック型 | ○ | ファネル各段階の分業 |
| M&A型 | △ | BASEの先行構築 |
| PF型 | △ | 回遊の仕組み化 |
| エキスパート型 | ◎ | ウニ丼モデル |

**詳細：** [[260402_事業の勝ち筋ループ_パターン類型化]]

### A-5. 経営者の停滞パターン5類型

**使う場面：** Scanレポート（類型判定）、セミナー（自分事化）、飲み会（パターン名で刺す）、Note記事（5本シリーズ）、広告コピー

| # | パターン名 | 別名 | 典型的な訴え | 根本原因 |
|---|----------|------|------------|---------|
| ① | 「社長に確認します」が止まらない | 全部オレが症候群 | 権限移譲したはずなのに全部来る | 判断権限の定義がない |
| ② | 正しいはずの経営が組織を止めている | 仮面経営者 | 正しいことをやっているのにモヤモヤ | 外発的動機に支配、本音と乖離 |
| ③ | 何を頑張れば評価されるか誰もわからない | 制度ブラックボックス | 評価制度を作りたいが何から | 30人超で人事制度が追いつかない |
| ④ | あの人が辞めたら事業が止まる | エース依存の綱渡り | あの人が辞めたらうちは終わる | 仕組みがなくエースの暗黙知に依存 |
| ⑤ | 方針を伝えたのに誰も動かない | データ溺死 | サーベイ取ったが使い方がわからない | 施策バラバラ、全体構造の診断がない |

**パターン間の関係：**
- ①と②は併発しやすい
- ③と④は成長フェーズの問題（30人超/50人超）
- 有事に最も危険なのは①と④

**共通の根本原因：** 経路依存性 — 過去に正しかった仕組みが、環境変化後も慣性で回り続けている

**詳細：** [[260325_経営者の停滞パターン]]

### A-6. 状態の6段階

**使う場面：** session-prep（クライアントのステージ判定→介入方針）、Meta広告設計（Stage 2-3がターゲット）、サービス接続（どのステージにどのサービスが刺さるか）

| Stage | 名称 | 状態 | 介入方針 |
|-------|------|------|---------|
| 1 | 無自覚 | 問題が日常に溶け込んでいる | 広告が刺さらない |
| 2 | 違和感 | 「何かおかしい」が言葉にできない | 違和感を言語化してあげる（Meta広告の最適ターゲット） |
| 3 | 直面 | 問題は認識したが構造が見えない | 構造を見せる。Scanが最も刺さる |
| 4 | 言語化 | 構造が見え始めた。方法論がない | フレームワークを提示。Gravity提案 |
| 5 | 覚悟 | 「自分が変わらなければ」と腹を括った | 実行の伴走。Coaching/Gravity開始 |
| 6 | 実行 | 変化の渦中。抵抗と戦っている | 揺り戻しへの対処。進捗の可視化 |

**サービスとの接続：**

| サービス | 主戦場 | 補助的に効く |
|---------|--------|------------|
| Scan | Stage 2〜3 | Stage 4 |
| Gravity | Stage 4〜5 | Stage 6 |
| CODE | Stage 2〜4 | Stage 5 |
| Coaching | Stage 5〜6 | Stage 4 |

## B. 設計フレーム

### B-1. 翻訳テーブル

**使う場面：** Scanレポートの「最初の一手」、WS「翻訳タイム」（1人5分）、飲み会（即座に翻訳）

石井のコアコンピタンス「翻訳力」を体系化したもの。常にこの3ステップで翻訳する：

```
① 経営者が言うこと（表面）
  ↓
② 石井の翻訳（構造的解釈）
  ↓
③ だから最初にやるべきこと（明日からできる一手）
```

**翻訳のコツ：**
- 経営者の言葉を否定しない。「それは正しい。ただ、裏にある構造は…」
- 「力」の名前を具体的に言い換える。「営業力」→「案件見極め基準の言語化力」
- 最初の一手は「明日からできること」レベルまで具体化

5パターン別に20+の翻訳パターンを整備済み。

**詳細：** [[260403_翻訳タイム_シミュレーション]]

### B-2. 3フレーム統合実装

**使う場面：** Gravity 6ヶ月デリバリーの骨格設計（ドラフト）

4象限＋組織の4レイヤー＋勝ち筋ループの3フレームをScan→Gravity→Orbitの全プロセスに統合する実装設計。

```
Scan：4象限で現在地 → 建物で空洞フロア → 勝ち筋ループでボトルネック
  ↓
Gravity M1-M2：勝ち筋ループから逆算した組織設計
Gravity M3-M4：空洞フロアの充填（制度設計・権限移譲）
Gravity M5-M6：仕組みの運用定着
  ↓
Orbit：仕組みの月次モニタリング＋拡張
```

**ステータス：** ドラフト。Gravity初受注後に実運用で検証。

**詳細：** [[260402_3フレーム実務実装設計]]


## D. 個人フレーム

### D-1. ソースコード解析v5（CODE専用）

**使う場面：** GravityCODE（クライアント向け）、自己分析

人間の行動特性を「コード」として解析するフレーム。

```
Layer 1: ASIC（検知装置）── 複雑系の相転移を検知する
Layer 2: フィルター ──── 入力を選別（発火条件・停止条件）
Layer 3: 実行パス ───── 組織/人/市場/自己の4ドメインで実行
```

CODEのヒアリングでは：発火条件（没頭）→ 停止条件（消耗）→ have to検出 → 動詞チェーン → コアプロセスの順で掘る。

**詳細：** [[260325_石井伸幸_ソースコード解析_v5]]
