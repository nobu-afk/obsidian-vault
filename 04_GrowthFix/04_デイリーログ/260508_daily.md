- Coaching LP は組織軸用 `gravity-shift-page.css` を読んでいたため青のまま残存 → 専用 `<style>` ブロックで個人軸オーバーライド
- 学び：**ブランドカラーの軸別ルールは LP の CSS 設計レベルでも明示する必要がある**（将来の LP テンプレ追加時のチェック項目化候補）

### 13. 二重防御ハーネス（Layer 1 SYSTEM プロンプト + Layer 2 PHP 強制置換）の有効性実証（260508 深夜）
- AI 出力で社内用語混入を防ぐ場合、**SYSTEM プロンプト指示（Layer 1）だけでは不十分**。AI が指示を無視するケースが実際に発生
- Layer 2（AI 出力後の PHP `str_replace` 強制置換）を入れることで「必ず対外語に変換される」最終保証ラインが確保できる
- jargon_map は **uksort で長文字列を先に置換**することで部分一致誤置換を防げる（「C-5 中判定」→「覚悟確信度が中位」を先に置換し、「C-5」→「経営者の覚悟確認」を後で処理）
- 副作用として「N ヶ月 N ヶ月」のような連続重複が発生するため、cleanup 用 preg_replace を後段に追加
- 学び：**AI ハーネスは「予防（SYSTEM プロンプト）+ 最終保証（出力後処理）」の二重防御が標準パターン**

### 14. 「コーチが判定する」設計の構造的限界（260508 深夜・ユーザー指摘）
- 旧 SCAN 進行マニュアル：PART 2「4 型判定」+ PART 3「Pre-Shift 適合診断」をコーチが対話中に確定する設計
- ユーザー指摘：自然対話で「拡散型です」「128 万が標準推奨」「中判定です」をコーチが宣告するのは構造的に不可能。深掘り質問が浅くなり、経営者目線では押し付けに見える
- 解決：**コーチは深掘り対話で素材を引き出すことに専念 → 全判定は Sonnet がレポート生成時に行う → 読み合わせフェーズ（T+68〜85）でレポートを見せながら推奨パッケージと重点アクションを経営者と確認・合意取得**
- これにより「症状診断 → 自動でパッケージ示唆 → 重点アクションまで」のフルフローが、判定根拠 100% 経営者の言葉で構成される
- 学び：**AI レポート生成型サービスでは、コーチの仕事は「判定」ではなく「素材収集 + 読み合わせの解説」**

### 15. priority-action-box 設計：「契約前に何をやるかが見える」状態の実装（260508 深夜・ユーザー指摘）
- ユーザー指摘：SCAN レポートが「拡散型 → Cultivate 推奨」までは出すが、「3 ヶ月で具体的に何をやるか」までは出さない → 経営者目線で「契約前に何をやるかが見えない」納得感の穴
- 解決：引力 7 項目スコアの △/✗ 欠損ポイントから R/C 各 4 案（α/β/γ/δ）の重点アクションを自動マッピング → priority-action-box として Block C に表示
- 経営者は「自分の症状（欠損ポイント）にピンポイントで効く処方箋（重点アクション）」を Pre-Shift の段階で見られる → 「これで実装していきましょう」で即クロージング可能
- 学び：**SaaS 型診断レポートでは「症状診断 → パッケージ示唆 → 具体行動マッピング」の 3 層が揃って初めて意思決定可能**

### 16. 「業界一般用語」と「社内用語」の境界（260508 深夜・ユーザー指摘）
- 「採用骨格」は人体メタファー由来の比喩語（社内ライティング感）／「採用基盤」は経営現場の業務用語
- LP コピーは「世界観語（OK）／業界一般用語（OK）／具体性訴求（OK）vs 社内用語（NG）」の 3 階層分類で運用
- 同じことを伝える 2 表現があるなら、**業界一般用語側に寄せる**（経営者の認知負荷を下げる）
- 学び：**ライティング時に「これは比喩か業界用語か」をチェックする習慣化**
