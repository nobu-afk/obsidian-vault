**気づき・判断メモ：**

**【3エージェント並列レビューの威力】**
- Reuse/Quality/Efficiency の3観点を同時に走らせることで、1つの観点では見落としがちな重複や非効率が浮き彫りに
- Top5 抽出までを agent に任せることで意思決定が速くなる（人間は「やるかどうか」だけ判断）

**【段階展開の重要性】**
- Tier1（即時・低リスク）→ Tier2（中規模・テスト後）→ Tier3（大規模・要慎重）の順に進めることで事故時の影響範囲を最小化
- 特に PurgeCSS は privacy-policy（最低リスク）→ 8 corp 一括 → top（高リスク） の順で展開した結果、top のみロールバックで他は OK

**【clean-css の落とし穴】**
- WP テーマ CSS のように構文エラーを持つファイルに clean-css を適用すると、エラー箇所だけでなく**正常な @media クエリも巻き込まれて削除**される
- 21KB の喪失（.fv 等のレスポンシブ）が top_本番 のレイアウト崩壊につながった
- 教訓：「コードを修正する」ツールは「壊れていない部分も触る」リスクがある

**【perl `-0` vs `-0777` の罠】**
- 多行正規表現置換は `-0777pe`（slurp mode）を必ず使う
- `-0pe` は null 区切り扱いで文字単位の置換が暴発する
- Edit ツール（precise replace）を優先・perl は最終手段

**【Skills の責任分離】**
- daily（朝）／ work-log（セッション後）／ weekly-close（週末）で役割分離が既に明確
- 信用装置セルフチェックのような思想的なものは「毎日」より「週次」の方が深掘りできる
- Note 判定ロジック8パターンは over-engineering だった（実用上3パターンで十分）

**【Vault お掃除の心理的メリット】**
- 物理削除ゼロ＋ archive 移動方式により「念のため取っておきたい」心理コストがクリア
- 1.3MB（GravityScan）の archive 化で本番フォルダ汚染が解消・新規 LP 作成時のノイズ低減

**【WhitePaper PDF 化の運用設計】**
- HTML→PDF を Chrome ヘッドレス 1 コマンド化することで、本文修正→PDF 反映が分断しない運用に
- 「メール本文の DL URL = `/whitepaper.pdf`」とすることで UTAGE サンキューメール側の更新も最小化
- 注意：PDF 実ページ数（25）と LP 表記「13ページ」が乖離していた → 統一更新で信頼性確保

**【スマホ overflow の本当の原因】**
- 当初「サービス LP の hero h1 が clamp(28px) で overflow」と推定 → 実は **hero-eyebrow inline style の長文タグライン** が overflow の主因（h1 は wrap 自体は機能していた）
- inline style に word-break が無いと、たとえ box は 90% width でも長文 1 行は overflow する
- 教訓：mobile overflow 調査では h1 だけでなく eyebrow・lead・badge 等の **すべての非 wrap テキスト** を点検

**【3 サービス参謀体系の発見】**
- Orbit を「ホームドクター → 幹部育成全振り」に変更（260424）した時点で医療メタファーが崩壊していた
- 「育成の参謀」ではブランドが弱い・「拡張」「増幅」「複製」では機械的 → **「共鳴」が Orbit（軌道）の世界観と完全整合**
- 物理メタファー（引力・天体・共鳴）で Gravity ブランド全体が貫通：個人の引力（Coaching=心）→ 組織の引力（Shift=変革）→ 幹部の引力（Orbit=共鳴）

**【LP コピーの責め口調リスク】**
- 「やるべきことはやった。なのに半年前と何が変わったか説明できない」は **経営者の self-image 攻撃** と読まれるリスク
- 主語を「あなた（経営者）」→「違い（事象）」に転換することで、責任主体が経営者から離れ、Gravity の役割（言語化を手伝う）が前面化
- 「説明できない」（あなたの失敗）→「言葉にならない」（私たちが手伝う領域）の **僅か 1 語の違いがラポールを左右**

**【旧 Gravity 単体時代のコピーが SERIES 体系に残っていた問題】**
- THE PROBLEM 結論「話すだけで、組織の仕組みに変わる場所」は単一サービス時代の表現で、5 サービス連結を表していなかった
- 「あなたの引力を組織に翻訳し、幹部に共鳴させ続ける伴走」に変更し、**個人→組織→共鳴の 3 領域を 1 行で**
- 教訓：サービス体系を変更したら全 LP の中核コピーをチェックする必要あり
