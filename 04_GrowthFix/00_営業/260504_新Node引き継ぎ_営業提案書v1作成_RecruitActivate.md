## 1. ミッション概要

### 目的

**「LP は KISS 原則で必要最小限・営業提案書は詳細に盛り込む」運用への移行準備**として、**全差別化資産を吸収する受け皿（営業提案書 v1）を先に作る**。提案書完成後、LP の minimal 化に着手できる状態にする。

### スコープ

| サービス | 提案書 | LP の扱い |
|---|:-:|---|
| **Recruit（採用基盤・80 万・3 ヶ月）**| **必須** | 提案書完成後に minimal 化（情報少なめ） |
| **Activate（躍動組織・80 万・3 ヶ月）**| **必須** | 提案書完成後に minimal 化 |
| **Shift（R+A 複合・150 万・6 ヶ月）**| 上記 2 つに包含（複合パッケージ章として）| minimal LP 維持（5/3 既施工） |
| **Coaching（38 万・6 ヶ月）**| 任意（minimal 提案書化）| minimal LP 維持（5/3 既施工） |
| **Scan（10 万・60 分）**| **不要** | LP 詳細維持（安価 / 単発・LP で意思決定する性質） |
| **CODE（5 万・60 分）**| **不要** | LP 詳細維持（同上） |
| **Orbit（月 15-25 万・継続運用）**| 商談時提示（minimal）| minimal LP 維持 |

### 完成判定基準

- A4 推定 15-25 ページ（MD 800-1200 行）
- Recruit / Activate の **全差別化資産を吸収**（49 質問 ★ コア / 採用コスト試算 / 採用基盤実装書 12 要素 / Week 0 体験設計書 / 躍動 3 点セット 27p / 3 ヶ月予言の書 / 価格パッケージ表 / 4 型 → サービス推奨ロジック）
- 5/8 長谷さんモニターで使える形（事前送付 or 当日提示）
- md_to_pdf.sh で PDF 化可能（既存スクリプト：`06_開発/scripts/md_to_pdf.sh`）

### 完成後アクション

1. 5/4-5/7 で骨子完成 → 5/8 長谷さんモニターで実地検証
2. 5/9 以降：モニターで「経営者が見たい情報」を反映して提案書 v1.1
3. 提案書 v1.1 完成タイミングで **意思決定**：
   - LP に残すべき差別化資産を厳選
   - LP minimal 化を実装（19 LP 大幅削減）
   - 提案のスケジュールよりも前倒しになる想定（user 指示）

## 7. Why この設計（背景）

### 旧 Node の議論サマリー（5/3 夜）

User 自己認識：「**不安から情報を盛り込みすぎる傾向**」（名刺事例 + LP 過多事例の一般化）

戦略的洞察：「**情報は削ぎ落とされた情報の総量に価値がある**」（情報理論的洞察）

運用転換：
- 旧：LP に全差別化資産を盛り込む（メンテ工数複利問題・19 LP 連鎖更新）
- 新：**営業提案書に詳細を集約 → LP は KISS 原則で必要最小限**

実装順序（重要）：
1. 営業提案書 v1 を **先に作る**（差別化資産の受け皿）
2. その後で LP minimal 化（受け皿があるから安心して削れる）

### user の傾向自己認識（230503 言語化）

- 名刺事例：盛り込みすぎ → 大幅削減で良くなった
- LP 事例：シャープ化（5/2）後も 49 質問・3 軸ハイブリッド・採用コスト試算等を LP に盛り込みすぎ
- 一般化：「載せれば伝わる」「載せないと不安」が罠 → 実際は「削げば伝わる」「載せると埋没する」

詳細：`09_会社OS/公開/文化/判断基準.md` § ★ KISS 原則 ／ 情報の削ぎ落とし価値（260503 夜 確立）
